## 一、线程基础

### Q1：创建线程有哪些方式？有什么区别？

**答：**

从本质上看，创建线程**只有一种方式**：`new Thread()`。所谓"多种方式"是指线程的**任务定义**方式不同：

1. **继承 `Thread` 类**：重写 `run()` 方法。缺点：Java 单继承，扩展性差。
2. **实现 `Runnable` 接口**：推荐，通过组合代替继承，可共享同一 `Runnable` 实例。
3. **实现 `Callable` + `FutureTask`**：支持返回值和异常抛出。
4. **线程池（`ExecutorService`）**：生产推荐，复用线程。

---

### Q2：`sleep()` 和 `wait()` 的区别？

**答：**

| 特性 | `Thread.sleep(ms)` | `Object.wait()` |
|------|--------------------|-----------------|
| 所属类 | `Thread` 静态方法 | `Object` 实例方法 |
| 锁释放 | **不释放**持有的锁 | **释放**持有的锁 |
| 使用前提 | 无要求 | 必须在 `synchronized` 块内 |
| 唤醒方式 | 超时自动唤醒 | `notify()`/`notifyAll()`/`interrupt()` |
| 异常 | `InterruptedException` | `InterruptedException` |

---

### Q3：如何正确停止一个线程？

**答：**

- **不要使用** `Thread.stop()`（已废弃，强制终止，可能导致数据不一致）
- **推荐方式**：利用**中断机制**

```java
// 方式1：中断标志位
public void run() {
    while (!Thread.currentThread().isInterrupted()) {
        // 执行任务
    }
}

// 方式2：在阻塞操作中捕获 InterruptedException 并退出
public void run() {
    try {
        while (!Thread.interrupted()) {
            doTask();
            Thread.sleep(100); // 阻塞时会抛 InterruptedException
        }
    } catch (InterruptedException e) {
        // 清理资源后退出
        Thread.currentThread().interrupt(); // 重新设置中断标记
    }
}

// 停止线程
thread.interrupt();
```

### Q4：`notify()` 和 `notifyAll()` 的区别？

**答：**

- `notify()`：随机唤醒等待队列中的**一个**线程，其他线程继续等待。
- `notifyAll()`：唤醒等待队列中**所有**线程，所有线程重新竞争锁，只有一个能获取。

**推荐使用 `notifyAll()`**，因为：
1. `notify()` 随机唤醒可能导致**死锁**（唤醒了不该被唤醒的线程，正确的线程永远等待）
2. 条件判断应使用 `while` 而非 `if`（防止虚假唤醒）

```java
// 正确的 wait 使用模式
synchronized (lock) {
    while (!condition) { // while 而不是 if
        lock.wait();
    }
    // 执行操作
    lock.notifyAll();
}
```

## 二、synchronized 与锁机制

### Q5：synchronized 锁升级的原理？什么情况下锁会升级？

**答：**

锁升级路径：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**（不可逆）

| 触发条件 | 状态变化 |
|----------|----------|
| 第一个线程加锁 | 无锁 → 偏向锁（写入线程ID到 Mark Word） |
| 另一线程竞争，偏向锁撤销 | 偏向锁 → 轻量级锁（CAS自旋） |
| 多线程激烈竞争，自旋超限 | 轻量级锁 → 重量级锁（OS mutex，线程阻塞） |
| 调用 `hashCode()` | 偏向锁直接升级为轻量级锁（Mark Word需存hashCode） |
| 调用 `wait()` | 直接升级为重量级锁 |

---

### Q6：死锁产生的四个必要条件是什么？如何避免？

**答：**

**四个必要条件（缺一不可）：**
1. **互斥**：资源同时只能被一个线程持有
2. **请求保持**：线程持有资源同时等待其他资源
3. **不剥夺**：已获得的资源不能被强制剥夺
4. **循环等待**：线程间形成循环等待资源的环

**预防/避免死锁：**
- 固定加锁顺序（打破循环等待）
- 使用 `ReentrantLock.tryLock(timeout)` 超时等待（打破请求保持）
- 使用 `lockInterruptibly()` 可中断等待
- 使用 `Ordered Locking`（按对象 hash 或 id 排序获取锁）
- 减少锁的使用范围，用局部变量替代共享变量

---

### Q7：什么是锁消除和锁粗化？

**答：**

- **锁消除**：JIT 编译器通过**逃逸分析**，若确定某段代码中加锁的对象不可能被多线程共享，则自动去掉该锁。如：对方法内 `StringBuilder` 调用 `append` 其实是有锁的（`StringBuffer`），但 JIT 发现它不逃逸出方法，会消除锁。

- **锁粗化**：若 JIT 发现代码对**同一对象**进行多次连续加锁/解锁，会将这些加锁范围合并为一个更大的范围，减少频繁进出同步块的开销。

```java
// 锁粗化示例（编译器可能将三次 synchronized 合并为一次）
for (int i = 0; i < 100; i++) {
    synchronized (lock) {
        list.add(i);
    }
}
// 优化后等价于：
synchronized (lock) {
    for (int i = 0; i < 100; i++) {
        list.add(i);
    }
}
```

## 三、volatile 与内存模型

### Q8：JMM 的三大特性是什么？synchronized 和 volatile 分别保证了哪些特性？

**答：**

| 特性 | synchronized | volatile |
|------|-------------|---------|
| 原子性 | ✅（同步块内操作） | ❌（复合操作不保证） |
| 可见性 | ✅（解锁前刷新到主内存） | ✅（写后立即刷新，读前重新加载） |
| 有序性 | ✅（保证串行执行） | ✅（禁止指令重排序，内存屏障） |

---

### Q9：volatile 的使用场景有哪些？

**答：**

1. **状态标志位**：单一写线程，多读线程
   ```java
   volatile boolean running = true;
   // 主线程 running = false;
   // 子线程 while(running) {}
   ```

2. **双重检查锁（DCL）单例**：防止半初始化对象被其他线程获取

3. **独立观察（发布不变对象）**：写入完整对象后，通过 volatile 发布引用

4. **开关控制**：如热更新配置时，`volatile Config config`

> **不适用场景**：自增（`i++`）、复合条件判断（先读后写），这些需要 `AtomicInteger` 或 `synchronized`。

## 四、CAS 与原子类

### Q10：CAS 的原理和缺点？

**答：**

**原理：** CAS（Compare And Swap）是 CPU 级别的原子指令（x86 的 `cmpxchg`），包含内存地址V、期望值E、新值N：若 V 的当前值等于 E，则将 V 更新为 N 并返回 true；否则什么都不做并返回 false。Java 通过 `Unsafe` 类暴露该能力。

**三大缺点：**

1. **ABA 问题**：值从 A → B → A，CAS 看不出变化。解决：`AtomicStampedReference`（版本号）。
2. **自旋开销**：CAS 失败会循环重试，高并发下消耗大量 CPU。解决：`LongAdder` 分段竞争。
3. **只能保证单变量原子性**：多变量需要封装为对象用 `AtomicReference` 或用锁。

## 五、AQS 与 ReentrantLock

### Q11：AQS 的核心设计是什么？

**答：**

`AbstractQueuedSynchronizer` 的核心是：

1. **`int state`（volatile）**：同步状态，语义由子类定义，通过 CAS 修改保证原子性。
2. **CLH 变体 FIFO 双向队列**：获取同步状态失败的线程被封装为 `Node`，加入队尾，通过 `LockSupport.park()` 阻塞。
3. **模板方法模式**：子类重写 `tryAcquire`/`tryRelease`（独占）或 `tryAcquireShared`/`tryReleaseShared`（共享），AQS 提供排队、唤醒的骨架逻辑。

---

### Q12：ReentrantLock 可重入是如何实现的？

**答：**

`ReentrantLock` 基于 AQS，`state` 代表重入次数：

1. **加锁时**：若当前持有锁的线程 == 当前线程，则 `state + 1`（可重入）；否则进入等待队列。
2. **解锁时**：`state - 1`，当 `state` 降为 0 时才真正释放锁，唤醒队列中下一个线程。

```java
// ReentrantLock.Sync.tryAcquire 核心逻辑
if (current == getExclusiveOwnerThread()) {
    int nextc = c + acquires; // state + 1（重入）
    setState(nextc);
    return true;
}
```

## 六、线程池

### Q13：线程池提交任务的流程？execute 和 submit 的区别？

**答：**

**提交流程（execute）：**
1. 当前线程数 < `corePoolSize` → 创建核心线程
2. 当前线程数 >= `corePoolSize` → 放入工作队列
3. 队列满 + 线程数 < `maximumPoolSize` → 创建非核心线程
4. 队列满 + 线程数 >= `maximumPoolSize` → 拒绝策略

**execute vs submit：**

| 方法 | 返回值 | 异常处理 |
|------|--------|---------|
| `execute(Runnable)` | 无 | 异常传播到线程的 `UncaughtExceptionHandler` |
| `submit(Callable/Runnable)` | `Future` | 异常被封装在 `Future` 中，调用 `get()` 时抛出 |

---

### Q14：线程池的核心线程数如何配置？

**答：**

- **CPU 密集型**（大量计算，少 IO）：`N + 1`（N = CPU 核数），+1 是为了在线程偶发暂停时 CPU 不空闲
- **IO 密集型**（大量 IO、数据库、网络）：`2N` 或更大，典型值 `N × (1 + 等待时间/计算时间)`
- **混合型**：拆分为两个线程池分别处理 CPU 密集和 IO 密集任务

实际生产中，应通过**压测**确定最优线程数，并使用 Arthas/JConsole 观察线程池利用率。

---

### Q15：如何实现动态线程池？

**答：**

`ThreadPoolExecutor` 支持运行时修改参数：

```java
executor.setCorePoolSize(newCore);
executor.setMaximumPoolSize(newMax);
executor.setKeepAliveTime(newTime, TimeUnit.SECONDS);
// 注意：先改 max 再改 core，避免 core > max 报错
```

配合配置中心（如 Nacos/Apollo）监听配置变更，自动下发新参数。  
开源方案：**美团 DynamicTp**、**hippo4j**，提供线程池注册、监控、告警、动态修改的完整能力。

## 七、并发容器

### Q16：ConcurrentHashMap 的 JDK 7 与 JDK 8 实现区别？

**答：**

| 版本 | 数据结构 | 加锁方式 | 并发度 |
|------|----------|---------|--------|
| JDK 7 | Segment 数组 + HashEntry 链表 | 分段锁（Segment 继承 ReentrantLock） | 最多 16 个并发写 |
| JDK 8 | Node 数组 + 链表/红黑树（同 HashMap） | CAS + synchronized（桶头节点） | 理论上 Node 数组长度个并发写 |

JDK 8 优化：
- 粒度更细（一个桶一把锁 vs 多个桶共用一把 Segment 锁）
- 链表长度 ≥ 8 且数组长度 ≥ 64 时转红黑树，查询 O(log n)
- 扩容时多线程协助迁移（`helpTransfer`）

---

### Q17：`HashMap` 在多线程下有哪些问题？

**答：**

1. **死循环（JDK 7）**：JDK 7 的 `resize` 使用头插法，多线程扩容时可能形成循环链表，导致 CPU 100%。JDK 8 改为尾插法，修复了此问题。
2. **数据丢失**：两个线程同时 `put` 到同一个桶，一个线程的数据被覆盖。
3. **`size()` 不准确**：并发修改下计数不准。
4. **`fail-fast` 迭代器**：迭代时修改抛 `ConcurrentModificationException`。

**解决方案**：多线程场景使用 `ConcurrentHashMap`，或在读多写少场景用 `Collections.synchronizedMap`（性能差）。

## 八、并发编程综合题

### Q18：如何实现一个线程安全的单例模式？有哪几种方式？

**答：**

**方式1：DCL 双重检查锁（推荐）**

```java
public class Singleton {
    private static volatile Singleton instance; // volatile 必须！
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**方式2：静态内部类（推荐，更简洁）**

```java
public class Singleton {
    private Singleton() {}
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE; // JVM 类加载保证线程安全
    }
}
```

**方式3：枚举（防反序列化破坏，Effective Java 推荐）**

```java
public enum Singleton {
    INSTANCE;
    public void doSomething() { /* ... */ }
}
```

### Q19：请设计一个简单的生产者-消费者模型

**答：**

```java
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class PCModel {
    private static final BlockingQueue<Integer> queue = new LinkedBlockingQueue<>(10);
    private static final AtomicInteger id = new AtomicInteger(0);
    private static volatile boolean producerDone = false;

    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(4);

        // 2个生产者
        for (int i = 0; i < 2; i++) {
            pool.submit(() -> {
                try {
                    for (int j = 0; j < 10; j++) {
                        int item = id.incrementAndGet();
                        queue.put(item);
                        System.out.println(Thread.currentThread().getName() + " 生产: " + item);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }

        // 2个消费者
        for (int i = 0; i < 2; i++) {
            pool.submit(() -> {
                try {
                    while (true) {
                        Integer item = queue.poll(500, TimeUnit.MILLISECONDS);
                        if (item == null) break;
                        System.out.println(Thread.currentThread().getName() + " 消费: " + item);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }

        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);
    }
}
```

### Q20：ThreadLocal 的原理与内存泄漏问题？

**答：**

**原理：** 每个 `Thread` 内部持有一个 `ThreadLocalMap`（以 `ThreadLocal` 实例为 key，实际值为 value）。`ThreadLocal.set(value)` 本质是 `Thread.currentThread().threadLocalMap.set(this, value)`。

**内存泄漏原因：**
- `ThreadLocalMap` 的 key（`ThreadLocal` 引用）是**弱引用**，GC 时 `ThreadLocal` 对象可能被回收，key 变为 null。
- 但 value 是**强引用**，若线程长期存活（如线程池线程），value 无法被回收，造成内存泄漏。

**预防措施：**
1. 使用完后必须调用 `threadLocal.remove()`
2. 使用 try-finally 确保清理：

```java
private static final ThreadLocal<SimpleDateFormat> formatter =
    ThreadLocal.withInitial(SimpleDateFormat::new);

public String format(Date date) {
    try {
        return formatter.get().format(date);
    } finally {
        formatter.remove(); // 线程池场景必须 remove
    }
}
```
