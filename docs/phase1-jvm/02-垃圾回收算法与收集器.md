## 一、判断对象存活

### 1.1 引用计数法（JVM 未采用）

给每个对象维护引用计数器，计数为 0 时可回收。  
**缺点**：无法解决**循环引用**问题（两个对象互相引用，计数永不为 0）。

### 1.2 可达性分析（GC Roots Tracing）

以 **GC Roots** 为起点，向下搜索，不可达的对象判定为可回收。

**GC Roots 包括：**
- 虚拟机栈中引用的对象（局部变量）
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈（JNI）引用的对象
- JVM 内部引用（基本类型 Class、常驻异常对象 NullPointerException 等）
- 所有被同步锁持有的对象
- JMXBean、JVMTI 中注册的回调

```java
// GC Roots 示例
public class GCRootsDemo {
    private static Object staticRef = new Object(); // GC Root（静态变量）

    public void method() {
        Object localRef = new Object(); // GC Root（栈帧局部变量）
        // localRef 出栈后，该对象失去 GC Root 引用，可被回收
    }
}
```

### 1.3 四种引用类型

| 引用类型 | 回收时机 | 常见用途 |
|---------|---------|---------|
| 强引用（StrongReference） | 永不回收（内存溢出也不回收） | `Object o = new Object()` |
| 软引用（SoftReference） | 内存不足时回收 | 缓存（如图片缓存） |
| 弱引用（WeakReference） | 下次 GC 时回收 | `WeakHashMap`、`ThreadLocalMap` 的 key |
| 虚引用（PhantomReference） | 随时回收，必须配合 `ReferenceQueue` | 跟踪对象回收，用于堆外内存管理（如 DirectByteBuffer） |

```java
import java.lang.ref.*;

Object obj = new Object();
SoftReference<Object> softRef = new SoftReference<>(obj);
WeakReference<Object> weakRef = new WeakReference<>(obj);
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);

obj = null; // 去除强引用
System.gc();

System.out.println(softRef.get());   // 内存充足时：非null
System.out.println(weakRef.get());   // null（已被回收）
System.out.println(phantomRef.get()); // 始终 null
```

### 1.4 finalize() 方法

对象被 GC 前，若重写了 `finalize()`，会被加入 `F-Queue` 执行，可自救（重新建立强引用）。  
**生产中不建议使用**，执行时机不确定，且只能自救一次。

## 二、垃圾回收算法

### 2.1 标记-清除（Mark-Sweep）

```
标记阶段：遍历所有对象，标记存活对象
清除阶段：回收未标记对象

Before:  [ 存活 | 垃圾 | 存活 | 垃圾 | 存活 ]
After:   [ 存活 |  ①  | 存活 |  ②  | 存活 ]  ←  产生内存碎片
```

**缺点：** 内存碎片（大对象可能找不到连续空间）；标记和清除效率不高。  
**应用：** CMS 老年代（核心算法）。

### 2.2 标记-复制（Copying）

```
将内存分为两半（from/to），只使用 from 区
GC 时：存活对象复制到 to 区，然后整块清空 from 区

From:  [ 存活 | 垃圾 | 存活 | 垃圾 ]
To:    [    ]

After:
From:  [    ]  ← 整块清空
To:    [ 存活 | 存活 ]  ← 连续，无碎片
```

**优点：** 无内存碎片，分配快（指针碰撞）。  
**缺点：** 内存利用率只有 50%；存活对象多时复制代价大。  
**应用：** 新生代（改进版：Eden 80% + 两个 Survivor 各 10%，利用率达 90%）。

### 2.3 标记-整理（Mark-Compact）

```
标记阶段：标记存活对象
整理阶段：将存活对象向一端移动，清理边界以外的内存

Before:  [ 存活 | 垃圾 | 存活 | 垃圾 | 存活 ]
After:   [ 存活 | 存活 | 存活 |     free     ]
```

**优点：** 无内存碎片，内存利用率高。  
**缺点：** 需要移动对象，更新引用，STW 时间较长。  
**应用：** 老年代（Serial Old、Parallel Old、G1 的 Full GC）。

### 2.4 算法对比

| 算法 | 速度 | 内存利用率 | 碎片 | 适用代 |
|------|------|-----------|------|--------|
| 标记-清除 | 中 | 高 | 有 | 老年代 |
| 标记-复制 | 快 | 低（50%） | 无 | 新生代 |
| 标记-整理 | 慢 | 高 | 无 | 老年代 |

## 三、分代收集理论

### 3.1 分代假说

1. **弱分代假说（Weak Generational Hypothesis）**：绝大多数对象是朝生夕灭的
2. **强分代假说（Strong Generational Hypothesis）**：熬过越多次 GC 的对象越难以消亡

基于此，JVM 将堆划分为**新生代**和**老年代**，针对各自特点使用不同算法。

### 3.2 GC 类型

| GC 类型 | 回收范围 | 触发条件 | STW 时间 |
|---------|---------|---------|---------|
| Minor GC（YGC） | 新生代 | Eden 满 | 毫秒级 |
| Major GC | 老年代 | 老年代满，或 CMS 触发 | 秒级（CMS 并发） |
| Full GC | 整个堆 + 方法区 | 老年代满、`System.gc()`、空间担保失败 | 秒级 |
| Mixed GC | 新生代 + 部分老年代 | G1 特有 | — |

## 四、垃圾收集器

### 4.1 收集器组合

```
新生代收集器         老年代收集器
┌──────────┐       ┌─────────────────┐
│  Serial  │ ────► │  Serial Old      │
├──────────┤       ├─────────────────┤
│ ParNew   │ ────► │  CMS             │
├──────────┤       ├─────────────────┤
│ Parallel │ ────► │  Parallel Old    │
│ Scavenge │
└──────────┘
┌─────────────────────────────────────┐
│              G1（独立）               │
├─────────────────────────────────────┤
│              ZGC（独立）              │
├─────────────────────────────────────┤
│          Shenandoah（独立）           │
└─────────────────────────────────────┘
```

### 4.2 Serial / Serial Old

- **单线程**收集器，GC 时必须停止所有用户线程（Stop-The-World）
- 适用：客户端模式、单核 CPU、内存较小的场景
- 参数：`-XX:+UseSerialGC`

### 4.3 ParNew

- **多线程**版 Serial，用于新生代，与 CMS 配合使用
- 参数：`-XX:+UseParNewGC`，`-XX:ParallelGCThreads=N`

### 4.4 Parallel Scavenge / Parallel Old

- 关注**吞吐量**（用户代码运行时间 / 总时间）
- 适用：后台计算、批处理任务（不关注停顿延迟）
- 参数：`-XX:+UseParallelGC`（JDK 8 默认）
- 支持自适应调节：`-XX:+UseAdaptiveSizePolicy`

### 4.5 CMS（Concurrent Mark-Sweep）

- 关注**最短 STW 时间**（停顿），JDK 14 废弃
- **六个阶段：**

```
① 初始标记（Initial Mark）    STW  ← 标记 GC Roots 直接关联对象（快）
② 并发标记（Concurrent Mark）      ← 与用户线程并发（慢，但不 STW）
③ 预清理（Preclean）               ← 处理并发标记期间的变化
④ 可中断预清理（AbortablePreclean）← 尽量减少重新标记的工作量
⑤ 重新标记（Remark）          STW  ← 修正并发期间的变动（增量更新）
⑥ 并发清除（Concurrent Sweep）     ← 与用户线程并发清除垃圾
```

**CMS 问题：**
- **浮动垃圾**：并发阶段产生的新垃圾，本次无法回收
- **并发模式失败（Concurrent Mode Failure）**：老年代在 CMS 运行期间填满，触发 Full GC
- **内存碎片**：标记-清除算法，不整理内存；配合 `-XX:CMSFullGCsBeforeCompaction` 定期整理

## 五、G1 收集器详解

### 5.1 G1 特点

- JDK 9 起成为**默认收集器**
- 基于 **Region** 的内存布局（不再物理分代）
- 可预测的停顿时间（`-XX:MaxGCPauseMillis=200`）
- 适用：大堆（> 6GB）、对停顿时间有要求的服务端应用

### 5.2 Region 模型

```
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ E │ E │ E │ S │ O │ O │ H │ O │
└───┴───┴───┴───┴───┴───┴───┴───┘
E = Eden  S = Survivor  O = Old  H = Humongous（大对象，直接分配 Old Region）
每个 Region 约 1MB~32MB（2的幂），大小统一
```

### 5.3 G1 收集阶段

```
① Young GC（Minor GC）
   - 回收所有 Eden Region
   - 存活对象复制到 Survivor Region 或晋升 Old Region
   - STW，多线程并行

② Concurrent Marking Cycle（并发标记）
   触发条件：老年代占比超过 -XX:InitiatingHeapOccupancyPercent（默认45%）
   - 初始标记（STW，随 Young GC 顺带完成）
   - 并发标记（与用户线程并发）
   - 最终标记（STW，SATB 快照处理）
   - 清理（STW，统计 Region 存活情况）

③ Mixed GC
   - 回收所有 Eden Region + 价值最高的 Old Region（优先级由存活对象少的Region排序）
   - 核心机制：优先回收价值大（垃圾多）的Region（Garbage First！）

④ Full GC（兜底，尽量避免）
   - 单线程标记-整理，STW 时间长
```

### 5.4 G1 重要参数

```bash
-XX:+UseG1GC                       # 启用 G1
-XX:MaxGCPauseMillis=200            # 期望最大停顿时间（软目标）
-XX:InitiatingHeapOccupancyPercent=45 # 触发并发标记的堆占比
-XX:G1HeapRegionSize=16m            # Region 大小
-XX:G1NewSizePercent=5              # 新生代最小占比
-XX:G1MaxNewSizePercent=60          # 新生代最大占比
-XX:ConcGCThreads=4                 # 并发 GC 线程数
```

### 5.5 G1 vs CMS

| 特性 | G1 | CMS |
|------|-----|-----|
| 碎片 | **无碎片**（Region 复制） | 有碎片 |
| 可预测停顿 | ✅ | ❌ |
| 内存占用 | 较高（Remember Sets 等） | 较低 |
| 适用堆大小 | > 6GB | 4~8GB |
| 状态 | JDK 9+ 默认 | JDK 14 废弃 |

## 六、ZGC 与 Shenandoah

### 6.1 ZGC

- JDK 11 实验性引入，JDK 15 正式发布，JDK 21 支持分代
- **设计目标**：停顿时间不超过 **1ms**（甚至 10μs 级别）
- 支持 TB 级别堆内存

**核心技术：**
- **染色指针（Colored Pointers）**：利用 64 位地址的高位存储 GC 元数据（Marked0/Marked1/Remapped/Finalized）
- **读屏障（Load Barrier）**：访问对象引用时触发，保证并发移动的正确性
- **并发整理**：GC 大部分阶段与用户线程并发进行

```bash
-XX:+UseZGC              # JDK 15+
-XX:ZAllocationSpikeTolerance=2  # 分配速率容错
-XX:+ZGenerational        # JDK 21+ 分代 ZGC（推荐）
```

### 6.2 Shenandoah

- Red Hat 开发，类似 ZGC，支持并发压缩
- 目标：任何堆大小下停顿时间都低于 10ms

### 6.3 各收集器 STW 对比

| 收集器 | 典型 STW | 适用场景 |
|--------|---------|---------|
| Serial | 数百 ms~秒 | 客户端、小堆 |
| Parallel | 数百 ms | 批处理，追求吞吐 |
| CMS | 数十 ms（初始/重新标记） | Web 服务，延迟敏感 |
| G1 | 可配，通常 < 200ms | 大堆服务端 |
| ZGC | < 1ms | 超大堆，延迟极敏感 |
| Shenandoah | < 10ms | 延迟敏感 |

## 八、大厂常见面试题

### Q1：GC Roots 包括哪些？为什么不用引用计数法？

**答：**

**GC Roots：**
1. 虚拟机栈中的局部变量引用
2. 方法区中类静态变量引用
3. 方法区中常量引用
4. 本地方法栈（JNI）引用
5. JVM 内部引用（String 常量池、基本类型 Class 对象等）
6. 被 synchronized 锁持有的对象

**不用引用计数法的原因：** 无法解决**循环引用**问题。A 持有 B 的引用，B 持有 A 的引用，若外部没有任何引用指向 A/B，两者的引用计数都不为 0，永远无法被回收，造成内存泄漏。

---

### Q2：什么是 STW（Stop-The-World）？为什么无法完全消除？

**答：**

STW（Stop The World）指 GC 时暂停所有用户线程，只有 GC 线程工作。

**无法完全消除的原因（以可达性分析为例）：** 若用户线程在分析过程中持续修改对象引用关系，分析结果可能不一致。虽然 CMS/G1/ZGC 通过各种并发技术（SATB、读/写屏障）减少了 STW 时间，但**初始标记**（枚举 GC Roots 时）通常仍需短暂 STW 保证快照一致性。

ZGC 通过染色指针和读屏障，将 STW 降低至毫秒级甚至更低。

---

### Q3：CMS 的并发模式失败（Concurrent Mode Failure）是什么原因？如何解决？

**答：**

**原因：** CMS 并发清除阶段与用户线程并发，此时用户线程仍在分配对象并向老年代晋升。若此期间老年代空间被耗尽，CMS 无法完成清理，触发 Concurrent Mode Failure，JVM 被迫触发 Serial Old 进行 Full GC（STW 时间很长）。

**解决方案：**
1. 降低 CMS 触发阈值：`-XX:CMSInitiatingOccupancyFraction=70`（默认 92%），更早开始 GC
2. 预留更多老年代空间，减少晋升压力
3. 处理内存碎片：`-XX:+UseCMSCompactAtFullCollection`
4. 迁移到 G1（G1 避免了这个问题）

---

### Q4：G1 为什么能预测停顿时间？

**答：**

G1 通过以下机制实现可预测停顿：

1. **基于 Region 的独立回收**：每次 GC 只选择部分 Region 回收（Mixed GC），可控制本次 GC 的工作量。
2. **历史统计模型**：G1 会统计每个 Region 的回收耗时和垃圾量，建立预测模型。
3. **优先队列**：根据用户设置的 `-XX:MaxGCPauseMillis`，优先选择**回收效益最高（垃圾占比最大）的 Region**，在停顿时间内尽可能多地回收。
4. **增量 GC**：不像 CMS 必须回收整个老年代，G1 可分多次逐步回收，每次 GC 的停顿时间相对稳定。

---

### Q5：Minor GC、Major GC、Full GC 的区别？触发条件分别是什么？

**答：**

| GC 类型 | 回收区域 | 触发条件 | STW |
|---------|---------|---------|-----|
| Minor GC | 新生代 | Eden 区满 | 短 |
| Major GC | 老年代 | 通常随 Full GC 一起，或 CMS 独立触发 | 较长 |
| Full GC | 整个堆 + 方法区 | ① 老年代空间不足 ② 元空间不足 ③ `System.gc()`（建议） ④ CMS 并发失败 ⑤ 空间分配担保失败 | 最长 |

**优化方向：**
- 减少 Full GC：合理调整堆大小、降低 CMS 触发阈值、避免大对象频繁创建
- 监控工具：`jstat -gcutil <pid> 1000`（每秒输出 GC 统计）
