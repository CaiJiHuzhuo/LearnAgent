## 一、类的生命周期

```
┌────────────────────────────────────────────────────────────┐
│                        类的生命周期                          │
│                                                             │
│  加载  →  验证  →  准备  →  解析  →  初始化  →  使用  →  卸载 │
│  ←────────────── 连接（Linking） ──────────────►           │
└────────────────────────────────────────────────────────────┘
```

整个过程中，**加载**、**验证**、**准备**、**初始化**、**卸载**的顺序是确定的；  
**解析**不一定：可以在初始化之后（支持动态绑定/晚期绑定）。

## 二、类加载的五个阶段

### 2.1 加载（Loading）

完成三件事：
1. 通过类的全限定名获取**二进制字节流**（可来自 .class 文件、网络、动态生成、ZIP 包等）
2. 将字节流转化为**方法区的运行时数据结构**
3. 在堆中生成该类的 **`java.lang.Class` 对象**（作为方法区数据的访问入口）

### 2.2 验证（Verification）

确保 Class 字节流符合 JVM 规范，防止恶意代码破坏 JVM：

| 验证阶段 | 内容 |
|---------|------|
| 文件格式验证 | 魔数（`0xCAFEBABE`）、版本号、常量池合法性 |
| 元数据验证 | 类继承关系、字段/方法是否与父类冲突 |
| 字节码验证 | 操作数栈类型匹配、跳转指令合法性 |
| 符号引用验证 | 符号引用的类/字段/方法是否存在、是否有访问权限 |

### 2.3 准备（Preparation）

为类变量（`static` 字段）分配内存并设置**初始零值**（非代码中的赋值）：

```java
public class PrepareDemo {
    public static int value = 123;
    // 准备阶段：value = 0（零值）
    // 初始化阶段：value = 123（执行 <clinit>）

    public static final int CONST = 456;
    // ConstantValue 属性，准备阶段直接 = 456
}
```

| 类型 | 零值 |
|------|------|
| int/byte/short/char | 0 |
| long | 0L |
| float | 0.0f |
| double | 0.0d |
| boolean | false |
| 引用类型 | null |

### 2.4 解析（Resolution）

将常量池中的**符号引用**替换为**直接引用**（内存地址/偏移量）：
- 类/接口解析
- 字段解析
- 类方法解析
- 接口方法解析

### 2.5 初始化（Initialization）

执行类构造器 `<clinit>()` 方法（编译器自动合并类变量赋值 + 静态代码块）。

**触发初始化的六种情况（主动引用）：**

```java
// 1. new、getstatic、putstatic、invokestatic
new MyClass();
int x = MyClass.staticField;
MyClass.staticMethod();

// 2. 反射调用
Class.forName("com.example.MyClass");

// 3. 初始化子类时，父类未初始化
class Child extends Parent {} // 触发 Parent 初始化

// 4. 包含 main() 的启动类

// 5. JDK 7+：MethodHandle 解析的 REF_getStatic 等句柄

// 6. JDK 8+：接口 default 方法实现
```

**不触发初始化（被动引用）：**

```java
// 通过子类引用父类静态字段，不初始化子类
System.out.println(Child.PARENT_FIELD); // 只初始化 Parent

// 数组定义
MyClass[] array = new MyClass[10];      // 不初始化 MyClass

// 引用编译期常量（ConstantValue）
System.out.println(MyClass.FINAL_CONST); // 编译期替换，不触发初始化
```

**`<clinit>` 与 `<init>` 的区别：**

| 方法 | 来源 | 调用时机 |
|------|------|---------|
| `<clinit>` | static 变量赋值 + static{} 合并 | 类初始化时（一次） |
| `<init>` | 构造方法 + 实例变量赋值 + {} 合并 | new 对象时（每次） |

```java
public class InitOrder {
    static { System.out.println("静态代码块"); }  // <clinit>
    { System.out.println("实例代码块"); }          // <init>
    public InitOrder() { System.out.println("构造方法"); }

    public static void main(String[] args) {
        new InitOrder();
        new InitOrder();
    }
}
// 输出：
// 静态代码块   （只执行一次）
// 实例代码块
// 构造方法
// 实例代码块
// 构造方法
```

## 三、类加载器

### 3.1 类加载器层次（JDK 8）

```
Bootstrap ClassLoader（引导类加载器）
    └─ 加载：$JAVA_HOME/jre/lib/rt.jar 等核心类库
    └─ C++ 实现，Java 中表示为 null

Extension ClassLoader（扩展类加载器）
    └─ 加载：$JAVA_HOME/jre/lib/ext 目录
    └─ sun.misc.Launcher$ExtClassLoader

Application ClassLoader（应用类加载器，系统类加载器）
    └─ 加载：ClassPath（-classpath/-cp）下的类
    └─ sun.misc.Launcher$AppClassLoader
    └─ ClassLoader.getSystemClassLoader() 返回的默认加载器

自定义 ClassLoader
    └─ 继承 ClassLoader，重写 findClass()
```

### 3.2 JDK 9+ 模块化变化

JDK 9 引入模块系统（JPMS），类加载器结构调整：

| JDK 8 | JDK 9+ |
|-------|--------|
| Bootstrap ClassLoader | Bootstrap ClassLoader（加载 java.base 等） |
| Extension ClassLoader | Platform ClassLoader（加载平台模块） |
| Application ClassLoader | Application ClassLoader |

### 3.3 自定义 ClassLoader

```java
import java.io.*;
import java.nio.file.*;

public class FileClassLoader extends ClassLoader {

    private final String classPath;

    public FileClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] classData = loadClassData(name);
        if (classData == null) {
            throw new ClassNotFoundException(name);
        }
        return defineClass(name, classData, 0, classData.length);
    }

    private byte[] loadClassData(String className) {
        // 将类名转为文件路径，如 com.example.Foo → com/example/Foo.class
        String path = classPath + File.separator +
            className.replace('.', File.separatorChar) + ".class";
        try {
            return Files.readAllBytes(Paths.get(path));
        } catch (IOException e) {
            return null;
        }
    }

    public static void main(String[] args) throws Exception {
        FileClassLoader loader = new FileClassLoader("/tmp/classes");
        Class<?> clazz = loader.loadClass("com.example.Foo");
        Object instance = clazz.newInstance();
        System.out.println(instance.getClass().getName());
    }
}
```

## 四、双亲委派模型

### 4.1 工作原理

```
加载类请求
    │
    ▼
Application ClassLoader
    │ 委托给父加载器
    ▼
Extension ClassLoader
    │ 委托给父加载器
    ▼
Bootstrap ClassLoader
    │ 尝试自己加载
    ├─ 找到 → 返回
    └─ 没找到 → 向下回退
    │
    ▼
Extension ClassLoader
    │ 尝试自己加载
    ├─ 找到 → 返回
    └─ 没找到 → 向下回退
    │
    ▼
Application ClassLoader
    │ 尝试自己加载
    ├─ 找到 → 返回
    └─ 没找到 → ClassNotFoundException
```

### 4.2 ClassLoader.loadClass() 源码

```java
// java.lang.ClassLoader#loadClass
protected Class<?> loadClass(String name, boolean resolve)
    throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 先检查是否已经加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 委托给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // 父加载器为 null 即 Bootstrap
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器无法加载，继续
            }
            if (c == null) {
                // 3. 父加载器失败，自己尝试加载
                c = findClass(name);
            }
        }
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

### 4.3 双亲委派的优点

1. **安全性**：防止自定义类覆盖核心类（如用户无法自定义 `java.lang.Object`，Bootstrap 最先加载）
2. **唯一性**：同一个类只会被加载一次，避免多份 Class 对象

## 五、打破双亲委派

### 5.1 常见打破场景

| 场景 | 说明 |
|------|------|
| **SPI 机制**（JDBC、JNDI） | 核心接口（`java.sql.Driver`）由 Bootstrap 加载，具体实现（`mysql-connector`）在 ClassPath 需要 AppClassLoader 加载 |
| **OSGi 模块化框架** | 每个 bundle 有独立 ClassLoader，动态加载/卸载模块 |
| **热部署/热加载** | Tomcat、Spring DevTools 等，重新加载类时创建新 ClassLoader |
| **JDK 9 模块系统** | 平级委派，不完全遵循双亲委派 |

### 5.2 JDBC SPI 解决方案：线程上下文类加载器

```java
// java.sql.DriverManager 使用线程上下文类加载器加载驱动（JDK SPI 核心代码）
ServiceLoader<Driver> loadedDrivers =
    ServiceLoader.load(Driver.class);
// ServiceLoader.load 内部：
// ClassLoader cl = Thread.currentThread().getContextClassLoader();
// 默认为 AppClassLoader，可加载 ClassPath 下的 SPI 实现类
```

```java
// 手动设置线程上下文类加载器
Thread.currentThread().setContextClassLoader(customClassLoader);
```

### 5.3 Tomcat 的类加载器结构

```
Bootstrap ClassLoader（JDK 核心类）
    │
Extension ClassLoader（JDK 扩展类）
    │
Application ClassLoader（JVM 标准）
    │
Tomcat Common ClassLoader（$CATALINA_HOME/lib/）
    │
Tomcat Catalina ClassLoader（容器自身）
    │
Tomcat WebApp ClassLoader（每个 Web 应用独立）
    └─ /WEB-INF/classes/
    └─ /WEB-INF/lib/*.jar
```

**Tomcat 打破双亲委派的原因：**
- 隔离不同 Web 应用（同名类互不影响）
- Web 应用优先使用自己的类（先 `findClass` 再委托父加载器），实现热部署

```java
// Tomcat WebAppClassLoader 重写了 loadClass，改变委派顺序：
// 1. Bootstrap 加载（安全类）
// 2. 本 WebApp 加载
// 3. 再委托给父加载器
```

## 六、类加载常见问题

### 6.1 ClassNotFoundException vs NoClassDefFoundError

| 异常 | 原因 | 时机 |
|------|------|------|
| `ClassNotFoundException` | 通过类名字符串动态加载类时找不到（如 `Class.forName`） | 运行时 |
| `NoClassDefFoundError` | 编译时存在，运行时 ClassPath 中找不到 | JVM 加载类时 |

### 6.2 类的唯一性

在 JVM 中，类的唯一性由 **ClassLoader + 全限定类名** 共同决定：

```java
// 两个 ClassLoader 加载同一个类文件，产生两个不同的 Class 对象
ClassLoader loader1 = new FileClassLoader("/path1");
ClassLoader loader2 = new FileClassLoader("/path2");
Class<?> c1 = loader1.loadClass("com.example.Foo");
Class<?> c2 = loader2.loadClass("com.example.Foo");

System.out.println(c1 == c2);          // false
System.out.println(c1.equals(c2));     // false
// c1 实例 instanceof c2 也会为 false
```

### 6.3 类的卸载条件

类被卸载需同时满足：
1. 该类的所有实例都已被 GC
2. 该类的 `Class` 对象没有任何引用
3. 加载该类的 `ClassLoader` 已被 GC

因此，JDK 自带的类（由 Bootstrap/Extension 加载）**永远不会被卸载**。

## 七、大厂常见面试题

### Q1：类加载的过程是什么？

**答：**

类加载分为 **加载 → 连接（验证→准备→解析） → 初始化** 三大阶段：

1. **加载**：将 .class 文件的字节流读入方法区，创建 `Class` 对象
2. **验证**：校验字节流格式和语义的正确性
3. **准备**：为类变量分配内存并设置零值（`static final` 常量直接赋值）
4. **解析**：将符号引用转为直接引用（内存地址）
5. **初始化**：执行 `<clinit>` 方法（静态变量赋值 + 静态代码块）

---

### Q2：什么是双亲委派模型？为什么需要它？

**答：**

双亲委派是 ClassLoader 的类加载委托机制：收到类加载请求时，先委托给父加载器，父加载器无法加载时，才由自己加载。

**两大好处：**
1. **安全**：核心类库（`java.lang.*`）由 Bootstrap 优先加载，用户无法用同名类替换，防止 API 被篡改
2. **唯一**：同一个类在 JVM 中只存在一份（同一 ClassLoader 保证），避免多个 Class 对象引发混乱

---

### Q3：Tomcat 为什么要打破双亲委派模型？

**答：**

Tomcat 中多个 Web 应用可能依赖同一个库的不同版本（如 `commons-logging-1.1.jar` 和 `commons-logging-1.2.jar`）。如果遵循双亲委派，一旦父加载器加载了某版本，所有 Web 应用都会使用该版本，无法隔离。

Tomcat 为每个 Web 应用（webapp）创建独立的 `WebAppClassLoader`，优先加载本应用 `/WEB-INF/classes` 和 `/WEB-INF/lib` 下的类，实现**类隔离**和**热部署**（重新创建 ClassLoader 加载新版本类，旧 ClassLoader 等待 GC）。

---

### Q4：如何实现 Java 热部署？

**答：**

热部署核心是**替换 ClassLoader**：

1. 创建新的 `ClassLoader` 实例，重新加载修改后的 `.class` 文件
2. 将新的 ClassLoader 替换旧的，重新实例化需要热更的对象
3. 等待旧 ClassLoader 及其加载的类被 GC（需保证没有强引用持有旧类实例）

**注意事项：**
- 静态变量状态丢失（每次新加载都是全新 Class）
- 对象引用问题（旧实例与新类型不兼容，`instanceof` 为 false）
- 生产中常用 Arthas 热更新（`retransform` 命令）、JRebel、Spring DevTools

```java
// 简化热加载示例
public class HotReload {
    private static ClassLoader loader = createLoader();

    private static ClassLoader createLoader() {
        return new URLClassLoader(new URL[]{Paths.get("/tmp/classes").toUri().toURL()});
    }

    public static void reload() throws Exception {
        // 创建新 ClassLoader
        loader = createLoader();
        // 重新加载目标类
        Class<?> clazz = loader.loadClass("com.example.HotClass");
        // 反射调用新版本的逻辑
        clazz.getMethod("execute").invoke(clazz.newInstance());
    }
}
```

### Q5：`Class.forName` 和 `ClassLoader.loadClass` 的区别？

**答：**

| 方法 | 是否初始化类 | 使用的 ClassLoader |
|------|------------|------------------|
| `Class.forName("...")` | **是**（执行 `<clinit>`） | 调用者的 ClassLoader |
| `Class.forName("...", false, loader)` | 否 | 指定 loader |
| `ClassLoader.loadClass("...")` | **否**（只加载不初始化） | 该 ClassLoader |

**典型案例：**
- JDBC 驱动注册：`Class.forName("com.mysql.cj.jdbc.Driver")` 必须触发初始化，因为驱动在静态块中注册自己到 `DriverManager`。
- Spring 框架懒加载：使用 `ClassLoader.loadClass` 只加载不初始化，延迟到真正使用时才初始化。
