## 二、运行时数据区

JVM 规范定义的运行时数据区分为**线程私有**和**线程共享**两部分：

```
┌──────────────────────────────────────────────────────────┐
│                    运行时数据区                             │
│  ┌──────────────────────────────────────────────────┐    │
│  │                  线程共享区域                      │    │
│  │  ┌────────────────────┐  ┌───────────────────┐   │    │
│  │  │       堆 (Heap)     │  │  方法区/元空间     │   │    │
│  │  │  - 新生代           │  │  - 类信息          │   │    │
│  │  │    - Eden           │  │  - 运行时常量池    │   │    │
│  │  │    - S0/S1          │  │  - 静态变量        │   │    │
│  │  │  - 老年代           │  │  - JIT 编译代码    │   │    │
│  │  └────────────────────┘  └───────────────────┘   │    │
│  └──────────────────────────────────────────────────┘    │
│                                                           │
│  ┌──────────────────────────────────────────────────┐    │
│  │              线程私有区域（每线程独立）              │    │
│  │  ┌──────────┐  ┌──────────────┐  ┌────────────┐  │    │
│  │  │ 程序计数器│  │  虚拟机栈     │  │ 本地方法栈 │  │    │
│  │  │ (PC Reg)  │  │ (VM Stack)   │  │(Native Stk)│  │    │
│  │  └──────────┘  └──────────────┘  └────────────┘  │    │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

### 2.1 程序计数器（Program Counter Register）

- **作用**：记录当前线程执行的字节码指令地址，是线程恢复执行的依据
- **线程私有**：每个线程独立，互不影响
- **特殊**：执行 Native 方法时，程序计数器值为 undefined
- **不会 OOM**：唯一不会发生 OOM 的内存区域

### 2.2 虚拟机栈（VM Stack）

每个方法调用都会创建一个**栈帧（Stack Frame）**，栈帧包含：

```
┌─────────────────────────────────────┐
│           栈帧（Stack Frame）         │
│  ┌─────────────────────────────┐    │
│  │     局部变量表（Local Vars） │    │  ← 基本类型 & 对象引用
│  ├─────────────────────────────┤    │
│  │     操作数栈（Operand Stack）│    │  ← 字节码指令的操作数
│  ├─────────────────────────────┤    │
│  │  动态链接（Dynamic Linking） │    │  ← 符号引用 → 直接引用
│  ├─────────────────────────────┤    │
│  │  方法返回地址（Return Addr） │    │  ← 调用方的 PC 地址
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

- **StackOverflowError**：栈深度超出限制（如无限递归）
- **OutOfMemoryError**：虚拟机栈无法扩展内存时
- JVM 参数：`-Xss`（设置每个线程栈大小，默认 512KB~1MB）

### 2.3 本地方法栈（Native Method Stack）

与虚拟机栈类似，但为 **Native 方法**（JNI 调用的 C/C++ 方法）服务。  
HotSpot JVM 将虚拟机栈与本地方法栈合并为一个。

### 2.4 堆（Heap）

- **最大的内存区域**，所有对象实例和数组都在这里分配
- **线程共享**，GC 主要管理区域
- JVM 参数：`-Xms`（初始堆大小）、`-Xmx`（最大堆大小）

### 2.5 方法区（Method Area）

- JDK 7 及以前：永久代（PermGen），存放类元数据，受 `-XX:MaxPermSize` 限制
- JDK 8+：**元空间（Metaspace）**，使用本地内存（Native Memory），受 `-XX:MaxMetaspaceSize` 限制
- 存放：类的结构信息、常量、静态变量、JIT 编译后的代码

## 三、堆内存详解

### 3.1 堆结构（G1 之前的经典分代）

```
┌──────────────────────────────────────────────────────┐
│                        堆 (Heap)                      │
│  ┌────────────────────────────┐  ┌─────────────────┐ │
│  │          新生代 (Young)     │  │   老年代 (Old)   │ │
│  │  ┌───────┐  ┌────┐  ┌────┐│  │                 │ │
│  │  │ Eden  │  │ S0 │  │ S1 ││  │                 │ │
│  │  │ (80%) │  │(10%)│ │(10%)││  │                 │ │
│  │  └───────┘  └────┘  └────┘│  │                 │ │
│  └────────────────────────────┘  └─────────────────┘ │
│         新生代默认占 1/3                老年代占 2/3     │
└──────────────────────────────────────────────────────┘
```

### 3.2 对象晋升过程

```
对象分配（TLAB 优先在 Eden 分配）
    │
    ▼
Eden 满 → Minor GC（YGC）
    │
    ├─ 存活对象 → S0（年龄+1）
    │
再次 Minor GC
    ├─ S0 存活对象 → S1（年龄+1）
    │
年龄 >= 15（默认）→ 晋升老年代
大对象（>= PretenureSizeThreshold）→ 直接进老年代
```

### 3.3 TLAB（Thread Local Allocation Buffer）

- 每个线程在 Eden 中有私有的小缓冲区（默认 Eden 的 1%）
- 线程分配对象时**无需加锁**，直接在 TLAB 指针上移动
- TLAB 用尽后，在 Eden 中重新申请或触发 GC

### 3.4 堆内存 JVM 参数

```bash
-Xms2g          # 初始堆大小
-Xmx4g          # 最大堆大小（建议与 Xms 相同，避免动态扩容的开销）
-Xmn1g          # 新生代大小（或用 -XX:NewRatio=2 设置老年代/新生代比）
-XX:SurvivorRatio=8   # Eden:S0:S1 = 8:1:1
-XX:MaxTenuringThreshold=15  # 晋升老年代年龄阈值
-XX:PretenureSizeThreshold=1m # 大对象直接进老年代
```

## 四、方法区与元空间

### 4.1 运行时常量池

每个 `.class` 文件都有一个常量池，加载到 JVM 后成为运行时常量池，存储：
- 字面量（字符串字面值、数字常量）
- 符号引用（类/接口名、方法名、字段名）

```java
// 字符串常量池（String Pool）在堆中（JDK 7 移出永久代）
String s1 = "hello";               // 常量池
String s2 = new String("hello");   // 堆中新对象
String s3 = s2.intern();           // 返回常量池中的引用

System.out.println(s1 == s2);  // false
System.out.println(s1 == s3);  // true
```

### 4.2 元空间参数

```bash
-XX:MetaspaceSize=256m         # 初始元空间大小（触发 Full GC 的阈值）
-XX:MaxMetaspaceSize=512m      # 最大元空间大小
-XX:+PrintGCDetails            # 打印 GC 详情
```

## 六、对象的内存布局

### 6.1 布局结构

```
┌─────────────────────────────────────┐
│           对象头（Object Header）    │
│  ┌──────────────────────────────┐   │
│  │  Mark Word（8 bytes）         │   │  ← 哈希码、GC年龄、锁信息
│  ├──────────────────────────────┤   │
│  │  Klass Pointer（4/8 bytes）   │   │  ← 类型指针（指向方法区的类元数据）
│  ├──────────────────────────────┤   │
│  │  数组长度（仅数组对象，4 bytes）│   │
│  └──────────────────────────────┘   │
├─────────────────────────────────────┤
│           实例数据（Instance Data）  │
│  ├─ 父类字段                         │
│  └─ 本类字段                         │
├─────────────────────────────────────┤
│           对齐填充（Padding）         │  ← 使对象大小是 8 字节的倍数
└─────────────────────────────────────┘
```

### 6.2 对象大小计算示例

```java
// 需要 jol-core 依赖
import org.openjdk.jol.info.ClassLayout;

public class ObjectSizeDemo {
    static class A {
        boolean flag;  // 1 byte
        int x;         // 4 bytes
        long y;        // 8 bytes
    }

    public static void main(String[] args) {
        System.out.println(ClassLayout.parseInstance(new A()).toPrintable());
        // 对象头: 12 bytes (Mark Word 8 + Klass Ptr 4，开启压缩指针)
        // flag:   1 byte
        // 填充:   3 bytes
        // x:      4 bytes
        // y:      8 bytes
        // 合计:   28 bytes → 对齐到 32 bytes
    }
}
```

### 6.3 指针压缩

- JVM 默认开启**指针压缩**（`-XX:+UseCompressedOops`）
- 堆 < 32GB 时，将 64 位引用压缩为 32 位，节省内存约 50%
- 堆 > 32GB 时无法使用（建议多个 JVM 实例而非单个超大堆）

## 七、对象的创建过程

```
new 指令
    │
    ▼
① 类加载检查：确认类是否被加载、解析、初始化
    │
    ▼
② 内存分配：
    ├─ 指针碰撞（Bump the Pointer）：堆内存规整时（CMS+Compact/Serial/Parallel）
    └─ 空闲列表（Free List）：堆内存不规整时（CMS）
    优先在 TLAB 中分配
    │
    ▼
③ 初始化零值：对象字段设置默认值（int=0, boolean=false, ref=null）
    │
    ▼
④ 设置对象头：Mark Word（哈希、GC年龄）、Klass Pointer
    │
    ▼
⑤ 执行 <init>：调用构造方法，按代码逻辑初始化字段
```

```java
// 对象创建验证
public class ObjectCreation {
    // 这三行不是原子的，DCL中需要 volatile
    // 1. 分配内存 → 2. 初始化对象 → 3. 引用赋值
    static Object obj = new Object();
}
```

## 八、大厂常见面试题

### Q1：JVM 运行时数据区有哪些？哪些是线程私有的？

**答：**

JVM 运行时数据区共 5 个：

| 区域 | 线程私有/共享 | OOM | SOF |
|------|-------------|-----|-----|
| 程序计数器 | 私有 | ❌ | ❌ |
| 虚拟机栈 | 私有 | ✅ | ✅ |
| 本地方法栈 | 私有 | ✅ | ✅ |
| 堆 | 共享 | ✅ | ❌ |
| 方法区/元空间 | 共享 | ✅ | ❌ |

---

### Q2：什么是栈帧？包含哪些信息？

**答：**

栈帧是虚拟机栈的基本单位，每次方法调用创建一个栈帧并压栈，方法返回时弹出。包含：

1. **局部变量表**：存储方法参数和局部变量（基本类型存值，引用类型存引用）
2. **操作数栈**：字节码指令的操作区域（如 `iadd` 从操作数栈弹出两个整数相加）
3. **动态链接**：运行时将常量池中的符号引用解析为直接引用
4. **方法返回地址**：保存调用方的程序计数器值，用于方法返回后恢复执行

---

### Q3：JDK 7 和 JDK 8 的内存模型有哪些变化？

**答：**

| 变化点 | JDK 7 | JDK 8 |
|--------|-------|-------|
| 永久代 | 存在（`-XX:MaxPermSize`） | 移除 |
| 元空间 | 无 | 使用 Native Memory（`-XX:MaxMetaspaceSize`） |
| 字符串常量池 | 从永久代移到堆 | 在堆中 |
| 原因 | 永久代 OOM 频繁（`java.lang.OutOfMemoryError: PermGen space`） | 元空间使用本地内存，默认无上限，减少 OOM 风险 |

---

### Q4：对象在堆中如何晋升到老年代？

**答：**

五种晋升场景：

1. **年龄阈值**：对象每经历一次 Minor GC 存活，年龄 +1；年龄达到 `-XX:MaxTenuringThreshold`（默认 15）时晋升。
2. **大对象直接进老年代**：超过 `-XX:PretenureSizeThreshold` 的对象直接分配到老年代（避免 Eden 无法容纳）。
3. **动态年龄判断**：Survivor 区中相同年龄所有对象大小之和 > Survivor 空间 50%，则年龄 >= 该年龄的对象晋升（不等年龄到 15）。
4. **Survivor 空间不足**：Minor GC 后 Survivor 无法容纳存活对象，直接进老年代（分配担保）。
5. **空间分配担保失败**：老年代剩余空间 < 历次晋升对象平均大小，触发 Full GC。

### Q5：如何排查 JVM OOM？

**答：**

1. **获取堆转储文件（Heap Dump）：**

```bash
# JVM 启动参数（OOM 时自动转储）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/heapdump.hprof

# 手动导出
jmap -dump:format=b,file=/tmp/heap.hprof <pid>
```

2. **分析工具：**
   - **MAT（Memory Analyzer Tool）**：可视化分析泄漏根因，支持 Dominator Tree
   - **VisualVM**：本地/远程 JVM 监控
   - **JProfiler**：商业工具，功能强大
   - **Arthas**：`heapdump` 命令，`memory` 查看内存区域使用情况

3. **常见 OOM 类型及原因：**

| OOM 类型 | 原因 | 解决 |
|----------|------|------|
| `Java heap space` | 对象太多/内存泄漏 | 扩堆/排查泄漏 |
| `PermGen space`（JDK 7） | 类太多/ClassLoader 泄漏 | 扩 PermGen/排查泄漏 |
| `Metaspace`（JDK 8+） | 类加载过多（如字节码生成框架） | 扩 Metaspace/排查动态类 |
| `GC overhead limit exceeded` | 98% 时间在 GC，只回收 2% 堆 | 排查内存泄漏，扩堆 |
| `unable to create new native thread` | 线程数超过 OS 限制 | 减少线程/使用线程池 |
