## 一、JVM 内存结构

### Q1：JVM 的运行时数据区包含哪些？各有什么作用？

**答：**

| 区域 | 线程 | 作用 | OOM/SOF |
|------|------|------|---------|
| **程序计数器** | 私有 | 记录当前线程执行的字节码行号 | 无 |
| **虚拟机栈** | 私有 | 每次方法调用创建栈帧（局部变量表、操作数栈、动态链接、返回地址） | OOM/SOF |
| **本地方法栈** | 私有 | 为 Native 方法服务 | OOM/SOF |
| **堆** | 共享 | 存放所有对象实例和数组，GC 的主要管理区域 | OOM |
| **方法区/元空间** | 共享 | 存放类元数据、常量、静态变量、JIT 编译代码 | OOM |

补充：JDK 8 用**元空间（Metaspace）**替换了永久代（PermGen），使用本地内存，减少 OOM。

---

### Q2：堆和栈的区别？

**答：**

| 维度 | 堆（Heap） | 栈（VM Stack） |
|------|-----------|--------------|
| 线程归属 | 线程共享 | 线程私有 |
| 存储内容 | 对象实例、数组 | 局部变量、方法参数、返回地址 |
| 内存管理 | GC 负责回收 | 随方法调用/返回自动 push/pop |
| 大小 | 通过 `-Xms/-Xmx` 指定 | 通过 `-Xss` 指定（每个线程） |
| 速度 | 较慢（GC 开销） | 极快（指针碰撞） |
| 生命周期 | 由 GC 决定 | 随方法生命周期 |

---

### Q3：什么是运行时常量池？字符串常量池在哪里？

**答：**

- **运行时常量池**：每个 Class 文件有一个常量池，加载后存入方法区的运行时常量池，包含字面量和符号引用。JDK 8 存于元空间（Metaspace）。

- **字符串常量池（String Pool）**：
  - JDK 6：在永久代（PermGen）
  - JDK 7+：移至**堆中**（避免 PermGen OOM）

```java
String s1 = "hello";                    // 常量池
String s2 = new String("hello");        // 堆（新对象）
String s3 = s2.intern();               // 返回常量池中的引用

System.out.println(s1 == s2);  // false（不同对象）
System.out.println(s1 == s3);  // true（同一常量池引用）
```

---

### Q4：对象在 JVM 中的内存布局是什么？

**答：**

对象由三部分组成：

1. **对象头（Header）**：
   - **Mark Word（8字节）**：存储哈希码、GC 年龄、锁状态（无锁/偏向/轻量/重量）
   - **Klass Pointer（4字节，压缩指针）**：指向类的元数据
   - 数组对象额外有 4 字节的数组长度

2. **实例数据（Instance Data）**：实例字段值（父类字段在前）

3. **对齐填充（Padding）**：使对象大小为 8 字节的倍数

最小空对象大小：16 字节（12字节对象头 + 4字节填充）。

## 二、垃圾回收

### Q5：GC 判断对象是否存活的方法有哪些？为什么 JVM 不用引用计数法？

**答：**

**两种方法：**
1. **引用计数法**：每个对象维护引用计数，归 0 时回收。JVM 不采用，原因是无法解决**循环引用**（A→B、B→A，计数各为 1，永不回收）。
2. **可达性分析（GC Roots Tracing）**：从 GC Roots 出发遍历引用链，不可达对象判定为垃圾。JVM 使用此方法。

**GC Roots** 包括：活跃线程的栈帧局部变量、类静态变量、常量、JNI 引用、被 synchronized 持有的对象等。

---

### Q6：简述 GC 算法及各自优缺点？

**答：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **标记-清除** | 标记存活对象，回收未标记对象 | 简单 | 内存碎片 |
| **标记-复制** | 存活对象复制到空白区域 | 无碎片、分配快 | 内存利用率 50% |
| **标记-整理** | 存活对象移向一端，清除边界外 | 无碎片、利用率高 | 移动对象代价大 |
| **分代收集** | 结合以上算法，新生代复制、老年代整理 | 综合最优 | 复杂 |

---

### Q7：G1 和 CMS 的区别是什么？为什么推荐用 G1？

**答：**

| 维度 | CMS | G1 |
|------|-----|-----|
| 适用版本 | JDK 6~13（JDK 14 废弃） | JDK 7+（JDK 9 默认） |
| 内存碎片 | 有（标记-清除） | 无（Region 内复制） |
| 停顿可预测 | 否 | **是**（`-XX:MaxGCPauseMillis`） |
| 内存占用 | 低 | 较高（Remember Sets 等辅助结构） |
| 并发模式失败 | 可能发生 | 极少（G1 保留预留空间） |
| 适用堆 | 4~8GB | **6GB+ 推荐** |

**推荐 G1 的原因：**
1. 无内存碎片，无需定期整理
2. 可设置期望最大停顿时间，适合 Web 服务
3. 大堆下比 CMS 更稳定，Concurrent Mode Failure 风险低
4. JDK 9+ 默认收集器，社区持续维护

---

### Q8：什么是 Minor GC、Major GC、Full GC？Full GC 的触发条件？

**答：**

- **Minor GC**：回收新生代（Eden 区满时触发），STW 时间短（毫秒级）
- **Major GC**：回收老年代（通常伴随 Full GC，或 CMS 独立触发）
- **Full GC**：回收整个堆 + 方法区，STW 时间最长

**Full GC 触发条件：**
1. 老年代空间不足（晋升失败或大对象直接进老年代）
2. 元空间/永久代不足
3. `System.gc()`（建议性调用，通常会触发）
4. CMS 并发模式失败（Concurrent Mode Failure）
5. Minor GC 后 Survivor 不足，老年代担保空间不足

---

### Q9：什么是对象晋升？动态年龄判断是什么？

**答：**

**晋升机制：**
1. 对象经历一次 Minor GC 存活，年龄 +1，达到 `MaxTenuringThreshold`（默认15）后晋升
2. 大对象（> `PretenureSizeThreshold`）直接进老年代

**动态年龄判断：** Survivor 中相同年龄所有对象大小之和 > Survivor 总容量的 50%，则年龄 >= 该值的对象全部晋升，不必等到 15 岁。这是对"长期存活对象"的提前识别。

## 三、类加载机制

### Q10：类加载的过程分几步？

**答：**

共 7 步（加载→验证→准备→解析→初始化→使用→卸载），核心是前 5 步：

1. **加载**：读取字节流，在堆中创建 `Class` 对象
2. **验证**：校验字节流合法性（文件格式/元数据/字节码/符号引用）
3. **准备**：为类变量分配内存并赋**零值**（`static int` = 0，`static final` 直接赋值）
4. **解析**：符号引用 → 直接引用
5. **初始化**：执行 `<clinit>`（static 变量赋值 + static 代码块）

---

### Q11：什么是双亲委派？为什么需要它？如何打破它？

**答：**

**双亲委派：** 类加载时，先委托父加载器，父加载器加载失败，才由自己加载。

**好处：** ① 防止核心类被替换（安全）；② 同一类只加载一次（唯一性）。

**打破场景：**
- **SPI 机制**：`java.sql.Driver`（Bootstrap 加载）需要加载 ClassPath 下的实现（AppClassLoader），使用线程上下文类加载器（Thread Context ClassLoader）解决
- **Tomcat**：每个 WebApp 有独立 ClassLoader，应用类优先加载，实现隔离和热部署
- **OSGi**：每个 bundle 独立 ClassLoader，网状委派而非树形

---

### Q12：`Class.forName()` 和 `ClassLoader.loadClass()` 的区别？

**答：**

| 方法 | 是否执行初始化（`<clinit>`） | 默认 ClassLoader |
|------|--------------------------|----------------|
| `Class.forName(name)` | **是** | 调用者的 ClassLoader |
| `Class.forName(name, false, loader)` | 否（可控制） | 指定 loader |
| `ClassLoader.loadClass(name)` | **否** | 该 ClassLoader |

**典型案例：** JDBC 驱动注册必须用 `Class.forName`，因为驱动在 static 块中向 `DriverManager` 注册自己，必须触发初始化。

## 四、JIT 编译与优化

### Q13：JIT 编译器有哪些优化手段？

**答：**

JIT 主要优化手段（C2 编译器）：

1. **方法内联**：将被调用方法体嵌入调用处，消除方法调用开销，为其他优化创造条件
2. **逃逸分析**：分析对象作用域，不逃逸的对象可进行栈上分配、标量替换、锁消除
3. **锁优化**：锁消除（不逃逸对象的同步）、锁粗化（合并连续的加解锁）
4. **公共子表达式消除（CSE）**：相同子表达式只计算一次
5. **循环展开**：减少循环控制开销
6. **向量化**：利用 SIMD 指令并行处理多个数据
7. **分支预测优化**：根据 profiling 将热路径放到快路径
8. **去虚化（Devirtualization）**：虚方法调用特化为直接调用

---

### Q14：什么是栈上分配？和对象在堆上分配有什么区别？

**答：**

- **堆上分配（默认）**：所有 `new` 对象都在堆上，需要 GC 回收，有一定开销
- **栈上分配（逃逸分析优化）**：若对象不逃逸出方法，JIT 可以在**栈帧**中分配该对象，随方法返回自动销毁，不产生 GC 压力

实际上 HotSpot 更常用的是**标量替换**（将对象分解为基本字段，分配到栈或寄存器），而非完整的栈上分配。

```bash
# 对比测试：关闭逃逸分析，观察 GC 压力
java -XX:-DoEscapeAnalysis -XX:+PrintGCDetails ...
java -XX:+DoEscapeAnalysis -XX:+PrintGCDetails ...  # 默认开启
```

## 五、JVM 调优与故障诊断

### Q15：线上 CPU 100%，如何快速排查？

**答：**

标准四步排查法：

```bash
# Step 1: 找出 CPU 最高的 Java 进程
top   # 记录 PID

# Step 2: 找出该进程中最高 CPU 的线程
top -H -p <PID>
# 记录最高 CPU 的 TID（十进制）

# Step 3: TID 转十六进制
printf '%x\n' <TID>  # 如 1234 → 0x4d2

# Step 4: jstack 定位代码
jstack <PID> | grep -A 30 "nid=0x4d2"
```

**Arthas 更快捷：** `thread -n 5`（直接显示 Top5 CPU 线程及堆栈）

**常见原因：** 死循环、频繁 GC（GC 线程 CPU 高）、正则回溯、加解密计算

---

### Q16：线上出现 OOM，如何排查？

**答：**

1. **配置 OOM 自动 Heap Dump**（提前配置）：
   ```bash
   -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/heapdump.hprof
   ```

2. **分析 Heap Dump（MAT）**：
   - Dominator Tree：找最大内存占用对象
   - Leak Suspects：自动分析嫌疑对象
   - Incoming References：找引用链，定位为什么未被 GC

3. **jmap 实时分析**：
   ```bash
   jmap -histo:live <pid> | head -30  # 查看存活对象按大小排序
   ```

4. **常见原因**：静态集合无限增长、ThreadLocal 未 remove、连接/流未关闭、缓存无限制增长

---

### Q17：如何分析 GC 日志？关注哪些指标？

**答：**

**关键指标：**

| 指标 | 含义 | 健康值 |
|------|------|--------|
| YGC 频率 | Minor GC 次数/分钟 | < 10次/分 |
| YGC 停顿 | 单次 Minor GC 时间 | < 50ms |
| Full GC 频率 | 每天 Full GC 次数 | 0 次（理想）/ < 1次/天 |
| Full GC 停顿 | 单次 Full GC 时间 | < 1s |
| 老年代使用率 | Old Gen 使用/总大小 | < 70% |

**工具：**
- `jstat -gcutil <pid> 1000`：实时监控
- GCEasy（在线）/ GCViewer（本地）：可视化 GC 日志

## 六、综合与进阶

### Q18：什么是 Safe Point？什么是 Safe Region？

**答：**

- **Safe Point（安全点）**：程序执行的特定位置，JVM 可以在此暂停线程进行 GC（STW）。不是所有地方都能停，只在：方法调用、循环回跳、异常跳转等字节码处。线程进入安全点后将状态标记为"已挂起"，GC 等待所有线程到达安全点。

- **Safe Region（安全区域）**：代码区域内，引用关系不会变化（如线程 `sleep`、`blocked` 状态）。GC 不必等待该区域的线程到达安全点，直接视为已安全。线程离开安全区域时需检查是否可以继续执行（GC 是否完成）。

---

### Q19：Java 中的四种引用类型及应用场景？

**答：**

| 类型 | 回收时机 | 实现类 | 应用场景 |
|------|---------|--------|---------|
| **强引用** | 永不（OOM 也不回收） | 普通引用 `Object o = new Object()` | 默认 |
| **软引用** | 内存不足时 | `SoftReference<T>` | 内存敏感的缓存（如图片缓存）|
| **弱引用** | 下次 GC 时 | `WeakReference<T>` | 缓存（`WeakHashMap`）、ThreadLocal key |
| **虚引用** | 随时，无法获取对象 | `PhantomReference<T>` | 跟踪 GC 通知（配合 ReferenceQueue），管理堆外内存（DirectByteBuffer）|

---

### Q20：为什么 String 被设计为不可变的？有哪些好处？

**答：**

1. **安全性**：字符串常用于网络连接 URL、数据库密码、类路径等，不可变防止被篡改
2. **字符串常量池**：不可变才能安全共享（多个变量可以指向同一字符串实例），节省内存
3. **哈希缓存**：`String.hashCode()` 结果可以缓存（`hash` 字段），因为值不会变，HashMap 的 key 性能更好
4. **线程安全**：不可变对象天然线程安全，无需同步

**实现方式：**
- `private final char[] value`（JDK 8）/ `private final byte[] value`（JDK 9+，Latin-1 压缩）
- `final` 类，不可被继承
