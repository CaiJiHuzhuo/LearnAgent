## 一、为什么要分库分表

### 1.1 单库瓶颈

| 瓶颈 | 表现 | 解决方案 |
|------|------|---------|
| 数据量大 | 单表超千万行，查询变慢 | **分表** |
| 并发量高 | 数据库连接数不够 | **分库** |
| 磁盘不足 | 单机磁盘容量有限 | **分库** |

### 1.2 分库分表 vs 其他方案

| 方案 | 适用场景 | 复杂度 |
|------|---------|--------|
| **读写分离** | 读多写少 | 低 |
| **分区表** | 单表数据大，按时间分区 | 低 |
| **垂直分库** | 业务解耦 | 中 |
| **水平分库分表** | 数据量和并发都大 | 高 |

## 二、拆分策略

### 2.1 垂直拆分

**垂直分库**：按业务拆分到不同数据库

```
原始数据库
├── t_user
├── t_order
├── t_product
└── t_payment

拆分后：
用户库  → t_user
订单库  → t_order
商品库  → t_product
支付库  → t_payment
```

**垂直分表**：将一个宽表拆分为多个表

```
t_user (id, name, age, avatar, bio, address, ...)
    ↓
t_user (id, name, age)               -- 常用字段
t_user_detail (user_id, avatar, bio, address)  -- 不常用字段
```

### 2.2 水平拆分

**水平分表**：将同一张表的数据按规则拆分到多张表

```
t_order → t_order_0, t_order_1, t_order_2, t_order_3
```

**水平分库**：将同一张表的数据按规则拆分到多个数据库

```
db_0.t_order, db_1.t_order, db_2.t_order, db_3.t_order
```

## 三、分片算法

### 3.1 常用分片算法

| 算法 | 说明 | 适用场景 |
|------|------|---------|
| **哈希取模** | `hash(shardKey) % N` | 数据均匀分布 |
| **范围分片** | 按 ID 范围或时间范围 | 时间序列数据 |
| **一致性哈希** | 哈希环 | 需要动态扩容 |
| **查表法** | 通过映射表查找 | 灵活但多一次查询 |

### 3.2 分片键选择原则

1. **高频查询条件**：分片键应该是最常用的查询条件
2. **数据均匀分布**：避免数据倾斜
3. **尽量避免跨片查询**

### 3.3 哈希取模示例

```
分片键：user_id
分片数：4

user_id = 1001 → 1001 % 4 = 1 → db_1.t_order
user_id = 1002 → 1002 % 4 = 2 → db_2.t_order
user_id = 1003 → 1003 % 4 = 3 → db_3.t_order
user_id = 1004 → 1004 % 4 = 0 → db_0.t_order
```

## 四、中间件选型

### 4.1 主流方案对比

| 中间件 | 类型 | 特点 |
|--------|------|------|
| **ShardingSphere-JDBC** | 客户端代理 | 轻量级，Java 应用内嵌 |
| **ShardingSphere-Proxy** | 服务端代理 | 独立部署，对应用透明 |
| **MyCat** | 服务端代理 | 基于 Cobar，社区活跃度下降 |

### 4.2 ShardingSphere 配置示例

```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        url: jdbc:mysql://host1:3306/db0
        username: root
        password: password
      ds1:
        url: jdbc:mysql://host2:3306/db1
        username: root
        password: password
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds$->{0..1}.t_order_$->{0..3}
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: db-hash-mod
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: table-hash-mod
        sharding-algorithms:
          db-hash-mod:
            type: HASH_MOD
            props:
              sharding-count: 2
          table-hash-mod:
            type: HASH_MOD
            props:
              sharding-count: 4
```

## 五、常见问题与解决方案

### 5.1 分布式 ID

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID** | 简单，无中心化 | 无序，索引性能差 |
| **雪花算法** | 有序、高性能 | 依赖时钟 |
| **号段模式** | 数据库生成，批量获取 | 依赖数据库 |
| **Leaf（美团）** | 号段 + 雪花双模式 | 需部署服务 |

### 5.2 跨片查询

| 问题 | 解决方案 |
|------|---------|
| 跨片 JOIN | 全局表 / 冗余字段 / 应用层组装 |
| 跨片聚合 | 各分片查询后应用层合并 |
| 跨片排序 | 各分片排序后归并排序 |
| 跨片分页 | 各分片查 offset+limit 行，应用层合并排序取 limit |

### 5.3 分布式事务

参考 Spring 事务管理章节中的分布式事务方案（Seata AT/TCC、本地消息表等）。

### 5.4 数据迁移

```
扩容迁移步骤：
1. 双写：新数据同时写入新旧库
2. 数据迁移：历史数据从旧库迁移到新库
3. 数据校验：对比新旧库数据一致性
4. 切换读：读流量切到新库
5. 停止双写：切断旧库写入
```

## 六、大厂常见面试题

### Q1：什么时候需要分库分表？

**答：**
- 单表数据量超过 **1000 万行**或 **10GB**，查询明显变慢
- 数据库并发连接数不够
- 先尝试：索引优化 → 读写分离 → 缓存 → 最后才分库分表

### Q2：水平分表和垂直分表的区别？

**答：**
- **垂直分表**：按列拆分，常用字段和不常用字段分开
- **水平分表**：按行拆分，同结构的表按规则分散数据
- **垂直分库**：按业务拆分到不同数据库
- **水平分库**：同一张表数据分散到多个数据库

### Q3：分片键如何选择？

**答：** 选择最常用的查询条件作为分片键（如 user_id、order_id），确保数据均匀分布，尽量让大部分查询在单片内完成，避免跨片查询。

### Q4：分库分表后如何解决分布式 ID？

**答：** 推荐使用**雪花算法**（时间戳 + 机器 ID + 序列号），生成全局唯一且趋势递增的 ID。也可以使用美团 Leaf（号段模式 + 雪花算法双模式）。

### Q5：分库分表后如何处理跨片查询？

**答：**
- **JOIN**：全局表冗余、字段冗余、应用层组装
- **聚合/排序**：各分片独立查询，应用层归并
- **分页**：各分片查 offset+limit 行，应用层合并后再分页（深分页问题用游标）
