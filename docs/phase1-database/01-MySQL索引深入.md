## 一、索引概述

### 1.1 什么是索引

索引是帮助 MySQL **高效获取数据**的**有序数据结构**。类似书籍目录，通过索引可以快速定位数据，而不用全表扫描。

### 1.2 索引的优缺点

| 优点 | 缺点 |
|------|------|
| 加快查询速度 | 占用额外磁盘空间 |
| 加速排序和分组 | 降低增删改性能（维护索引） |
| 保证数据唯一性（唯一索引） | 过多索引增加优化器选择成本 |

## 二、索引数据结构

### 2.1 B+ 树（核心考点 ⭐⭐⭐）

MySQL InnoDB 默认使用 **B+ 树**作为索引数据结构。

**B+ 树 vs B 树：**

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点存数据 | 只有叶子节点存数据 |
| 叶子节点链表 | 无 | 有（双向链表） |
| 查询稳定性 | 不稳定（可能在非叶子找到） | 稳定（必须到叶子节点） |
| 范围查询 | 需要中序遍历 | 沿叶子链表扫描，高效 |
| 磁盘 IO | 更多（非叶子也存数据，节点大） | 更少（非叶子只存键，扇出更大） |

### 2.2 B+ 树结构

```
                [10 | 20 | 30]              ← 非叶子节点（只存键值）
               /    |     |    \
    [1,3,5,8] → [10,12,15] → [20,22,25] → [30,35,40]  ← 叶子节点（存数据，双向链表）
```

**为什么用 B+ 树而不用：**

| 结构 | 不适合原因 |
|------|----------|
| 哈希表 | 不支持范围查询、排序 |
| 二叉搜索树 | 极端情况退化为链表 |
| AVL/红黑树 | 树高度大，磁盘 IO 多 |
| B 树 | 非叶子存数据，扇出小；不支持高效范围查询 |

### 2.3 InnoDB 一页 16KB 能存多少索引？

- 非叶子节点：主键 8B + 指针 6B = 14B → 16KB / 14B ≈ **1170** 个索引项
- 三层 B+ 树可存：1170 × 1170 × 16 ≈ **2000 万**行数据
- 这就是为什么 InnoDB B+ 树通常只有 **3-4 层**

## 三、索引类型

### 3.1 按数据结构分

| 类型 | 说明 | 引擎 |
|------|------|------|
| **B+ 树索引** | 默认，支持范围查询 | InnoDB、MyISAM |
| **哈希索引** | 精确匹配快，不支持范围 | Memory、InnoDB 自适应哈希 |
| **全文索引** | 文本搜索 | InnoDB（5.6+）、MyISAM |

### 3.2 按功能分

| 类型 | 说明 |
|------|------|
| **主键索引** | PRIMARY KEY，唯一 + 非空，聚簇索引 |
| **唯一索引** | UNIQUE，值唯一，允许 NULL |
| **普通索引** | INDEX，无唯一性约束 |
| **组合索引** | 多列组成的索引 |
| **前缀索引** | 对字符串前 N 个字符建索引 |

### 3.3 聚簇索引 vs 非聚簇索引（核心考点 ⭐⭐⭐）

| 特性 | 聚簇索引 | 非聚簇索引（二级索引） |
|------|---------|---------------------|
| 叶子节点 | 存储完整行数据 | 存储主键值 |
| 数量 | 每张表只有一个 | 可以有多个 |
| 默认 | 主键索引 | 其他索引 |
| 查询 | 直接获取数据 | 可能需要回表 |

**回表查询：**
```
1. 通过二级索引找到主键值
2. 用主键值到聚簇索引中查找完整行数据
这个过程叫"回表"
```

## 四、索引使用原则

### 4.1 适合建索引的场景

1. WHERE 条件中频繁使用的列
2. ORDER BY / GROUP BY 的列
3. JOIN 关联的列
4. 高选择性（区分度高）的列
5. 频繁查询但较少修改的列

### 4.2 不适合建索引的场景

1. 数据量小的表（全表扫描更快）
2. 频繁增删改的列
3. 区分度低的列（如性别：男/女）
4. WHERE 中用不到的列

### 4.3 组合索引与最左前缀原则（核心考点 ⭐⭐⭐）

```sql
-- 组合索引 (a, b, c)
CREATE INDEX idx_abc ON t_user(a, b, c);

-- ✅ 走索引
WHERE a = 1                      -- 使用 a
WHERE a = 1 AND b = 2            -- 使用 a, b
WHERE a = 1 AND b = 2 AND c = 3 -- 使用 a, b, c
WHERE a = 1 AND c = 3            -- 使用 a（c 不连续，MySQL 8 索引跳跃扫描可能优化）

-- ❌ 不走索引
WHERE b = 2                      -- 缺少最左列 a
WHERE b = 2 AND c = 3            -- 缺少最左列 a
WHERE c = 3                      -- 缺少最左列 a
```

## 五、索引失效场景

### 5.1 常见失效场景（核心考点 ⭐⭐⭐）

| 序号 | 场景 | 示例 |
|------|------|------|
| 1 | 对索引列使用函数 | `WHERE YEAR(create_time) = 2024` |
| 2 | 对索引列做运算 | `WHERE age + 1 = 20` |
| 3 | 隐式类型转换 | `WHERE phone = 13800138000`（phone 是 varchar） |
| 4 | LIKE 左模糊 | `WHERE name LIKE '%张'` |
| 5 | 使用 OR 且有一边无索引 | `WHERE a = 1 OR b = 2`（b 无索引） |
| 6 | 不满足最左前缀 | 组合索引 (a,b,c)，只查 b |
| 7 | IS NOT NULL | 某些情况下 `WHERE col IS NOT NULL` 不走索引 |
| 8 | 使用 != / NOT IN | 优化器可能选择全表扫描 |

### 5.2 避免索引失效的建议

```sql
-- ❌ 函数导致失效
WHERE DATE(create_time) = '2024-01-01'
-- ✅ 改为范围查询
WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02'

-- ❌ 隐式转换
WHERE phone = 13800138000
-- ✅ 类型匹配
WHERE phone = '13800138000'

-- ❌ 左模糊
WHERE name LIKE '%张'
-- ✅ 右模糊
WHERE name LIKE '张%'
```

## 六、覆盖索引与索引下推

### 6.1 覆盖索引（核心考点 ⭐⭐⭐）

查询的列全部在索引中，不需要回表。

```sql
-- 组合索引 idx_name_age (name, age)

-- ✅ 覆盖索引（Extra: Using index）
SELECT name, age FROM t_user WHERE name = '张三';

-- ❌ 需要回表（还需要 email 列）
SELECT name, age, email FROM t_user WHERE name = '张三';
```

### 6.2 索引下推（ICP, Index Condition Pushdown）

MySQL 5.6+ 特性，在索引遍历时先过滤不满足条件的记录，减少回表次数。

```sql
-- 组合索引 idx_name_age (name, age)
SELECT * FROM t_user WHERE name LIKE '张%' AND age = 20;

-- 无 ICP：用索引找 name LIKE '张%' 的记录 → 全部回表 → 再过滤 age = 20
-- 有 ICP：用索引找 name LIKE '张%' 的记录 → 在索引中过滤 age = 20 → 只对满足的回表
```

## 七、EXPLAIN 执行计划

### 7.1 核心字段

| 字段 | 说明 | 关注点 |
|------|------|--------|
| **type** | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| **key** | 实际使用的索引 | NULL 表示没用索引 |
| **rows** | 预估扫描行数 | 越小越好 |
| **Extra** | 额外信息 | Using index（覆盖索引）、Using filesort（需优化） |

### 7.2 type 详解

| type | 说明 | 性能 |
|------|------|------|
| **const** | 主键/唯一索引精确匹配 | 极好 |
| **eq_ref** | JOIN 时主键/唯一索引匹配 | 很好 |
| **ref** | 非唯一索引等值查询 | 好 |
| **range** | 索引范围扫描 | 较好 |
| **index** | 全索引扫描 | 一般 |
| **ALL** | 全表扫描 | 差 |

### 7.3 Extra 关键信息

| Extra | 说明 |
|-------|------|
| **Using index** | 覆盖索引，不需要回表 ✅ |
| **Using where** | 服务层过滤 |
| **Using index condition** | 索引下推（ICP） ✅ |
| **Using filesort** | 额外排序，需优化 ⚠️ |
| **Using temporary** | 使用临时表，需优化 ⚠️ |

## 八、大厂常见面试题

### Q1：MySQL 为什么使用 B+ 树作为索引结构？

**答：**
1. B+ 树非叶子节点只存键，扇出大，树高度低（3-4 层），**磁盘 IO 少**
2. 叶子节点双向链表连接，**范围查询高效**
3. 所有查询都到叶子节点，**查询性能稳定**
4. 对比：哈希不支持范围查询；红黑树太高；B 树非叶子存数据，扇出小

### Q2：什么是聚簇索引和非聚簇索引？什么是回表？

**答：**
- **聚簇索引**：叶子节点存储完整行数据，每表只有一个（主键索引）
- **非聚簇索引**：叶子节点存储主键值
- **回表**：通过非聚簇索引找到主键，再用主键到聚簇索引查完整数据

### Q3：什么是覆盖索引？

**答：** 查询的列都包含在索引中，直接从索引获取数据，不需要回表。EXPLAIN 中 Extra 显示 `Using index`。设计组合索引时尽量覆盖常用查询的列。

### Q4：索引失效的常见场景？

**答：** 对索引列使用函数/运算、隐式类型转换、LIKE 左模糊、OR 有一边无索引、不满足最左前缀、使用 != / NOT IN（优化器判断全扫更快）。

### Q5：组合索引的最左前缀原则？

**答：** 查询条件必须从组合索引的最左列开始，且不能跳过中间列。索引 (a,b,c) 可优化 a、ab、abc 的查询，不能优化 b、bc、c 的查询。
