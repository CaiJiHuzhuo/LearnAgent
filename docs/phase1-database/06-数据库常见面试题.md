## 一、MySQL 索引

### Q1：MySQL 为什么使用 B+ 树而不是 B 树？

**答：**
1. B+ 树非叶子节点只存键值，扇出更大，树更矮，**磁盘 IO 更少**
2. 叶子节点形成双向链表，**范围查询高效**
3. 所有查询都到叶子节点，**查询性能稳定**

### Q2：什么是聚簇索引和非聚簇索引？

**答：**
- **聚簇索引**：叶子节点存储完整行数据（主键索引），每表一个
- **非聚簇索引**：叶子节点存储主键值（二级索引），查询需回表
- **回表**：通过二级索引找到主键，再到聚簇索引查完整数据

### Q3：什么是覆盖索引？

**答：** 查询的所有列都在索引中，无需回表。EXPLAIN 中 Extra 显示 `Using index`。

### Q4：组合索引的最左前缀原则？

**答：** 组合索引 (a,b,c) 相当于创建了 (a)、(a,b)、(a,b,c) 三个索引。查询条件必须从最左列开始使用。

### Q5：索引失效的常见场景？

**答：**
1. 对索引列使用函数/运算
2. 隐式类型转换（varchar 列用数字查询）
3. LIKE 左模糊（'%xxx'）
4. OR 有一边无索引
5. 不满足最左前缀原则
6. 使用 != 或 NOT IN（优化器可能全扫）

## 二、MySQL 事务与锁

### Q6：MySQL 的隔离级别？默认是什么？

**答：** READ UNCOMMITTED、READ COMMITTED、**REPEATABLE READ**（默认）、SERIALIZABLE。MySQL 的 RR 通过 MVCC + Next-Key Lock 基本解决了幻读。

### Q7：MVCC 的实现原理？

**答：**
- 每行隐藏字段：`trx_id`（事务 ID）、`roll_pointer`（回滚指针）
- **undo log 版本链**：同一行数据的多个版本
- **ReadView**：记录活跃事务列表，判断版本可见性
- RC 每次 SELECT 创建新 ReadView；RR 第一次 SELECT 创建后复用

### Q8：InnoDB 的行锁类型？

**答：**
- **Record Lock**：锁定单条记录
- **Gap Lock**：锁定索引间隙（防止插入）
- **Next-Key Lock**：Record + Gap（左开右闭区间）
- RR 级别默认用 Next-Key Lock 防止幻读

### Q9：什么是死锁？如何避免？

**答：** 多个事务互相等待对方的锁。避免：固定加锁顺序、缩短事务、合理索引减少锁范围。InnoDB 默认开启死锁检测自动回滚。

### Q10：乐观锁和悲观锁的区别？

**答：**
| 维度 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| 实现 | 版本号/CAS | SELECT ... FOR UPDATE |
| 适用 | 读多写少 | 写多读少 |
| 死锁 | 不会 | 可能 |
| 性能 | 无锁开销，冲突重试 | 加锁等待 |

## 三、SQL 优化

### Q11：如何定位慢 SQL？

**答：**
1. 开启慢查询日志（`slow_query_log`）
2. 使用 APM 工具（SkyWalking/Arthas）监控
3. `EXPLAIN` 分析执行计划
4. 关注 type（至少 range）、key、rows、Extra

### Q12：EXPLAIN 中哪些字段最重要？

**答：**
- **type**：访问类型，ALL（全扫描）→ const（最优）
- **key**：实际使用的索引
- **rows**：预估扫描行数
- **Extra**：`Using index`（好）、`Using filesort`（需优化）

### Q13：深分页如何优化？

**答：**
1. **延迟关联**：子查询走索引取 ID，再关联取完整数据
2. **游标分页**：`WHERE id > lastId LIMIT n`
3. 复杂搜索用 Elasticsearch

### Q14：COUNT(*) 和 COUNT(1) 和 COUNT(列) 的区别？

**答：** `COUNT(*)` = `COUNT(1)` 性能相同（MySQL 已优化）。`COUNT(列)` 排除 NULL 值。大表推荐维护计数器。

## 四、分库分表

### Q15：什么时候需要分库分表？

**答：** 单表超 1000 万行或 10GB 查询变慢时。先尝试索引优化 → 读写分离 → 缓存 → 最后才分库分表。

### Q16：分片键如何选择？

**答：** 选最常用查询条件（如 user_id），保证数据均匀分布，减少跨片查询。

### Q17：分库分表后的分布式 ID 方案？

**答：**
- **雪花算法**：有序、高性能、分布式唯一（推荐）
- **号段模式**：数据库批量获取 ID
- **Leaf（美团）**：支持号段和雪花两种模式

### Q18：分库分表带来的问题？

**答：** 跨片查询（JOIN/聚合/排序）、分布式事务、分布式 ID、数据迁移扩容。解决方案：全局表/冗余字段、Seata 分布式事务、雪花 ID、双写迁移。

## 五、Redis 基础

### Q19：Redis 为什么快？

**答：** 内存存储、单线程无竞争、IO 多路复用（epoll）、高效数据结构。

### Q20：Redis 的数据类型和应用场景？

**答：**
| 类型 | 场景 |
|------|------|
| String | 缓存、计数器、分布式锁 |
| Hash | 对象缓存 |
| List | 消息队列、最新列表 |
| Set | 标签、共同关注 |
| ZSet | 排行榜、延迟队列 |

### Q21：Redis 的持久化方式？

**答：**
- **RDB**：内存快照，恢复快但可能丢数据
- **AOF**：命令追加，数据安全但文件大
- **混合持久化**（推荐）：RDB + 增量 AOF

### Q22：Redis 过期策略和内存淘汰策略？

**答：**
- **过期删除**：惰性删除（访问时检查）+ 定期删除（随机抽样）
- **内存淘汰策略**：
  - `noeviction`：不淘汰，内存满拒绝写入
  - `allkeys-lru`：所有 key 中淘汰最近最少使用的（推荐）
  - `volatile-lru`：设了过期时间的 key 中淘汰 LRU
  - `allkeys-lfu`：淘汰使用频率最低的
  - `volatile-ttl`：淘汰即将过期的

## 六、Redis 缓存与高可用

### Q23：缓存穿透、击穿、雪崩的区别？

**答：**
| 问题 | 原因 | 方案 |
|------|------|------|
| **穿透** | 查不存在的数据 | 布隆过滤器/缓存空值 |
| **击穿** | 热点 Key 过期 | 互斥锁/逻辑过期 |
| **雪崩** | 大量 Key 同时过期 | 随机 TTL/多级缓存 |

### Q24：如何保证缓存与数据库一致性？

**答：** Cache Aside 模式：写操作先更新 DB 再删除缓存。极端情况用延迟双删或 Canal 订阅 binlog 异步更新缓存。

### Q25：Redis 分布式锁怎么实现？

**答：**
- `SET key value NX EX`（原子加锁+过期时间）
- 释放锁用 Lua 脚本保证原子性（判断 value 相等才删除）
- 推荐使用 **Redisson**（看门狗续期、可重入锁）

### Q26：Redis Cluster 的原理？

**答：** 16384 个哈希槽分配到各节点，`CRC16(key) % 16384` 路由。每个主节点有从节点做故障转移。扩容通过迁移哈希槽实现。

### Q27：Redis 的主从复制原理？

**答：**
- **全量复制**：从节点第一次连接主节点，主节点 BGSAVE 生成 RDB 发送给从节点
- **增量复制**：断线重连后，基于 replication offset 传输增量数据
- 主从延迟不可避免，读从库可能读到旧数据

### Q28：Redis 的 Pipeline 和 Lua 脚本？

**答：**
- **Pipeline**：批量发送命令，减少网络往返次数，不保证原子性
- **Lua 脚本**：在 Redis 服务端原子执行多个命令，常用于分布式锁释放
