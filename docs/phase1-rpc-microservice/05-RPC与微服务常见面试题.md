## 一、RPC基础

### Q1：什么是RPC？和HTTP有什么区别？

**答：**

> RPC（Remote Procedure Call）是远程过程调用，允许程序调用远程服务就像调用本地方法一样。

| 维度 | RPC | HTTP |
|------|-----|------|
| **本质** | 一种调用方式/协议规范 | 应用层传输协议 |
| **关系** | 可基于 HTTP 实现（如 gRPC） | 传输协议 |
| **使用方式** | 像调用本地方法一样（代理封装） | 手动构造请求和解析响应 |
| **序列化** | 二进制（Protobuf/Hessian） | 通常 JSON |
| **性能** | 高（二进制 + 长连接） | 相对较低 |
| **适合场景** | 微服务内部调用 | 对外 API、跨组织调用 |

```
RPC 调用过程（对开发者透明）:
  userService.getUser(1)
        │
        ▼ 代理拦截
  ┌──────────────────────┐
  │ 1. 序列化参数          │
  │ 2. 网络传输            │
  │ 3. 远程执行            │
  │ 4. 反序列化结果         │
  └──────────────────────┘
        │
        ▼
  User{id=1, name="张三"}  ← 开发者感觉像本地调用
```

### Q2：一次RPC调用的完整流程是什么？

**答：**

```
Client                                   Server
  │                                        │
  │ 1. 调用本地代理(Stub/Proxy)             │
  │ 2. 序列化请求(方法名+参数)              │
  │ 3. 网络传输(TCP/HTTP2)                  │
  │ ────────── 请求数据 ─────────────────→  │
  │                                        │ 4. 反序列化请求
  │                                        │ 5. 根据方法名路由
  │                                        │ 6. 调用真实实现
  │                                        │ 7. 序列化响应
  │ ←─────────── 响应数据 ────────────────  │
  │ 8. 反序列化响应                         │
  │ 9. 返回结果给调用方                      │
  │                                        │
```

关键环节：
1. **动态代理**：屏蔽网络调用细节，让远程调用透明
2. **序列化**：将对象转为字节流传输
3. **网络传输**：TCP 长连接 + 多路复用
4. **服务路由**：根据接口名和方法名找到实现类

### Q3：RPC框架需要解决哪些核心问题？

**答：**

| 核心问题 | 说明 | 常见方案 |
|---------|------|---------|
| **服务发现** | 如何找到服务地址 | 注册中心（ZK/Nacos） |
| **网络通信** | 如何高效传输数据 | Netty、HTTP/2 |
| **序列化** | 如何编解码数据 | Protobuf、Hessian、JSON |
| **负载均衡** | 如何分配请求 | 加权随机、一致性哈希 |
| **容错** | 如何处理调用失败 | 重试、熔断、降级 |
| **超时控制** | 如何避免无限等待 | 设置超时 + Future |
| **服务治理** | 如何管理服务 | 限流、灰度、监控 |

## 二、Dubbo相关

### Q4：Dubbo的SPI机制有什么作用？为什么不用Java SPI？

**答：**

Dubbo SPI 是框架可扩展性的基石，协议、负载均衡、序列化等几乎所有核心组件都通过 SPI 加载。

Java SPI 的不足：
1. **全量加载**：`ServiceLoader` 一次性实例化所有实现类，浪费资源
2. **无法按需获取**：不能根据 key 获取指定实现
3. **无 IoC/AOP**：不支持依赖注入和包装器模式

Dubbo SPI 的增强：
1. **按需加载**：按 key 只加载需要的实现（`ExtensionLoader.getExtension("dubbo")`）
2. **自适应扩展**：`@Adaptive` 注解支持运行时根据 URL 参数动态选择实现
3. **依赖注入**：自动注入 SPI 接口依赖
4. **AOP 包装**：支持 Wrapper 包装器，实现切面增强

### Q5：Dubbo的负载均衡策略有哪些？如何自定义？

**答：**

五种内置策略：

| 策略 | 说明 |
|------|------|
| **RandomLoadBalance（默认）** | 加权随机，按权重概率分配 |
| **RoundRobinLoadBalance** | 加权轮询，平滑分配 |
| **LeastActiveLoadBalance** | 最小活跃数，慢节点自动少分配 |
| **ConsistentHashLoadBalance** | 一致性哈希，相同参数路由到同一节点 |
| **ShortestResponseLoadBalance** | 最短响应时间优先 |

自定义负载均衡：

```java
// 1. 实现 LoadBalance 接口
public class MyLoadBalance implements LoadBalance {
    @Override
    public <T> Invoker<T> select(List<Invoker<T>> invokers,
                                  URL url, Invocation invocation) {
        // 自定义选择逻辑
        return invokers.get(0);
    }
}

// 2. SPI 配置：META-INF/dubbo/org.apache.dubbo.rpc.cluster.LoadBalance
// myLB=com.example.MyLoadBalance

// 3. 使用：@DubboReference(loadbalance = "myLB")
```

### Q6：Dubbo的容错策略怎么选？

**答：**

```
场景判断流程:

  操作是否幂等？
    │
    ├── 是（读操作、Delete）
    │     └── Failover（默认，自动重试2次）
    │
    └── 否（写操作、Create）
          │
          ├── 核心操作（下单、支付）
          │     └── Failfast（立即失败，不重试）
          │
          ├── 非核心操作（日志、通知）
          │     └── Failsafe（忽略异常）
          │
          └── 异步操作（消息推送）
                └── Failback（后台定时重发）
```

特殊场景：
- **Forking**：实时性要求极高的读操作，如实时报价，并行调用多个节点
- **Broadcast**：需要通知所有节点的场景，如清除缓存

### Q7：Dubbo的通信协议是怎样的？为什么不用HTTP？

**答：**

Dubbo 使用自定义二进制协议 + TCP 长连接：
- **协议头 16 字节**：Magic(2) + Flag(1) + Status(1) + RequestID(8) + DataLength(4)
- **序列化**：默认 Hessian2 二进制序列化

为什么不用 HTTP/1.1：
1. **协议开销**：HTTP 头部冗余（Content-Type、Cookie 等），Dubbo 协议头仅 16 字节
2. **连接效率**：HTTP/1.1 请求-响应模型，Dubbo TCP 长连接 + 异步多路复用
3. **序列化**：HTTP 通常用 JSON 文本，Dubbo 用二进制序列化，体积更小

注意：Dubbo 3.x 已支持 Triple 协议（基于 HTTP/2 + Protobuf），兼容 gRPC。

## 三、gRPC与Protobuf相关

### Q8：Protobuf的编码为什么比JSON小？

**答：**

1. **字段标识**：Protobuf 用字段编号（1~2字节的Varint），JSON 用完整字段名字符串
2. **整数编码**：Protobuf 用 Varint 变长编码（小数字只需1字节），JSON 用文本（数字300占3字节字符）
3. **无冗余字符**：Protobuf 没有 `{}`、`""`、`,`、`:` 等分隔符
4. **二进制格式**：不需要 Base64 编码二进制数据

```
示例: {id: 1, name: "Tom"}

JSON:  {"id":1,"name":"Tom"}     → 21 字节
       ^ ^ ^ ^ ^ ^ ^              冗余字符

Protobuf: 08 01 12 03 54 6F 6D   → 7 字节
          |    |    |
          字段1 字段2 "Tom"的UTF8
          值=1  长度3
```

### Q9：gRPC的四种通信模式分别用在什么场景？

**答：**

| 模式 | 场景示例 |
|------|---------|
| **Unary** | 查询用户信息、创建订单——最常用，等同于传统 RPC 调用 |
| **Server Streaming** | 股票行情推送、大数据集分批返回、日志实时推送 |
| **Client Streaming** | 文件上传（分块发送）、传感器数据批量上报、大批量数据导入 |
| **Bidirectional Streaming** | 即时聊天、在线游戏、实时协同编辑、双向监控数据 |

选择原则：
- 简单请求/响应 → Unary
- 服务端持续推送 → Server Streaming
- 客户端批量发送 → Client Streaming
- 双方实时交互 → Bidirectional Streaming

### Q10：HTTP/2相比HTTP/1.1有哪些改进？对gRPC有什么意义？

**答：**

| HTTP/2 改进 | 对 gRPC 的意义 |
|-------------|---------------|
| **多路复用** | 单连接并行处理多个 RPC 调用，减少连接开销 |
| **二进制帧** | 天然适合 Protobuf 二进制数据传输 |
| **头部压缩（HPACK）** | 减少元数据传输开销，高频调用受益大 |
| **服务端推送** | 支持 Server Streaming 模式 |
| **流控制** | 每个 Stream 独立流控，避免慢请求阻塞 |

关键点：HTTP/2 的多路复用消除了 HTTP 层队头阻塞，但 TCP 层丢包仍会阻塞所有 Stream。HTTP/3（QUIC）使用 UDP + 独立流，彻底解决此问题。

### Q11：gRPC和Dubbo怎么选？

**答：**

```
选择路径:
  纯 Java 微服务？
    ├── 是 → Dubbo（生态成熟、功能全面、Spring集成好）
    └── 否 → 多语言？
              ├── 是 → gRPC（跨语言最佳、Protobuf高效）
              └── 否 → 云原生(K8s/Istio)？
                        ├── 是 → gRPC（Service Mesh首选协议）
                        └── 否 → 按团队技术栈选择
```

实际生产中，Dubbo 3.x 通过 Triple 协议已兼容 gRPC，可以同时享受 Dubbo 的治理能力和 gRPC 的跨语言优势。

## 四、服务治理相关

### Q12：限流算法有哪些？如何选择？

**答：**

| 算法 | 适用场景 | 代表实现 |
|------|---------|---------|
| **固定窗口计数器** | 简单限流（有窗口边界问题） | 简单实现 |
| **滑动窗口计数器** | 精确限流 | Sentinel |
| **漏桶算法** | 严格匀速流量整形 | Nginx limit_req |
| **令牌桶算法** | API 限流（允许突发） | Guava RateLimiter、Sentinel |

选择建议：
- **API 网关**：令牌桶（允许短时突发，用户体验好）
- **消息消费**：漏桶（匀速消费，保护下游）
- **通用限流**：滑动窗口（精确、无边界问题）

### Q13：熔断和降级有什么区别？

**答：**

| 维度 | 熔断 | 降级 |
|------|------|------|
| **触发条件** | 下游服务错误率达到阈值 | 系统压力大或服务不可用 |
| **作用对象** | 对下游的调用链路 | 当前服务的非核心功能 |
| **行为** | 切断请求，快速失败 | 返回缓存、默认值或简化结果 |
| **恢复方式** | 半开探测，自动恢复 | 手动开关或自动策略 |
| **类比** | 电路保险丝，过载自动断开 | 飞机紧急情况下关闭娱乐系统 |

关系：熔断触发后通常执行降级逻辑。熔断是"手段"，降级是"结果"。

### Q14：如何设计一个分布式限流方案？

**答：**

```
方案一: Redis + Lua 分布式令牌桶

  ┌──────────┐     ┌──────────┐
  │ Gateway1 │────→│          │
  └──────────┘     │  Redis   │ ← Lua 脚本原子操作
  ┌──────────┐────→│ 令牌桶    │
  │ Gateway2 │     │          │
  └──────────┘     └──────────┘
```

```java
// Redis + Lua 令牌桶限流
String luaScript = """
    local key = KEYS[1]
    local capacity = tonumber(ARGV[1])
    local rate = tonumber(ARGV[2])
    local now = tonumber(ARGV[3])
    
    local tokens = tonumber(redis.call('hget', key, 'tokens') or capacity)
    local lastTime = tonumber(redis.call('hget', key, 'lastTime') or now)
    
    local elapsed = now - lastTime
    tokens = math.min(capacity, tokens + elapsed * rate / 1000)
    
    if tokens >= 1 then
        tokens = tokens - 1
        redis.call('hset', key, 'tokens', tokens)
        redis.call('hset', key, 'lastTime', now)
        return 1
    end
    return 0
    """;
```

```
方案二: Sentinel 集群限流
```

### Q15：超时和重试策略如何设计？

**答：**

**超时设计原则：**
1. 上游超时 > 下游超时（如 Gateway 5s > ServiceA 3s > ServiceB 1s）
2. 超时值 = P99 延迟 × 2（根据历史监控数据设置）
3. 连接超时（短，如 500ms）和读超时（长，如 3s）分开设置

**重试设计原则：**
1. 只对幂等操作重试（GET、DELETE、PUT）
2. 使用指数退避 + 随机抖动（1s → 2s+jitter → 4s+jitter）
3. 重试总时间 < 上游超时时间
4. 区分可重试异常（超时、网络错误）和不可重试异常（业务错误）
5. 设置最大重试次数（通常 2~3 次）

## 五、注册中心相关

### Q16：CAP理论是什么？注册中心应该选CP还是AP？

**答：**

**CAP 理论：**
- **C（Consistency）**：所有节点看到相同数据
- **A（Availability）**：每个请求都能得到响应
- **P（Partition Tolerance）**：网络分区时系统仍能运行

分布式系统必须满足 P，因此只能在 C 和 A 之间选择。

**注册中心建议选 AP：**
1. 注册中心短暂不一致（返回旧地址列表）通常可以接受
2. 注册中心不可用（无法获取任何地址）会导致整个系统瘫痪
3. 大多数场景下，可用性比强一致性更重要
4. 因此 Nacos AP 模式或 Eureka 更适合做注册中心

**CP 适合场景：** 配置管理、分布式锁、Leader 选举等需要强一致性的场景。

### Q17：Zookeeper的Watch机制是怎样的？

**答：**

Watch 是 ZK 的核心事件通知机制：
1. **注册 Watch**：客户端在读取数据时注册 Watch（getData、getChildren、exists）
2. **触发通知**：当 ZNode 数据变化时，ZK 向注册了 Watch 的客户端发送通知
3. **一次性**：Watch 触发后自动失效，需要客户端重新注册
4. **顺序性**：Watch 事件按照 ZK 操作顺序通知
5. **轻量级**：通知只包含事件类型和节点路径，不包含具体数据

用于服务发现：Consumer Watch Provider 目录节点，Provider 上下线触发通知。

```
Watch 的一次性问题:
  注册Watch → 数据变更A → Watch触发通知
                                │
          数据变更B ← 重新注册Watch ← 处理通知
                 ↑
       这段时间的变更可能丢失！

  解决: 收到通知后立即重新注册Watch + 全量拉取最新数据
```

### Q18：Nacos和Zookeeper做注册中心有什么区别？

**答：**

| 维度 | Nacos | Zookeeper |
|------|-------|-----------|
| **CAP** | AP（临时实例）+ CP（永久实例） | CP |
| **健康检查** | 客户端心跳（临时）/ 服务端探活（永久） | Session + 临时节点 |
| **数据模型** | 服务-分组-集群-实例 | 树形 ZNode |
| **变更通知** | UDP 推送 + 长轮询 | Watch 机制（一次性） |
| **配置中心** | ✅ 内置强大的配置管理 | 可用但不专业 |
| **雪崩保护** | ✅ 有保护阈值 | ❌ 无 |
| **运维** | ✅ Dashboard、API | 需第三方工具 |
| **学习成本** | 低 | 较高 |

结论：新项目首选 Nacos，功能更全面，运维更方便。

### Q19：Eureka的自我保护机制解决了什么问题？有什么缺点？

**答：**

**解决的问题：** 网络分区导致大量实例心跳丢失时，防止 Eureka 误删健康实例。

**工作原理：** 最近 1 分钟心跳续约数 < 预期数量 × 85% → 触发自我保护 → 不再剔除任何实例。

**缺点：**
1. 保护期间可能返回已下线实例的地址，客户端调用会失败
2. 保护期间新注册的实例也不会被剔除（即使它们真的不健康）
3. 客户端需要自行处理调用失败（如重试其他实例）

**应对方案：** 客户端结合重试、健康检查（如 Ribbon 心跳）过滤不可用实例。

## 六、综合架构设计题

### Q20：设计一个完整的微服务RPC调用链路，包含服务治理

**答：**

```
完整调用链路:

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  Client  │───→│ Gateway  │───→│Service A │───→ Service B
  └──────────┘    └──────────┘    └──────────┘
                       │               │
                  ┌────┴────┐    ┌─────┴─────┐
                  │ 限流     │    │ 熔断+降级  │
                  │ 认证鉴权  │    │ 超时+重试  │
                  │ 灰度路由  │    │ 负载均衡   │
                  └─────────┘    └───────────┘
```

各环节治理措施：
1. **网关层**：统一限流（令牌桶）、认证鉴权、灰度染色
2. **RPC 调用层**：负载均衡（加权随机）、超时（3s）、重试（2次）
3. **服务端**：熔断（错误率50%触发）、降级（返回缓存）、线程池隔离
4. **注册中心**：服务发现（Nacos）、健康检查、动态配置
5. **监控**：链路追踪（SkyWalking）、指标采集（Prometheus）

### Q21：如果让你设计一个RPC框架，你会如何设计？

**答：**

```
核心模块设计:

┌──────────────────────────────────────────────┐
│                   RPC 框架                     │
│                                               │
│  ┌─────────┐  ┌────────────┐  ┌───────────┐  │
│  │  代理层  │  │  协议层     │  │  传输层    │  │
│  │ JDK/CGLIB│  │ 编解码器    │  │  Netty    │  │
│  └────┬────┘  └─────┬──────┘  └─────┬─────┘  │
│       │             │               │         │
│  ┌────┴────┐  ┌─────┴──────┐  ┌─────┴─────┐  │
│  │  路由层  │  │  序列化层   │  │  注册中心  │  │
│  │ 负载均衡 │  │ Protobuf   │  │  Nacos    │  │
│  │  容错   │  │ Hessian    │  │  ZK       │  │
│  └─────────┘  └────────────┘  └───────────┘  │
│                                               │
│  ┌─────────────────────────────────────────┐  │
│  │            SPI 扩展机制                   │  │
│  │  所有核心组件均可通过 SPI 替换和扩展        │  │
│  └─────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

核心设计要点：
1. **代理层**：基于 JDK/CGLIB 动态代理，屏蔽远程调用细节
2. **序列化**：支持多种协议（Protobuf/Hessian/JSON），SPI 可扩展
3. **传输层**：Netty 异步非阻塞 IO，支持长连接和连接池
4. **协议设计**：自定义二进制协议，包含 Magic、RequestID、序列化类型
5. **服务发现**：集成注册中心，支持动态感知
6. **负载均衡**：内置多种策略，支持自定义
7. **容错机制**：Failover/Failfast 等策略
8. **SPI 扩展**：核心组件均通过 SPI 加载，支持用户自定义扩展

### Q22：微服务之间的调用出现超时，如何排查？

**答：**

排查步骤：

```
1. 确认超时类型
   ├── 连接超时 → 网络问题或服务端未启动
   └── 读超时   → 服务端处理慢

2. 查看监控数据
   ├── 链路追踪（SkyWalking）→ 定位慢在哪个环节
   ├── 服务端 RT 监控 → 确认是否是服务端处理慢
   └── 网络监控 → 确认是否有网络抖动

3. 服务端排查
   ├── 线程池是否满 → jstack 查看线程状态
   ├── GC 情况 → GC 日志是否有 Full GC
   ├── 数据库慢查询 → 检查 SQL 执行时间
   └── 依赖服务 → 是否因下游服务慢导致连锁反应

4. 网络排查
   ├── 网络延迟 → ping/traceroute
   ├── 带宽是否打满 → iftop/nethogs
   └── 连接数是否耗尽 → netstat/ss
```

### Q23：如何实现服务的优雅上下线？

**答：**

**优雅下线流程：**

```
  收到停止信号（SIGTERM）
        │
        ▼
  ┌──────────────────┐
  │ 1. 注册中心注销   │ ← 通知 Consumer 不再路由到本节点
  └────────┬─────────┘
           ▼
  ┌──────────────────┐
  │ 2. 等待进行中请求 │ ← 给正在处理的请求一个完成的机会
  │    处理完成       │    （设置最大等待时间，如30s）
  └────────┬─────────┘
           ▼
  ┌──────────────────┐
  │ 3. 关闭服务端口   │ ← 不再接受新请求
  └────────┬─────────┘
           ▼
  ┌──────────────────┐
  │ 4. 释放资源       │ ← 关闭连接池、线程池、缓存等
  └──────────────────┘
```

```java
// Spring Boot 优雅停机配置
// application.yml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s

// Dubbo 优雅停机（自动支持）
dubbo:
  application:
    shutwait: 10000  # 等待10秒
```

**优雅上线流程：**
1. 服务启动完成，通过健康检查
2. 预热（Warmup）：新上线的服务权重逐步增加，避免冷启动被大量请求冲击
3. 注册到注册中心，开始接收流量

### Q24：在大流量场景下，如何保证RPC调用的稳定性？

**答：**

多层防护策略：

| 层级 | 措施 | 具体实践 |
|------|------|---------|
| **接入层** | 限流 | 网关令牌桶限流，Sentinel 集群限流 |
| **调用层** | 超时+重试 | 合理超时（P99×2），指数退避重试 |
| **调用层** | 负载均衡 | 加权随机 + 自适应（LeastActive） |
| **服务层** | 熔断 | 错误率/慢调用率触发，半开探测恢复 |
| **服务层** | 降级 | 非核心功能关闭，返回缓存或默认值 |
| **服务层** | 隔离 | 线程池隔离关键服务，防止故障蔓延 |
| **基础设施** | 注册中心 | 多节点部署，本地缓存兜底 |
| **基础设施** | 监控告警 | 链路追踪+指标监控，快速定位问题 |

核心原则：**快速失败**（不让请求堆积）、**弹性恢复**（自动探测恢复）、**降级保底**（核心功能不受影响）。
