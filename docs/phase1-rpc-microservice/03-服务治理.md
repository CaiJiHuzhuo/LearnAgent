## 一、限流

> 限流（Rate Limiting）是保护系统的第一道防线，通过限制请求速率，防止过载导致系统雪崩。

### 1.1 限流算法总览

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定窗口计数器** | 固定时间窗口内计数 | 实现简单 | 窗口边界突刺问题 | 粗粒度限流 |
| **滑动窗口计数器** | 将窗口划分为多个小格 | 平滑限流 | 实现较复杂 | 精确限流 |
| **漏桶算法** | 请求进桶，匀速流出 | 流量平滑 | 无法应对突发流量 | 流量整形 |
| **令牌桶算法** | 匀速产生令牌，有令牌才放行 | 允许一定突发 | 实现较复杂 | API 网关限流 |

### 1.2 固定窗口计数器

```
时间窗口: 1秒，阈值: 100

  ┌─── 窗口1 ───┐┌─── 窗口2 ───┐
  │  请求: 80    ││  请求: 90   │
  │  ✅ 放行     ││  ✅ 放行     │
  └─────────────┘└─────────────┘
  0s            1s            2s

  问题：窗口边界突刺
  ┌─── 窗口1 ───┐┌─── 窗口2 ───┐
  │        90请求 ││ 90请求      │
  └─────────────┘└─────────────┘
  0s    0.9s    1s    1.1s    2s
        └────180请求/0.2秒────┘  ← 超过阈值！
```

```java
// 固定窗口计数器
public class FixedWindowRateLimiter {
    private final int maxRequests;
    private final long windowMs;
    private long windowStart;
    private int count;

    public FixedWindowRateLimiter(int maxRequests, long windowMs) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
        this.windowStart = System.currentTimeMillis();
        this.count = 0;
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        if (now - windowStart >= windowMs) {
            windowStart = now;
            count = 0;
        }
        if (count < maxRequests) {
            count++;
            return true;
        }
        return false;
    }
}
```

### 1.3 滑动窗口计数器

```
将1秒划分为10个100ms的小窗口:

  ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
  │10│12│ 8│15│ 9│11│ 7│13│10│ 5│ = 100（总窗口内请求数）
  └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
  0  100 200 300 400 500 600 700 800 900 (ms)

  时间前进100ms，窗口滑动:
     ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
     │12│ 8│15│ 9│11│ 7│13│10│ 5│14│ = 104 → 超过阈值，拒绝！
     └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
     100 200 300 400 500 600 700 800 900 1000 (ms)
```

```java
// 滑动窗口计数器
public class SlidingWindowRateLimiter {
    private final int maxRequests;
    private final long windowMs;
    private final int bucketCount;
    private final long bucketMs;
    private final int[] buckets;
    private long lastBucketTime;
    private int currentBucket;

    public SlidingWindowRateLimiter(int maxRequests, long windowMs,
                                    int bucketCount) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
        this.bucketCount = bucketCount;
        this.bucketMs = windowMs / bucketCount;
        this.buckets = new int[bucketCount];
        this.lastBucketTime = System.currentTimeMillis();
        this.currentBucket = 0;
    }

    public synchronized boolean tryAcquire() {
        slideWindow();
        int total = 0;
        for (int count : buckets) {
            total += count;
        }
        if (total < maxRequests) {
            buckets[currentBucket]++;
            return true;
        }
        return false;
    }

    private void slideWindow() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastBucketTime;
        int slideBuckets = (int) (elapsed / bucketMs);
        if (slideBuckets > 0) {
            for (int i = 0; i < Math.min(slideBuckets, bucketCount); i++) {
                currentBucket = (currentBucket + 1) % bucketCount;
                buckets[currentBucket] = 0;
            }
            lastBucketTime = now;
        }
    }
}
```

### 1.4 漏桶算法

```
  请求到达（速率不均）            匀速流出
        │                           │
        ▼                           ▼
  ┌─────────────┐           ┌─────────────┐
  │ ● ● ● ● ●  │           │             │
  │  ● ● ●     │ ─桶满溢出→ │  丢弃/排队  │
  │   ● ●      │           │             │
  └──────┬──────┘           └─────────────┘
         │
         ▼ 匀速漏出（固定速率处理）
    ● ─ ● ─ ● ─ ● ─ ●

  特点: 无论入口流量多大，出口流量恒定
  缺点: 无法处理合法的突发流量
```

```java
// 漏桶算法
public class LeakyBucketRateLimiter {
    private final int capacity;      // 桶容量
    private final double leakRate;   // 漏出速率（请求/毫秒）
    private double water;            // 当前水量
    private long lastLeakTime;       // 上次漏水时间

    public LeakyBucketRateLimiter(int capacity, int leakPerSecond) {
        this.capacity = capacity;
        this.leakRate = leakPerSecond / 1000.0;
        this.water = 0;
        this.lastLeakTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        leak();
        if (water < capacity) {
            water++;
            return true;
        }
        return false;  // 桶满，拒绝
    }

    private void leak() {
        long now = System.currentTimeMillis();
        double leaked = (now - lastLeakTime) * leakRate;
        water = Math.max(0, water - leaked);
        lastLeakTime = now;
    }
}
```

### 1.5 令牌桶算法

```
  令牌以固定速率放入桶中
        │ r 个/秒
        ▼
  ┌─────────────┐
  │ ○ ○ ○ ○ ○  │  ← 桶容量 b（最多存 b 个令牌）
  │  ○ ○ ○     │
  └──────┬──────┘
         │
         ▼ 请求到达，取令牌
  请求1: 取1个令牌 ✅ → 放行
  请求2: 取1个令牌 ✅ → 放行
  请求3: 无令牌   ❌ → 拒绝/等待

  特点: 桶满时可以一次性处理 b 个突发请求（允许突发）
  优于漏桶: 既能限流又能应对短时突发流量
```

```java
// 令牌桶算法（Guava RateLimiter 简化实现）
public class TokenBucketRateLimiter {
    private final int capacity;        // 桶容量
    private final double refillRate;   // 令牌填充速率（个/毫秒）
    private double tokens;             // 当前令牌数
    private long lastRefillTime;       // 上次填充时间

    public TokenBucketRateLimiter(int capacity, int permitsPerSecond) {
        this.capacity = capacity;
        this.refillRate = permitsPerSecond / 1000.0;
        this.tokens = capacity;  // 初始满桶
        this.lastRefillTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        refill();
        if (tokens >= 1) {
            tokens--;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.currentTimeMillis();
        double newTokens = (now - lastRefillTime) * refillRate;
        tokens = Math.min(capacity, tokens + newTokens);
        lastRefillTime = now;
    }
}
```

## 二、熔断降级

> 熔断（Circuit Breaker）是微服务保护的核心机制，当下游服务出现故障时，熔断器自动切断请求，防止故障蔓延导致系统雪崩。

### 2.1 熔断器状态机

```
                 失败率 ≥ 阈值
  ┌────────┐ ──────────────────→ ┌────────┐
  │ Closed │                     │  Open  │
  │ 关闭态  │ ←── 探测成功 ─────── │  打开态 │
  └────────┘                     └───┬────┘
       ↑                             │
       │                             │ 超时等待
       │                             ▼
       │                       ┌──────────┐
       └─── 探测成功 ─────────── │ Half-Open│
             （恢复正常）        │  半开态   │
                                └──────────┘
                                     │
                              探测失败 │
                                     ↓
                                ┌────────┐
                                │  Open  │
                                └────────┘
```

| 状态 | 说明 |
|------|------|
| **Closed（关闭）** | 正常状态，请求正常通过，统计失败率 |
| **Open（打开）** | 熔断状态，拒绝所有请求，直接返回降级响应 |
| **Half-Open（半开）** | 试探状态，放行少量请求测试下游是否恢复 |

### 2.2 熔断器核心实现

```java
public class CircuitBreaker {
    private enum State { CLOSED, OPEN, HALF_OPEN }

    private State state = State.CLOSED;
    private int failureCount = 0;
    private int successCount = 0;
    private final int failureThreshold = 5;     // 失败阈值
    private final int halfOpenMaxRequests = 3;    // 半开放行数
    private final long openTimeoutMs = 10000;     // 熔断超时时间
    private long lastFailureTime = 0;

    public synchronized boolean allowRequest() {
        switch (state) {
            case CLOSED:
                return true;
            case OPEN:
                if (System.currentTimeMillis() - lastFailureTime
                        > openTimeoutMs) {
                    state = State.HALF_OPEN;
                    successCount = 0;
                    return true;
                }
                return false;  // 熔断中，拒绝请求
            case HALF_OPEN:
                return successCount < halfOpenMaxRequests;
            default:
                return false;
        }
    }

    public synchronized void recordSuccess() {
        switch (state) {
            case HALF_OPEN:
                successCount++;
                if (successCount >= halfOpenMaxRequests) {
                    state = State.CLOSED;
                    failureCount = 0;
                }
                break;
            case CLOSED:
                failureCount = 0;
                break;
        }
    }

    public synchronized void recordFailure() {
        failureCount++;
        lastFailureTime = System.currentTimeMillis();
        if (state == State.HALF_OPEN) {
            state = State.OPEN;
        } else if (failureCount >= failureThreshold) {
            state = State.OPEN;
        }
    }
}
```

### 2.4 Sentinel vs Hystrix

| 维度 | Sentinel | Hystrix |
|------|----------|---------|
| **开发者** | 阿里巴巴 | Netflix |

## 三、服务隔离

> 服务隔离通过限制资源使用范围，确保某个服务的故障不会耗尽整个系统的资源。

### 3.1 线程池隔离

```
┌─────────────────────────────────────────────┐
│                  应用进程                      │
│                                              │
│  ┌──────────────┐  ┌──────────────┐          │
│  │ 用户服务线程池 │  │ 订单服务线程池 │          │
│  │ 核心: 10     │  │ 核心: 20     │          │
│  │ 最大: 20     │  │ 最大: 50     │          │
│  └──────┬───────┘  └──────┬───────┘          │
│         │                  │                  │
│         ▼                  ▼                  │
│   用户服务(慢)        订单服务(正常)            │
│   线程池满 ❌         不受影响 ✅               │
│   只影响用户服务       订单服务正常运行           │
└─────────────────────────────────────────────┘
```

```java
// Hystrix 线程池隔离配置
@HystrixCommand(
    groupKey = "UserService",
    commandKey = "getUser",
    threadPoolKey = "UserServicePool",
    threadPoolProperties = {
        @HystrixProperty(name = "coreSize", value = "10"),
        @HystrixProperty(name = "maxQueueSize", value = "100"),
        @HystrixProperty(name = "queueSizeRejectionThreshold", value = "80")
    },
    fallbackMethod = "getUserFallback"
)
public User getUser(Long id) {
    return userServiceClient.getUser(id);
}
```

**优缺点：**

| 维度 | 说明 |
|------|------|
| ✅ 优点 | 线程资源完全隔离，一个服务故障不影响其他服务 |
| ✅ 优点 | 可以设置超时，异步执行 |
| ❌ 缺点 | 线程切换开销大（创建、销毁、上下文切换） |
| ❌ 缺点 | 线程池数量多会占用大量系统资源 |

### 3.2 信号量隔离

```
┌─────────────────────────────────────────────┐
│                  应用进程                      │
│                                              │
│  ┌──────────────┐  ┌──────────────┐          │
│  │ 用户服务信号量 │  │ 订单服务信号量 │          │
│  │ 许可: 10     │  │ 许可: 20     │          │
│  └──────┬───────┘  └──────┬───────┘          │
│         │                  │                  │
│   请求线程直接执行     请求线程直接执行           │
│   （不切换线程）       （不切换线程）             │
│                                              │
│   当并发请求数 > 10    当并发请求数 > 20         │
│   → 拒绝新请求        → 拒绝新请求              │
└─────────────────────────────────────────────┘
```

```java
// Sentinel 信号量隔离（默认方式）
FlowRule rule = new FlowRule();
rule.setResource("getUser");
rule.setGrade(RuleConstant.FLOW_GRADE_THREAD);  // 线程数限流≈信号量隔离
rule.setCount(10);  // 最多10个并发
```

### 3.3 线程池隔离 vs 信号量隔离

| 维度 | 线程池隔离 | 信号量隔离 |
|------|-----------|-----------|
| **执行线程** | 独立线程池 | 调用方线程 |
| **线程开销** | 有线程切换开销 | 无额外开销 |
| **超时控制** | 支持（Future.get timeout） | 不支持主动超时 |
| **异步调用** | 支持 | 不支持 |
| **并发控制** | 线程池大小 | 信号量许可数 |
| **适用场景** | 外部 HTTP 调用、高延迟场景 | 内部方法调用、低延迟场景 |
| **代表框架** | Hystrix（默认） | Sentinel（默认） |

## 四、服务降级策略

> 服务降级是在系统压力过大或下游不可用时，暂时关闭非核心功能，保障核心功能可用的策略。

### 4.1 降级分类

```
┌─────────────────────────────────────────────┐
│                  降级策略                      │
├──────────────────┬──────────────────────────┤
│     静态降级      │       动态降级             │
├──────────────────┼──────────────────────────┤
│ • 返回默认值      │ • 根据错误率自动降级        │
│ • 返回缓存数据    │ • 根据 RT 自动降级         │
│ • 返回空对象      │ • 根据系统负载自动降级       │
│ • 返回Mock数据    │ • 手动开关控制降级          │
└──────────────────┴──────────────────────────┘
```

### 4.2 降级策略示例

```java
@Service
public class ProductService {

    @Autowired
    private ProductClient productClient;

    @Autowired
    private RedisTemplate<String, Product> redisTemplate;

    public Product getProduct(Long id) {
        try {
            // 1. 正常调用远程服务
            Product product = productClient.getProduct(id);
            // 更新缓存
            redisTemplate.opsForValue().set("product:" + id, product);
            return product;
        } catch (Exception e) {
            // 2. 降级策略链
            return fallbackChain(id, e);
        }
    }

    private Product fallbackChain(Long id, Exception e) {
        // 策略1: 读取缓存
        Product cached = redisTemplate.opsForValue().get("product:" + id);
        if (cached != null) {
            return cached;
        }

        // 策略2: 返回默认值
        return Product.builder()
                .id(id)
                .name("商品信息暂不可用")
                .price(BigDecimal.ZERO)
                .status("DEGRADED")
                .build();
    }
}
```

### 4.3 多级降级体系

```
                  请求到达
                    │
                    ▼
             ┌──────────────┐
             │  正常调用服务  │
             └──────┬───────┘
                    │ 失败
                    ▼
             ┌──────────────┐
   降级L1 →  │  读取本地缓存  │
             └──────┬───────┘
                    │ 缓存未命中
                    ▼
             ┌──────────────┐
   降级L2 →  │  读取Redis缓存 │
             └──────┬───────┘
                    │ 缓存未命中
                    ▼
             ┌──────────────┐
   降级L3 →  │  返回静态默认值 │
             └──────────────┘
```

## 五、超时与重试策略

### 5.1 超时设置原则

```
                     请求链路超时设置
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ Gateway │────→│ Service │────→│ Service │
  │ 超时:5s  │     │ A 超时:3s│     │ B 超时:1s│
  └─────────┘     └─────────┘     └─────────┘

  原则: 上游超时 > 下游超时（留出重试和处理时间）
  Gateway(5s) > ServiceA(3s) > ServiceB(1s)
```

| 原则 | 说明 |
|------|------|
| **上游 > 下游** | 上游超时时间必须大于下游，防止上游先超时 |
| **合理范围** | 根据 P99 延迟设置，通常为 P99 × 2 |
| **分级设置** | 连接超时（短）+ 读超时（较长）+ 写超时（较长） |
| **全链路考虑** | 考虑重试次数对总超时的影响 |

### 5.2 重试策略

```java
// Spring Retry 配置
@Retryable(
    value = {RemoteCallException.class},    // 可重试异常
    maxAttempts = 3,                         // 最大重试次数
    backoff = @Backoff(
        delay = 1000,       // 初始延迟 1 秒
        multiplier = 2,     // 延迟倍数
        maxDelay = 10000    // 最大延迟 10 秒
    )
)
public User getUser(Long id) {
    return userServiceClient.getUser(id);
}

@Recover
public User getUserRecover(RemoteCallException e, Long id) {
    return new User(id, "默认用户", "重试耗尽后的降级返回");
}
```

### 5.3 退避策略

```
固定间隔:     1s → 1s → 1s → 1s
指数退避:     1s → 2s → 4s → 8s
指数退避+抖动: 1s → 2.3s → 4.1s → 7.8s（加随机值避免惊群）

推荐: 指数退避 + 抖动（Exponential Backoff with Jitter）
```

```java
// 指数退避 + 抖动
public long calculateDelay(int retryCount, long baseDelay) {
    long exponentialDelay = baseDelay * (long) Math.pow(2, retryCount);
    long jitter = ThreadLocalRandom.current().nextLong(exponentialDelay / 2);
    return Math.min(exponentialDelay + jitter, MAX_DELAY);
}
```

### 5.4 重试注意事项

| 注意点 | 说明 |
|--------|------|
| **幂等性** | 只对幂等操作重试（GET、DELETE），非幂等操作（POST）需谨慎 |
| **重试风暴** | 大量请求同时重试可能压垮下游，需要退避 + 抖动 |
| **超时联动** | 重试次数 × 单次超时 < 上游总超时 |
| **异常区分** | 网络超时可重试，业务异常（如参数错误）不应重试 |

## 六、灰度发布与流量染色

### 6.1 灰度发布策略

> 灰度发布（Canary Release）是将新版本逐步放量给部分用户，验证无误后再全量发布的策略。

```
┌─────────────────────────────────────────────────┐
│                  流量入口（网关）                   │
│                                                  │
│  全部流量 ─→ 路由规则判断                          │
│              │                                    │
│              ├── 90% ──→ ┌──────────┐             │
│              │           │ V1 稳定版  │             │
│              │           │ (3 实例)   │             │
│              │           └──────────┘             │
│              │                                    │
│              └── 10% ──→ ┌──────────┐             │
│                          │ V2 新版本  │             │
│                          │ (1 实例)   │             │
│                          └──────────┘             │
└─────────────────────────────────────────────────┘
```

### 6.2 灰度策略类型

| 策略 | 说明 | 示例 |
|------|------|------|
| **权重灰度** | 按比例分配流量 | 10% 流量到新版本 |
| **用户灰度** | 按用户ID分组 | 尾号 0-1 的用户走新版本 |
| **地域灰度** | 按地域分配 | 北京用户走新版本 |
| **Header灰度** | 按请求头路由 | 携带 x-gray: true 走新版本 |
| **Cookie灰度** | 按 Cookie 路由 | 灰度用户种特定 Cookie |

### 6.3 流量染色

> 流量染色是在请求头中附加标记（如版本号、灰度标签），沿调用链路传递，确保请求在整条链路中都路由到对应版本的服务。

```
用户请求（携带灰度标签 gray=v2）
    │
    ▼
┌──────────┐  gray=v2  ┌──────────┐  gray=v2  ┌──────────┐
│  网关     │─────────→│ 服务A-v2  │─────────→│ 服务B-v2  │
│ 染色入口  │          │          │          │          │
└──────────┘          └──────────┘          └──────────┘

普通请求（无标签）
    │
    ▼
┌──────────┐           ┌──────────┐           ┌──────────┐
│  网关     │─────────→│ 服务A-v1  │─────────→│ 服务B-v1  │
└──────────┘          └──────────┘          └──────────┘
```

```java
// 网关层染色（Spring Cloud Gateway Filter）
@Component
public class GrayFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                             GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String userId = request.getHeaders().getFirst("X-User-Id");

        // 根据规则判断是否走灰度
        if (isGrayUser(userId)) {
            ServerHttpRequest grayRequest = request.mutate()
                    .header("X-Gray-Tag", "v2")
                    .build();
            return chain.filter(exchange.mutate().request(grayRequest).build());
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() { return 0; }
}

// Dubbo 灰度路由（自定义 Router）
public class GrayRouter implements Router {
    @Override
    public <T> List<Invoker<T>> route(List<Invoker<T>> invokers,
                                       URL url, Invocation invocation) {
        String grayTag = RpcContext.getContext()
                .getAttachment("X-Gray-Tag");
        if ("v2".equals(grayTag)) {
            return invokers.stream()
                    .filter(i -> "v2".equals(i.getUrl()
                            .getParameter("version")))
                    .collect(Collectors.toList());
        }
        return invokers.stream()
                .filter(i -> !"v2".equals(i.getUrl()
                        .getParameter("version")))
                .collect(Collectors.toList());
    }
}
```

## 七、大厂常见面试题

### Q1：令牌桶和漏桶有什么区别？

**答：**

| 维度 | 令牌桶 | 漏桶 |
|------|--------|------|
| **核心思想** | 以固定速率生成令牌，请求消费令牌 | 请求进桶，以固定速率漏出处理 |
| **突发流量** | 允许（桶中有令牌可以一次性消费） | 不允许（出口速率恒定） |
| **限流效果** | 平均速率限流 + 允许一定突发 | 严格匀速限流 |
| **典型实现** | Guava RateLimiter、Sentinel | Nginx 限流 |
| **适合场景** | API 网关（允许短时突发） | 流量整形（严格匀速） |

### Q2：熔断器的三种状态是什么？状态如何流转？

**答：**

1. **Closed（关闭）**：正常状态，统计失败率，当失败率达到阈值时转为 Open
2. **Open（打开）**：熔断状态，所有请求直接走降级逻辑，经过超时时间后转为 Half-Open
3. **Half-Open（半开）**：放行少量请求探测下游健康状态，若成功则转为 Closed 恢复正常，若失败则回到 Open

关键参数：失败率阈值（如 50%）、熔断超时时间（如 10 秒）、半开探测请求数（如 3 个）。

### Q3：线程池隔离和信号量隔离的区别？

**答：**

- **线程池隔离**：为每个依赖分配独立线程池，调用在独立线程中执行。优点是完全隔离、支持超时和异步，缺点是线程切换开销大。适合外部 HTTP/RPC 调用
- **信号量隔离**：通过信号量控制并发数，调用在请求线程中执行。优点是无线程切换开销、轻量，缺点是不支持主动超时。适合内部方法调用或低延迟场景

### Q4：微服务中如何防止雪崩效应？

**答：**

雪崩防护需要多层措施配合：
1. **限流**：在入口层控制请求速率，令牌桶或滑动窗口
2. **熔断**：快速失败，防止大量请求堆积在故障服务
3. **隔离**：线程池或信号量隔离，单个服务故障不耗尽全局资源
4. **降级**：返回缓存、默认值或简化结果，保障核心功能
5. **超时**：合理设置超时，避免请求无限等待
6. **重试**：指数退避 + 抖动，避免重试风暴

### Q5：灰度发布中如何保证全链路灰度？

**答：**

1. **流量染色**：在网关层给灰度请求打标签（如 Header: X-Gray-Tag=v2）
2. **标签传递**：通过 RPC Context 或消息头沿调用链路自动传递标签
3. **路由匹配**：每个服务根据标签路由到对应版本实例
4. **消息染色**：MQ 消息也需要携带灰度标签，确保消费到对应版本的消费者
5. **数据隔离**：灰度环境和稳定环境使用相同数据库，但需注意数据兼容性

### Q6：Sentinel和Hystrix有什么区别？生产中如何选择？

**答：**

- **Hystrix** 已停止维护（2018年），基于线程池隔离，资源消耗较大，无内置限流
- **Sentinel** 活跃维护，基于信号量隔离（轻量），支持丰富的限流规则、熔断策略和系统保护，提供可视化 Dashboard
- **生产选择**：新项目统一使用 Sentinel，特别是 Spring Cloud Alibaba 技术栈。老项目如果用 Hystrix 可逐步迁移到 Sentinel 或 Resilience4j
