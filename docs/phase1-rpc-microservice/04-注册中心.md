## 一、注册中心的作用与核心功能

> 注册中心是微服务架构的核心基础设施，负责服务的注册、发现和健康管理，是服务间通信的"通讯录"。

### 1.1 为什么需要注册中心

```
没有注册中心（硬编码地址）:
┌──────────┐  http://192.168.1.1:8080  ┌──────────┐
│ Consumer │──────────────────────────→│ Provider │
└──────────┘                           └──────────┘
  问题: Provider 地址变化 → Consumer 代码要改 → 重新部署

有注册中心:
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Consumer │←───→│ Registry │←───→│ Provider │
└──────────┘     └──────────┘     └──────────┘
  优势: Provider 地址变化 → 注册中心自动推送 → Consumer 无感知
```

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| **服务注册** | Provider 启动时将自己的地址、接口、元数据注册到注册中心 |
| **服务发现** | Consumer 从注册中心获取 Provider 地址列表 |
| **健康检查** | 定期检测 Provider 是否存活，剔除不健康节点 |
| **变更通知** | Provider 上下线时实时通知 Consumer 更新地址列表 |
| **配置管理** | 部分注册中心（如 Nacos）还提供配置中心功能 |

### 1.3 注册发现流程

```
  ① Provider启动，注册服务
  ┌──────────┐  注册  ┌──────────────┐
  │ Provider │──────→│              │
  │ A:8080   │       │              │
  └──────────┘       │   Registry   │
  ┌──────────┐  注册  │              │
  │ Provider │──────→│  服务列表:    │
  │ A:8081   │       │  ServiceA:   │
  └──────────┘       │   [A:8080,   │
                     │    A:8081]   │
                     │              │
  ② Consumer启动，   │              │
     订阅服务        │              │
  ┌──────────┐  订阅  │              │
  │ Consumer │──────→│              │
  │          │←──────│  推送地址列表 │
  └──────────┘  通知  └──────────────┘

  ③ Provider下线
  ┌──────────┐       ┌──────────────┐
  │ Provider │ 下线   │              │
  │ A:8080 ✗ │──────→│  更新列表:    │
  └──────────┘       │  ServiceA:   │
                     │   [A:8081]   │
                     │       │      │
                     │       ▼      │
                     │  通知Consumer │
                     └──────────────┘
```

## 二、Zookeeper

> Zookeeper 是 Apache 开源的分布式协调服务，常用作注册中心、分布式锁、配置管理等。它保证 CP（一致性 + 分区容错性）。

### 2.1 数据模型（ZNode）

```
/（根节点）
├── /dubbo
│   ├── /com.example.UserService
│   │   ├── /providers          ← Provider 注册的临时节点
│   │   │   ├── dubbo://192.168.1.1:20880/...
│   │   │   └── dubbo://192.168.1.2:20880/...
│   │   ├── /consumers          ← Consumer 注册的临时节点
│   │   │   └── consumer://192.168.2.1/...
│   │   ├── /configurators      ← 动态配置
│   │   └── /routers            ← 路由规则
│   └── /com.example.OrderService
│       └── ...
└── /config
    └── ...
```

| ZNode 类型 | 说明 |
|-----------|------|
| **持久节点（Persistent）** | 创建后一直存在，除非显式删除 |
| **临时节点（Ephemeral）** | 与创建者的 Session 绑定，Session 断开自动删除 |
| **持久顺序节点** | 持久节点 + 自动递增编号 |
| **临时顺序节点** | 临时节点 + 自动递增编号（常用于分布式锁） |

### 2.2 ZAB协议原理

> ZAB（Zookeeper Atomic Broadcast）是 Zookeeper 的核心一致性协议，保证集群数据的一致性。

```
ZAB 协议两个核心阶段:

阶段一: Leader 选举（崩溃恢复）
  当 Leader 宕机时，集群进入选举流程
  ┌────────┐  ┌────────┐  ┌────────┐
  │ Node 1 │  │ Node 2 │  │ Node 3 │
  │ Vote=1 │  │ Vote=2 │  │ Vote=3 │
  └───┬────┘  └───┬────┘  └───┬────┘
      │           │           │
      └─── 投票 ──┴─── 投票 ──┘
              │
              ▼
        Node 3 当选 Leader
        （ZXID 最大 → myid 最大）

阶段二: 消息广播（正常工作）
  ┌────────┐         ┌────────┐
  │ Client │── 写 ──→│ Leader │
  └────────┘         └───┬────┘
                         │ Proposal（提案）
                    ┌────┴────┐
                    ↓         ↓
              ┌────────┐ ┌────────┐
              │Follower│ │Follower│
              │  ACK   │ │  ACK   │
              └───┬────┘ └───┬────┘
                  │          │
                  └──── ACK ─┘
                       │
                       ▼
                 过半ACK → Leader Commit
                       → 通知所有 Follower Commit
```

### 2.3 Leader选举（FastLeaderElection）

> Zookeeper 使用 FastLeaderElection 算法进行 Leader 选举，核心依据是 ZXID（事务ID）和 myid（节点编号）。

```
选举规则（优先级从高到低）:
  1. epoch（选举轮次）大的优先
  2. ZXID（事务ID）大的优先 → 数据最新的节点
  3. myid（服务器编号）大的优先 → 打破平局

选举过程示例（3节点集群）:
  Node1(myid=1, ZXID=100): 投票 (1, 100) → 发送给所有节点
  Node2(myid=2, ZXID=100): 投票 (2, 100) → 发送给所有节点
  Node3(myid=3, ZXID=99):  投票 (3, 99)  → 发送给所有节点

  第一轮:
    Node1 收到 (2,100), (3,99) → ZXID 相同比 myid → 改投 (2,100)
    Node2 收到 (1,100), (3,99) → 自己 myid 最大 → 保持 (2,100)
    Node3 收到 (1,100), (2,100) → ZXID 更大 → 改投 (2,100)

  结果: Node2 获得 3 票(过半) → 当选 Leader
```

### 2.4 Watch机制

```
Consumer 订阅服务变更（Watch）:

  ┌──────────┐  Watch(/dubbo/.../providers)  ┌──────────┐
  │ Consumer │──────────────────────────────→│Zookeeper │
  └──────────┘                               └──────────┘
                                                  │
          Provider 上线/下线，触发 Watch 通知         │
  ┌──────────┐                               ┌────┴─────┐
  │ Consumer │←── WatchEvent(NodeChanged) ───│Zookeeper │
  └──────────┘                               └──────────┘
       │
       ▼
  更新本地服务地址列表缓存
  重新注册 Watch（Watch 是一次性的）
```

**Watch 特点：**
- **一次性**：Watch 触发后自动失效，需要重新注册
- **顺序性**：Watch 事件按注册顺序通知
- **轻量级**：只通知变更事件，不传具体数据，需要客户端主动获取

### 2.5 Zookeeper的CP模型

```
CAP 理论:
  C（Consistency）: 所有节点数据一致
  A（Availability）: 每个请求都能收到响应
  P（Partition Tolerance）: 网络分区时仍能工作

Zookeeper 选择 CP:
  ┌────────┐      ┌────────┐      ┌────────┐
  │ Leader │      │Follower│      │Follower│
  │  写入   │──→  │  同步   │      │  同步   │
  └────────┘      └────────┘      └────────┘

  写入必须过半节点确认 → 保证一致性（C）
  Leader 宕机时集群不可用（选举期间） → 牺牲可用性（A）
  网络分区时少数派无法提供写服务 → 保证分区容错（P）
```

## 三、Nacos

> Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的服务发现和配置管理平台，同时支持 AP 和 CP 模式。

### 3.1 Nacos架构

```
┌─────────────────────────────────────────────────────┐
│                    Nacos Server                       │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────┐  │
│  │  Naming 模块  │  │  Config 模块  │  │  Console  │  │
│  │  服务发现     │  │  配置管理     │  │  控制台    │  │
│  └──────┬───────┘  └──────┬───────┘  └───────────┘  │
│         │                  │                          │
│  ┌──────┴──────────────────┴──────┐                  │
│  │         一致性协议层             │                  │
│  │  ┌──────────┐  ┌────────────┐  │                  │
│  │  │   Raft   │  │   Distro   │  │                  │
│  │  │ (CP模式) │  │  (AP模式)   │  │                  │
│  │  └──────────┘  └────────────┘  │                  │
│  └────────────────────────────────┘                  │
│                                                      │
│  ┌────────────────────────────────┐                  │
│  │         存储层                  │                  │
│  │  内嵌Derby / MySQL / 内存       │                  │
│  └────────────────────────────────┘                  │
└─────────────────────────────────────────────────────┘
```

### 3.2 AP/CP切换原理

| 模式 | 协议 | 适用场景 | 实例类型 |
|------|------|---------|---------|
| **AP 模式** | Distro 协议 | 服务发现（临时实例） | 临时实例（默认） |
| **CP 模式** | Raft 协议 | 配置管理、永久实例 | 永久实例 |

```
临时实例（AP模式 - Distro协议）:
  ┌──────────┐  心跳(5s)  ┌──────────┐
  │ Provider │──────────→│  Nacos   │
  └──────────┘           │  Server  │
                         │          │
  超过15秒无心跳 → 标记不健康
  超过30秒无心跳 → 剔除实例

永久实例（CP模式 - Raft协议）:
  ┌──────────┐  注册  ┌──────────┐
  │ Provider │──────→│  Nacos   │
  └──────────┘       │  Server  │
                     │          │
  服务端主动健康检查（TCP/HTTP探活）
  不健康只标记，不剔除
```

### 3.3 Distro协议（AP模式）

> Distro 是 Nacos 自研的 AP 模式一致性协议，适用于临时实例的服务注册与发现。

```
Distro 协议数据同步:

  Client 注册请求
        │
        ▼
  ┌──────────┐  异步同步  ┌──────────┐  异步同步  ┌──────────┐
  │  Node 1  │─────────→│  Node 2  │─────────→│  Node 3  │
  │ (负责节点)│           │          │           │          │
  └──────────┘           └──────────┘           └──────────┘

  数据分片: 每个节点只负责一部分数据（按服务名 hash 分配）
  写请求路由: 非负责节点收到写请求 → 转发给负责节点
  数据同步: 负责节点异步推送给其他节点（最终一致性）
  新节点加入: 全量拉取其他节点数据
```

### 3.4 Raft协议（CP模式）

```
Raft 协议核心流程:

  ┌────────────────────────────────────────────┐
  │              Leader 选举                     │
  │                                             │
  │  所有节点初始为 Follower                      │
  │  超时未收到心跳 → 变为 Candidate → 发起投票    │
  │  获得过半投票 → 成为 Leader                   │
  └────────────────────────────────────────────┘
                    │
                    ▼
  ┌────────────────────────────────────────────┐
  │              日志复制                        │
  │                                             │
  │  Client ──写请求──→ Leader                   │
  │  Leader ──AppendEntries──→ Followers        │
  │  过半 Follower 确认 → Leader 提交 → 应用状态机 │
  └────────────────────────────────────────────┘

  Leader 任期（Term）:
    Term 1: Node-A 是 Leader
    Term 2: Node-A 宕机，Node-B 当选新 Leader
    每个 Term 最多一个 Leader
```

### 3.5 配置中心功能

```java

}
```

## 四、Eureka

> Eureka 是 Netflix 开源的服务发现组件，遵循 AP 模型，强调高可用性。

### 4.1 Eureka架构

```
┌───────────────────────────────────────────────────┐
│              Eureka Server 集群                     │
│                                                    │
│  ┌──────────┐  互相复制  ┌──────────┐              │
│  │ Eureka   │←────────→│ Eureka   │              │
│  │ Server 1 │          │ Server 2 │              │
│  └────┬─────┘          └────┬─────┘              │
│       │                      │                    │
└───────┼──────────────────────┼────────────────────┘
        │                      │
   注册+心跳                注册+心跳
        │                      │
  ┌─────┴────┐           ┌─────┴────┐
  │ Provider │           │ Consumer │
  │ (注册)    │           │ (发现)    │
  └──────────┘           └──────────┘
```

### 4.2 AP模型与自我保护机制

```
正常状态:
  Provider 每 30 秒发送一次心跳
  Eureka 90 秒未收到心跳 → 剔除实例

自我保护机制:
  当短时间内大量实例心跳丢失（如网络分区）:
  ┌──────────────────────────────────────────────┐
  │ Eureka 检测: 最近1分钟心跳续约数              │
  │             < 预期数量 × 85%                  │
  │                  │                            │
  │                  ▼                            │
  │ 触发自我保护模式:                              │
  │  ✅ 不再剔除任何实例（即使心跳过期）             │
  │  ✅ 仍然接受新服务注册                         │
  │  ✅ 仍然响应服务发现请求                        │
  │  ❌ 可能返回已经下线的实例（过时数据）           │
  │                                              │
  │ 目的: 宁可保留过时数据，也不误删健康实例         │
  │       网络恢复后，心跳数正常，自动退出保护模式    │
  └──────────────────────────────────────────────┘
```

### 4.3 Eureka vs Zookeeper 注册中心

| 维度 | Eureka | Zookeeper |
|------|--------|-----------|
| **CAP 模型** | AP | CP |
| **一致性** | 最终一致性 | 强一致性 |
| **可用性** | 高（自我保护） | Leader 选举期间不可用 |
| **健康检查** | 客户端心跳（30s） | Session + 临时节点 |
| **节点下线** | 90s 无心跳剔除（有自我保护） | Session 过期立即删除 |
| **集群结构** | 对等复制（无 Leader） | Leader-Follower |

## 六、大厂常见面试题

### Q1：注册中心的核心功能是什么？为什么需要注册中心？

**答：**

核心功能：服务注册、服务发现、健康检查、变更通知。
必要性：
1. **动态扩缩容**：服务实例增减时，无需修改调用方配置
2. **自动剔除**：不健康节点自动下线，避免调用失败
3. **负载均衡**：结合地址列表实现客户端负载均衡
4. **解耦**：调用方无需知道提供方的具体地址

### Q2：Zookeeper的ZAB协议是什么？选举过程是怎样的？

**答：**

ZAB（Zookeeper Atomic Broadcast）是 ZK 的一致性协议，分为两个阶段：
1. **Leader 选举（崩溃恢复）**：集群启动或 Leader 宕机时，通过 FastLeaderElection 选举新 Leader。优先比较 ZXID（数据最新），其次比较 myid（节点编号），获得过半投票者当选
2. **消息广播（正常工作）**：所有写操作由 Leader 处理，Leader 将提案广播给 Follower，过半 Follower 确认后提交

### Q3：Zookeeper为什么不适合做注册中心？

**答：**

这是一个有争议的观点。ZK 的 CP 模型意味着：
1. **Leader 选举期间不可用**：选举可能持续 30~120 秒，期间无法处理写请求
2. **网络分区时少数派不可写**：可能导致部分节点注册失败
3. **服务发现更需要 AP**：对注册中心来说，返回稍旧的地址列表好过完全不可用

但 ZK 在 Dubbo 生态中仍然广泛使用，因为：
- 临时节点天然支持服务注册/注销
- Watch 机制适合服务变更通知
- 实际生产中 ZK 集群稳定性很高

### Q4：Nacos的AP和CP模式分别用在什么场景？

**答：**

- **AP 模式（Distro 协议）**：用于临时实例的服务发现。特点是高可用、最终一致性、客户端心跳检测。适合大多数微服务场景
- **CP 模式（Raft 协议）**：用于永久实例和配置管理。特点是强一致性、服务端主动健康检查。适合需要强一致性的场景

选择原则：服务注册一般用 AP（临时实例），配置管理自动用 CP。

### Q5：Eureka的自我保护机制是什么？

**答：**

当 Eureka Server 在短时间内检测到大量实例心跳丢失（低于预期的 85%），进入自我保护模式：
1. **不再剔除**任何实例（包括心跳过期的实例）
2. **仍然接受**新注册和服务发现请求
3. **返回数据可能过时**（包含已下线的实例）

目的：网络分区导致大量心跳丢失时，宁可返回过时数据也不误删健康实例。当心跳恢复正常后自动退出保护模式。

### Q6：Zookeeper、Nacos、Eureka、Consul如何选型？

**答：**

| 场景 | 推荐 | 理由 |
|------|------|------|
| Dubbo 微服务 | Nacos 或 Zookeeper | Nacos 更现代，ZK 更经典 |
| Spring Cloud Alibaba | Nacos | 原生集成，注册+配置一体 |
| 多语言微服务 | Consul | Go 开发，多语言 SDK |
| 需要配置中心 | Nacos | 最强配置管理能力 |
| 强一致性要求 | Zookeeper 或 Consul | CP 模型 |

核心原则：新项目优先选 Nacos（功能全面、社区活跃），多语言选 Consul。
