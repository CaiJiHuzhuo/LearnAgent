## 一、基础概念题

### Q1：为什么要使用消息队列？项目中哪些场景用到了？

**答：**

消息队列的三大核心价值：

| 核心价值 | 说明 | 典型场景 |
|---------|------|---------|
| **异步处理** | 将耗时的非核心操作异步化，降低主链路响应时间 | 用户注册后发短信/邮件 |
| **应用解耦** | 上下游通过消息通信，降低系统耦合度 | 订单系统通知库存/物流/积分 |
| **流量削峰** | 高并发请求入队缓冲，保护下游服务 | 秒杀请求缓冲 |

```
项目实战示例：

  用户下单 → 写入订单(50ms) → 发MQ消息(5ms) → 返回"下单成功"
                                    │
                                    ├──→ 库存服务消费 → 扣减库存
                                    ├──→ 物流服务消费 → 创建物流单
                                    └──→ 积分服务消费 → 增加积分
  
  好处：主链路只需 55ms，新增下游服务无需修改订单代码
```

**使用 MQ 的缺点：**
1. **系统复杂度增加**：需处理消息丢失、重复消费、消息积压
2. **一致性问题**：异步消费导致数据可能存在延迟
3. **运维成本增加**：MQ 集群自身也需要保证高可用

### Q2：如何进行消息队列的技术选型？

**答：**

```
技术选型决策树：

  需要事务消息 + 延迟消息？ ──→ RocketMQ（电商/金融首选）
  需要超高吞吐 + 流处理？ ──→ Kafka（大数据/日志首选）
  需要复杂路由 + 运维简单？ ──→ RabbitMQ（中小项目首选）
  需要云原生 + 多租户？ ──→ Pulsar（新一代选择）
```

| 维度 | RabbitMQ | RocketMQ | Kafka |
|------|----------|----------|-------|
| **语言** | Erlang | Java | Scala/Java |
| **单机吞吐** | 万级 | 十万级 | 百万级 |
| **消息延迟** | 微秒级 | 毫秒级 | 毫秒级 |
| **功能** | 丰富（路由/优先级） | 丰富（事务/延迟/顺序） | 精简（专注流处理） |
| **社区** | 国际活跃 | 国内活跃（Apache顶级） | 非常活跃（大数据生态） |
| **适用场景** | 中小项目 | 电商/金融 | 大数据/日志 |

**面试回答技巧**：先说业务需求，再从吞吐、延迟、功能、团队技术栈等维度对比，最后给出结论。切忌死记硬背参数。

### Q3：消息队列的 Push 模式和 Pull 模式有什么区别？

**答：**

| 维度 | Push（推模式） | Pull（拉模式） |
|------|--------------|---------------|
| **消息获取** | Broker 主动推送给 Consumer | Consumer 主动从 Broker 拉取 |
| **实时性** | 高，消息到达立即推送 | 取决于拉取间隔 |
| **消费速率** | Broker 控制，可能压垮消费者 | Consumer 自控，按能力拉取 |
| **代表 MQ** | RabbitMQ | Kafka |

**RocketMQ 的长轮询机制**（结合两者优点）：
- Consumer 发起拉取请求
- Broker 如果没有新消息，**挂起请求**等待（默认 15 秒）
- 新消息到达时**立即返回**，兼具 Push 的实时性和 Pull 的消费可控

```
长轮询流程：
  Consumer ──拉取请求──→ Broker（无新消息，挂起等待）
                              │
                         新消息到达
                              │
                              ↓
  Consumer ←──立即返回消息──── Broker
```

### Q4：点对点模型和发布/订阅模型的区别？RocketMQ/Kafka 用的哪种？

**答：**

```
点对点（Queue）：
  Producer ──→ Queue ──→ Consumer A  ← 消息只被其中一个消费
                    └──→ Consumer B

发布/订阅（Topic）：
  Producer ──→ Topic ──→ Subscriber A  ← 每个订阅者都收到消息
                    ├──→ Subscriber B
                    └──→ Subscriber C
```

**RocketMQ 和 Kafka 都是基于发布/订阅模型**，通过 **Consumer Group** 实现两种语义的统一：
- **同一 Consumer Group 内**：竞争消费（Queue 语义）
- **不同 Consumer Group 间**：独立消费（Topic 语义）

```
  Producer ──→ Topic ──→ Group-A: [Consumer-1, Consumer-2] ← 竞争消费
                    └──→ Group-B: [Consumer-3]             ← 独立消费
  
  msg1 在 Group-A 中只被 Consumer-1 或 Consumer-2 消费（竞争）
  msg1 在 Group-B 中被 Consumer-3 消费（独立）
```

### Q5：消息队列如何保证高可用？

**答：**

高可用方案取决于具体的 MQ：

**RocketMQ 高可用：**
```
NameServer 集群（无状态，互相独立）：
  ┌─────────────┐  ┌─────────────┐
  │ NameServer-1 │  │ NameServer-2 │   ← 任一宕机不影响
  └─────────────┘  └─────────────┘

Broker 主从架构：
  ┌────────────┐ 同步/异步复制 ┌────────────┐
  │ Master     │ ──────────→ │ Slave      │
  │ Broker-A   │              │ Broker-A-S │
  └────────────┘              └────────────┘
  Master 宕机 → Slave 可继续提供消费服务（DLedger 模式可自动选主）
```

**Kafka 高可用：**
```
分区副本机制：
  Topic-A, Partition-0:
    Broker-1: Leader    ← 读写都走 Leader
    Broker-2: Follower  ← 同步数据
    Broker-3: Follower  ← 同步数据

  Leader 宕机 → 从 ISR 列表中选举新 Leader
```

## 二、RocketMQ 专题

### Q6：RocketMQ 的事务消息是怎么实现的？

**答：**

RocketMQ 事务消息采用**半消息 + 事务回查**机制，保证本地事务与消息发送的最终一致性。

```
事务消息流程（两阶段提交）：

  Producer                    Broker                      Consumer
     │                          │                            │
     │ ① 发送半消息(Half Msg)    │                            │
     │ ────────────────────→   │ 存储半消息（不可消费）         │
     │                          │                            │
     │ ② 执行本地事务            │                            │
     │ （扣款、减库存等）         │                            │
     │                          │                            │
     │ ③ 提交/回滚              │                            │
     │ ────────────────────→   │                            │
     │   Commit → 消息可消费     │ ─────────────────────────→│
     │   Rollback → 删除消息     │                            │
     │                          │                            │
     │     ④ 事务回查            │                            │
     │ ←────────────────────   │ (如果③超时未收到结果)         │
     │ 返回 COMMIT/ROLLBACK     │                            │
```

```java
// RocketMQ 事务消息代码
@Component
public class TransactionProducer {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void sendTransactionMsg(OrderDTO order) {
        Message<OrderDTO> message = MessageBuilder.withPayload(order).build();
        // 发送事务消息
        rocketMQTemplate.sendMessageInTransaction(
                "order-tx-topic", message, order);
    }
}

@RocketMQTransactionListener
public class OrderTransactionListener
        implements RocketMQLocalTransactionListener {

    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
        try {
            OrderDTO order = (OrderDTO) arg;
            // 执行本地事务（扣款）
            accountService.deductBalance(order.getUserId(), order.getAmount());
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 事务回查：检查本地事务是否执行成功
        String orderId = msg.getHeaders().get("orderId", String.class);
        boolean exists = orderService.existsOrder(orderId);
        return exists ?
                RocketMQLocalTransactionState.COMMIT :
                RocketMQLocalTransactionState.ROLLBACK;
    }
}
```

**面试关键点**：
1. 半消息对消费者**不可见**，存储在 `RMQ_SYS_TRANS_HALF_TOPIC`
2. 回查机制保证了即使 Producer 宕机，事务状态也能最终确定
3. 默认最多回查 15 次，超过则默认回滚

### Q7：RocketMQ 的延迟消息是怎么实现的？

**答：**

RocketMQ 延迟消息的实现原理：

```
延迟消息存储流程：

  ① Producer 发送延迟消息（delayLevel=3, 即10秒）
     │
     ↓
  ② Broker 收到后，不直接写入目标 Topic
     而是写入系统 Topic: SCHEDULE_TOPIC_XXXX
     │
     ↓
  ③ 定时任务 ScheduleMessageService 每秒扫描
     检查是否到达投递时间
     │
     ↓ (到达时间)
  ④ 将消息从 SCHEDULE_TOPIC 转移到目标 Topic
     │
     ↓
  ⑤ Consumer 正常消费
```

**RocketMQ 支持 18 个延迟级别**：

| 级别 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9 |
|------|---|---|---|---|---|---|---|---|---|
| 延迟 | 1s | 5s | 10s | 30s | 1m | 2m | 3m | 4m | 5m |

| 级别 | 10 | 11 | 12 | 13 | 14 | 15 | 16 | 17 | 18 |
|------|----|----|----|----|----|----|----|----|-----|
| 延迟 | 6m | 7m | 8m | 9m | 10m | 20m | 30m | 1h | 2h |

> **RocketMQ 5.x** 开始支持任意时间的精确延迟消息。

### Q8：RocketMQ 如何实现顺序消息？

**答：**

RocketMQ 顺序消息分为**全局有序**和**分区有序**：

```
分区有序（推荐）：
  
  Producer 端：同一业务Key（如订单ID）的消息路由到同一 Queue
  Broker 端：同一 Queue 内消息严格 FIFO
  Consumer 端：ConsumeMode.ORDERLY，同一 Queue 由同一线程顺序消费

  订单A: 创建→支付→发货 ──Hash(A)──→ Queue-0 ──→ Consumer-1 (单线程顺序消费)
  订单B: 创建→支付     ──Hash(B)──→ Queue-1 ──→ Consumer-2 (单线程顺序消费)
```

**实现要点**：
1. **生产端**：使用 `MessageQueueSelector` 按业务 Key Hash 选择队列
2. **消费端**：使用 `ConsumeMode.ORDERLY` 模式，内部对 Queue 加锁，保证单线程顺序消费
3. **注意**：顺序消费模式下，某条消息消费失败会**阻塞**该队列后续消息

### Q9：RocketMQ 的 NameServer 有什么作用？为什么不用 ZooKeeper？

**答：**

**NameServer 的作用**（类似注册中心）：
1. **Broker 注册**：Broker 启动时向所有 NameServer 注册，并定期发送心跳
2. **路由发现**：Producer/Consumer 从 NameServer 获取 Broker 路由信息
3. **故障剔除**：NameServer 检测 Broker 心跳超时（默认 120 秒），剔除故障节点

```
NameServer 架构：

  ┌─────────────┐  ┌─────────────┐
  │ NameServer-1 │  │ NameServer-2 │  ← 无状态，互相独立
  └──────┬──────┘  └──────┬──────┘
         │                │
    心跳注册           心跳注册
         │                │
  ┌──────┴──────────────┴──────┐
  │ Broker-A    Broker-B       │
  └────────────────────────────┘
         ↑
    路由查询
         │
  ┌──────┴──────┐
  │ Producer /   │
  │ Consumer     │
  └─────────────┘
```

**为什么不用 ZooKeeper？**

| 维度 | NameServer | ZooKeeper |
|------|-----------|-----------|
| **架构** | 无状态，各节点独立 | 有状态，需选主（ZAB协议） |
| **一致性** | 最终一致性（AP） | 强一致性（CP） |
| **复杂度** | 极简实现 | 功能复杂（Watch、临时节点） |
| **性能** | 轻量高效 | 较重 |
| **运维** | 简单 | 较复杂（选举、脑裂） |

NameServer 选择 AP 而非 CP，原因是：MQ 场景下短暂的路由不一致是可接受的（最多导致少量消息发送到旧 Broker），而高可用和简单性更重要。

### Q10：RocketMQ 的消息存储结构是怎样的？

**答：**

RocketMQ 采用 **CommitLog + ConsumeQueue** 的存储结构：

```
存储结构：

  CommitLog（所有消息顺序写入一个文件）：
  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
  │msg1 │msg2 │msg3 │msg4 │msg5 │msg6 │ ... │
  │TopA │TopB │TopA │TopC │TopA │TopB │     │
  └─────┴─────┴─────┴─────┴─────┴─────┴─────┘
    ↓           ↓           ↓
  ConsumeQueue（每个 Topic-Queue 一个索引文件）：
  TopicA-Queue0: [msg1-offset][msg3-offset][msg5-offset]
  TopicB-Queue0: [msg2-offset][msg6-offset]
  TopicC-Queue0: [msg4-offset]
```

**设计优势**：
1. **顺序写**：所有消息追加写入 CommitLog，充分利用磁盘顺序写性能（接近内存速度）
2. **随机读优化**：通过 ConsumeQueue 索引 + PageCache 加速消费
3. **单文件**：避免了多文件随机写的性能问题

## 三、Kafka 专题

### Q11：Kafka 为什么吞吐量这么高？

**答：**

Kafka 高吞吐的六大设计：

```
┌─────────────────────────────────────────────────────────┐
│                   Kafka 高吞吐设计                       │
│                                                          │
│  ① 分区并行      Topic 分为多个 Partition，并行读写       │
│  ② 顺序磁盘I/O   追加写入（Append Only），顺序读         │
│  ③ PageCache     利用 OS 页缓存，减少磁盘 I/O           │
│  ④ 零拷贝        sendfile 系统调用，减少数据拷贝次数      │
│  ⑤ 批量处理      批量发送、批量压缩、批量消费             │
│  ⑥ 压缩         支持 GZIP/Snappy/LZ4/ZStd 压缩         │
└─────────────────────────────────────────────────────────┘
```

**零拷贝（Zero-Copy）详解**：
```
传统数据传输（4次拷贝）：
  磁盘 → 内核缓冲区 → 用户空间缓冲区 → Socket缓冲区 → 网卡

零拷贝（2次拷贝，使用 sendfile）：
  磁盘 → 内核缓冲区 → 网卡
  （跳过用户空间，数据直接从内核发送到网卡）
```

**顺序写 vs 随机写性能对比**：

| 操作 | 吞吐量 |
|------|--------|
| 磁盘顺序写 | 600MB/s |
| 磁盘随机写 | 100KB/s |
| 内存随机读写 | ~GB/s |

> Kafka 的磁盘顺序写速度接近内存，这是高吞吐的物理基础。

### Q12：Kafka 的 ISR 机制是什么？

**答：**

ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合。

```
ISR 机制：

  Partition-0 副本分布：
  ┌──────────────────────────────────────────────────┐
  │ Broker-1: Leader  (offset=100)  ← 最新           │
  │ Broker-2: Follower(offset=98)   ← 在ISR内(差2条) │
  │ Broker-3: Follower(offset=60)   ← 不在ISR(差40条)│
  └──────────────────────────────────────────────────┘
  
  ISR = {Broker-1, Broker-2}
  OSR = {Broker-3}（Out-of-Sync Replicas）
```

**ISR 核心规则**：
1. Follower 在 `replica.lag.time.max.ms`（默认 30 秒）内未向 Leader 发送 Fetch 请求，则从 ISR 中移除
2. `acks=all` 时，只有 ISR 中所有副本都确认，消息才算发送成功
3. `min.insync.replicas=2` 表示 ISR 中至少需要 2 个副本，否则拒绝写入

**Leader 选举**：
- Leader 宕机时，从 ISR 中选举新 Leader
- `unclean.leader.election.enable=false`：禁止 OSR 中的副本当选 Leader（防止数据丢失）
- `unclean.leader.election.enable=true`：允许 OSR 当选（优先可用性，可能丢数据）

### Q13：Kafka 的 Rebalance 是什么？有什么问题？如何优化？

**答：**

Rebalance 是 Consumer Group 内分区重新分配的过程。

**触发条件**：
1. Consumer 加入或离开 Group
2. Consumer 心跳超时（被认为已退出）
3. Topic 分区数变化
4. 订阅的 Topic 列表变化

```
Rebalance 过程：

  Before:
    Consumer-1: [P0, P1]    Consumer-2: [P2, P3]

  Consumer-3 加入，触发 Rebalance:
    ① 所有 Consumer 停止消费
    ② Group Coordinator 重新分配分区
    ③ 分配结果：
       Consumer-1: [P0, P1]
       Consumer-2: [P2]
       Consumer-3: [P3]
    ④ 各 Consumer 从新分配的分区继续消费
```

**Rebalance 的问题**：
- **Stop The World**：Rebalance 期间所有 Consumer 停止消费
- **重复消费**：已消费但未提交 Offset 的消息会被重复消费
- **消费延迟**：Rebalance 过程耗时较长

**优化方案**：

| 优化方向 | 具体措施 |
|---------|---------|
| **避免不必要的 Rebalance** | 增大 `session.timeout.ms`（默认 10s → 25s） |
| **加快心跳** | 减小 `heartbeat.interval.ms`（默认 3s） |
| **增大处理超时** | 增大 `max.poll.interval.ms`（默认 5min） |
| **减少单次拉取量** | 减小 `max.poll.records`，避免处理超时 |
| **使用 StickyAssignor** | 尽量保持原有分配，减少分区迁移 |

```properties
# Kafka Consumer 防 Rebalance 配置
session.timeout.ms=25000
heartbeat.interval.ms=3000
max.poll.interval.ms=600000
max.poll.records=200
partition.assignment.strategy=org.apache.kafka.clients.consumer.StickyAssignor
```

### Q14：Kafka 如何实现 Exactly-Once 语义？

**答：**

Kafka Exactly-Once 由三部分组成：

```
┌─────────────────────────────────────────────────────────┐
│               Kafka Exactly-Once 三大机制                │
│                                                          │
│  ① 幂等 Producer（单分区内去重）                          │
│     enable.idempotence=true                              │
│     Producer 给每条消息分配序列号(Sequence Number)         │
│     Broker 对比序列号去重                                 │
│                                                          │
│  ② 事务（跨分区原子写入）                                 │
│     transactional.id="my-tx-id"                          │
│     beginTransaction → send → commitTransaction          │
│     保证多个分区的写入要么全部成功，要么全部失败             │
│                                                          │
│  ③ Consumer 端                                           │
│     isolation.level=read_committed                       │
│     只读取已提交的事务消息                                 │
└─────────────────────────────────────────────────────────┘
```

```java
// Kafka 事务消息示例
@Service
public class KafkaTransactionProducer {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void sendInTransaction(String orderId, String content) {
        kafkaTemplate.executeInTransaction(operations -> {
            // 在事务中发送多条消息，要么全部成功，要么全部回滚
            operations.send("order-topic", orderId, content);
            operations.send("audit-topic", orderId, "ORDER_CREATED");
            return true;
        });
    }
}
```

**三种投递语义对比**：

| 语义 | 含义 | 可能的问题 | Kafka 实现 |
|------|------|-----------|-----------|
| **At Most Once** | 最多投递一次 | 可能丢消息 | acks=0 |
| **At Least Once** | 至少投递一次 | 可能重复 | acks=all |
| **Exactly Once** | 精确一次 | 无 | 幂等+事务 |

### Q15：Kafka 的消费者组（Consumer Group）是怎么工作的？

**答：**

```
Consumer Group 工作原理：

  Topic-A (4 个 Partition):

  情况1：2 个 Consumer（每人消费 2 个 Partition）
    Consumer-1: [P0, P1]
    Consumer-2: [P2, P3]

  情况2：4 个 Consumer（每人消费 1 个 Partition）
    Consumer-1: [P0]
    Consumer-2: [P1]
    Consumer-3: [P2]
    Consumer-4: [P3]

  情况3：5 个 Consumer（有 1 个空闲！）
    Consumer-1: [P0]
    Consumer-2: [P1]
    Consumer-3: [P2]
    Consumer-4: [P3]
    Consumer-5: [空闲] ← 无法分配 Partition
```

**核心规则**：
1. 一个 Partition 只能被同组内的一个 Consumer 消费
2. 消费者数量 > 分区数时，多余的消费者空闲
3. 不同 Consumer Group 之间独立消费（消息广播）

### Q17：如何设计消息幂等方案？

**答：**

**为什么需要幂等**：MQ 只保证 At Least Once（至少一次投递），网络抖动、Rebalance、超时重试都会导致重复消费。

**四种幂等方案**：

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **唯一ID+去重表** | 数据库存消息ID，消费前查询 | 可靠持久 | DB压力大 | 金融级场景 |
| **Redis SETNX** | 原子操作设置幂等键 | 高性能 | Redis宕机失效 | 高并发场景 |
| **唯一约束** | 数据库 UNIQUE KEY | 天然支持 | 只适用插入 | 订单创建 |
| **状态机** | CAS更新，WHERE status=期望值 | 语义清晰 | 只适用有状态场景 | 订单流转 |

```java
// 推荐方案：Redis SETNX 幂等
public void consumeIdempotent(String msgId, OrderDTO order) {
    String key = "mq:idempotent:" + msgId;
    // SETNX + 过期时间，原子操作
    Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 24, TimeUnit.HOURS);
    if (Boolean.FALSE.equals(success)) {
        return;  // 已消费，幂等返回
    }
    try {
        orderService.processOrder(order);
    } catch (Exception e) {
        redisTemplate.delete(key);  // 失败删key，允许重试
        throw e;
    }
}
```

### Q18：消息的事务一致性如何保证？

**答：**

**方案一：RocketMQ 事务消息**（推荐）
- 半消息 + 本地事务 + 事务回查（详见 Q6）

**方案二：本地消息表**

  ┌────────────────────────────────────────────────────┐
  │ 1. 开启本地事务                                     │
  │    ├── 执行业务操作（如扣款）                         │
  │    └── 写入本地消息表（status=待发送）                 │

@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<String, String> template) {
    // 最多重试 3 次，间隔 1 秒
    DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(template);
    return new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 3));
}

## 五、消息积压与顺序专题

### Q20：消息积压了怎么办？

**答：**

**紧急处理方案**（按优先级排列）：

```
第一步：排查原因
  │
  ├── 消费端异常？ → 修复bug/重启消费者
  ├── 消费太慢？ → 优化消费逻辑
  └── 流量突增？ → 执行以下扩容方案

第二步：紧急扩容
  │
  ├── 方案A：增加消费者实例（不超过分区数）
  ├── 方案B：增加分区数 + 消费者数
  └── 方案C：建立临时 Topic（更多分区），转存消费

第三步：降级处理
  │
  ├── 跳过非核心消息（如日志、通知）
  └── 仅处理核心消息（如订单、支付）
```

**长期优化方案**：
1. **提前规划容量**：根据业务量预估分区数和消费者数
2. **消费端优化**：批量处理、多线程消费、异步写库
3. **监控告警**：消费延迟监控，积压量超阈值自动告警
4. **弹性伸缩**：容器化部署，积压时自动扩容消费者

### Q21：如何保证消息的顺序消费？

**答：**

**核心思路**：将需要保序的消息路由到同一分区，分区内 FIFO 保证顺序。

```
顺序消息三要素：

  ① 生产端：同一业务Key的消息 → 同一分区
     rocketMQTemplate.syncSendOrderly(topic, msg, orderId);
     kafkaTemplate.send(topic, orderId, msg);

  ② Broker端：同一分区内消息 FIFO 存储

  ③ 消费端：
     RocketMQ: ConsumeMode.ORDERLY（队列级别加锁，单线程消费）
     Kafka: 一个分区只分配给一个消费者
```

**注意事项**：
- 顺序消费会降低吞吐量（无法并行）
- 某条消息消费失败会阻塞后续消息（RocketMQ ORDERLY 模式）
- 分区数决定了顺序消费的最大并行度

---

### Q22：RocketMQ 和 Kafka 的顺序消息有什么区别？

**答：**

| 维度 | RocketMQ | Kafka |
|------|----------|-------|
| **生产端** | syncSendOrderly + MessageQueueSelector | send(topic, key, value)，按 Key 哈希 |
| **消费端** | ConsumeMode.ORDERLY，队列级锁 | 一个 Partition 只分配一个 Consumer |
| **失败处理** | 消费失败会暂停该队列，等待重试 | 取决于 Offset 提交策略 |
| **并行度** | Queue 数量 | Partition 数量 |
| **粒度** | 支持 Tag 过滤后的顺序 | 分区内严格有序 |

### Q24：秒杀场景中如何使用消息队列？

**答：**

```
秒杀系统架构：

  用户请求 ──→ Nginx(限流) ──→ 秒杀服务 ──→ MQ ──→ 订单服务
                                  │                    │
                              Redis预检              DB操作
                              (库存/重复)          (扣库存+创建订单)
```

**完整流程**：

```java
// 1. 秒杀接口（快速返回）
@PostMapping("/seckill")
public Result seckill(@RequestBody SeckillReq req) {
    // ① Redis 预检：库存是否足够
    Long stock = redisTemplate.opsForValue()
            .decrement("seckill:stock:" + req.getItemId());
    if (stock < 0) {
        redisTemplate.opsForValue()
                .increment("seckill:stock:" + req.getItemId());
        return Result.fail("商品已售罄");
    }

    // ② 重复下单检查（Redis SET）
    Boolean isNew = redisTemplate.opsForSet()
            .add("seckill:users:" + req.getItemId(),
                 req.getUserId().toString()) > 0;
    if (!isNew) {
        redisTemplate.opsForValue()
                .increment("seckill:stock:" + req.getItemId());
        return Result.fail("请勿重复抢购");
    }

    // ③ 发送 MQ，异步处理订单
    rocketMQTemplate.syncSend("seckill-topic",
            JSON.toJSONString(req));

    return Result.success("抢购成功，请等待出单");
}

// 2. 订单消费者（匀速消费）
@RocketMQMessageListener(topic = "seckill-topic",
                          consumerGroup = "seckill-order-group")
public class SeckillOrderConsumer implements RocketMQListener<String> {

    @Override
    public void onMessage(String message) {
        SeckillReq req = JSON.parseObject(message, SeckillReq.class);
        // 真正扣减数据库库存 + 创建订单
        seckillService.createOrder(req);
    }
}
```

**MQ 在秒杀中的作用**：
1. **削峰**：10 万请求入队，下游每秒消费 2000 条
2. **异步**：用户快速得到"抢购成功"响应，订单异步创建
3. **解耦**：秒杀接口与订单服务解耦，互不影响

### Q25：如何用 MQ 实现分布式事务的最终一致性？

**答：**

> 场景：电商下单 → 扣库存 → 加积分，需要保证最终一致性。

```
基于 RocketMQ 事务消息的分布式事务：

  订单服务                  RocketMQ               库存服务    积分服务
     │                        │                      │          │
     │ ① 发送半消息            │                      │          │
     │ ──────────────────→   │                      │          │
     │                        │                      │          │
     │ ② 执行本地事务          │                      │          │
     │ （创建订单）            │                      │          │
     │                        │                      │          │
     │ ③ Commit               │                      │          │
     │ ──────────────────→   │                      │          │
     │                        │ ④ 投递消息            │          │
     │                        │ ──────────────────→ │          │
     │                        │ ──────────────────────────────→│
     │                        │                      │          │
     │                        │     ⑤ 消费+幂等处理    │          │
     │                        │     (扣库存)          │          │
     │                        │                 (加积分)        │
```

**完整代码示例**：

```java
// 订单服务：发送事务消息
@Service
public class OrderService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void createOrder(OrderDTO orderDTO) {
        String txId = UUID.randomUUID().toString();
        Message<OrderDTO> msg = MessageBuilder
                .withPayload(orderDTO)
                .setHeader("txId", txId)
                .build();
        rocketMQTemplate.sendMessageInTransaction(
                "order-created-topic", msg, orderDTO);
    }
}

// 事务监听器
@RocketMQTransactionListener
public class OrderTxListener
        implements RocketMQLocalTransactionListener {

    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
        OrderDTO order = (OrderDTO) arg;
        try {
            // 执行本地事务：创建订单
            orderMapper.insert(order);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        String txId = (String) msg.getHeaders().get("txId");
        boolean exists = orderMapper.existsByTxId(txId);
        return exists ? RocketMQLocalTransactionState.COMMIT
                      : RocketMQLocalTransactionState.ROLLBACK;
    }
}

// 库存服务：消费消息（幂等）
@RocketMQMessageListener(topic = "order-created-topic",
                          consumerGroup = "stock-consumer-group")
public class StockConsumer implements RocketMQListener<MessageExt> {

    @Override
    public void onMessage(MessageExt message) {
        OrderDTO order = JSON.parseObject(
                new String(message.getBody()), OrderDTO.class);
        // 幂等扣减库存
        stockService.deductStock(order.getItemId(), order.getQuantity());
    }
}
```

### Q26：如果让你排查线上消息丢失问题，你会怎么做？

### Q27：消息队列和 RPC 调用有什么区别？什么时候该用哪个？

**答：**

| 维度 | MQ（消息队列） | RPC（远程过程调用） |
|------|--------------|-------------------|
| **通信方式** | 异步 | 同步 |
| **耦合度** | 松耦合（通过 Topic 解耦） | 强耦合（直接调用） |
| **实时性** | 非实时（有消费延迟） | 实时（立即返回结果） |
| **可靠性** | 消息持久化，不易丢失 | 依赖网络，可能超时 |
| **吞吐量** | 高（异步 + 批量） | 受限于被调方性能 |
| **适用场景** | 异步处理、解耦、削峰 | 需要即时返回结果 |

```
选择建议：

  需要立即得到结果？ ──→ RPC（如：查询用户信息、校验权限）
  不需要立即处理？ ──→ MQ（如：发通知、记日志、更新缓存）
  需要削峰限流？ ──→ MQ
  需要广播通知多个服务？ ──→ MQ
  强一致性、实时响应？ ──→ RPC
```
