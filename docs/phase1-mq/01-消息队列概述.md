## 一、什么是消息队列

> 消息队列（Message Queue，简称 MQ）是一种应用间的异步通信机制，通过消息中间件在生产者和消费者之间传递消息，实现系统的解耦、异步和削峰。

### 1.1 核心思想

```
┌──────────┐     消息      ┌──────────────┐     消息      ┌──────────┐
│ Producer │ ──────────→  │  Message Queue │ ──────────→  │ Consumer │
│  生产者   │              │   消息队列      │              │  消费者   │
└──────────┘              └──────────────┘              └──────────┘
```

**类比理解：**
- 消息队列就像**邮局**——寄信人（Producer）将信件投递到邮局（MQ），收信人（Consumer）从邮局取信
- 寄信人不需要等收信人在家，邮局负责暂存和投递

### 1.2 为什么要使用消息队列

| 问题场景 | 不使用 MQ | 使用 MQ |
|---------|----------|---------|
| 系统耦合 | A 系统直接调用 B/C/D，任一宕机影响整体 | A 只发消息，B/C/D 各自消费 |
| 响应时间 | 串行调用累加延迟，用户体验差 | 核心逻辑完成即返回，其余异步处理 |
| 流量突增 | 瞬时高并发压垮数据库 | MQ 缓冲请求，下游匀速消费 |
| 通信方式 | 同步 RPC 强依赖，超时重试复杂 | 异步通信，天然支持重试与补偿 |

## 二、消息队列三大应用场景

### 2.1 异步处理

**场景：用户注册后需要发送短信和邮件通知**

**同步处理（无 MQ）：**

```
用户注册 → 写入数据库(50ms) → 发送短信(200ms) → 发送邮件(200ms) → 返回结果
总耗时: 450ms
```

**异步处理（使用 MQ）：**

```
用户注册 → 写入数据库(50ms) → 发送MQ消息(5ms) → 返回结果
                                    │
                                    ├──→ 短信服务消费消息 → 发送短信
                                    └──→ 邮件服务消费消息 → 发送邮件
总耗时: 55ms（用户感知）
```

```java
// 注册服务 - 发送消息
@Service
public class UserService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void register(User user) {
        // 1. 核心业务：写入数据库
        userMapper.insert(user);

        // 2. 发送异步消息，通知短信和邮件服务
        rocketMQTemplate.convertAndSend("user-register-topic", user);
    }
}

// 短信服务 - 消费消息
@Component
@RocketMQMessageListener(topic = "user-register-topic",
                          consumerGroup = "sms-consumer-group")
public class SmsConsumer implements RocketMQListener<User> {
    @Override
    public void onMessage(User user) {
        smsService.sendSms(user.getPhone(), "注册成功");
    }
}
```

### 2.2 应用解耦

**场景：订单系统需要通知库存、物流、积分等多个下游系统**

**强耦合模式（无 MQ）：**

```
┌──────────┐     ┌──────────┐
│          │────→│ 库存系统  │
│          │     └──────────┘
│  订单系统 │────→┌──────────┐
│          │     │ 物流系统  │
│          │     └──────────┘
│          │────→┌──────────┐
└──────────┘     │ 积分系统  │
                 └──────────┘
问题：新增一个下游系统，订单系统需要修改代码！
```

**解耦模式（使用 MQ）：**

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│          │     │          │←────│ 库存系统  │
│          │     │          │     └──────────┘
│  订单系统 │────→│  消息队列  │←────┌──────────┐
│          │     │          │     │ 物流系统  │
│          │     │          │     └──────────┘
└──────────┘     │          │←────┌──────────┐
                 └──────────┘     │ 积分系统  │
                                  └──────────┘
优点：新增下游系统只需订阅消息，订单系统无需修改！
```

### 2.3 流量削峰

**场景：秒杀活动，瞬时 10 万请求冲击数据库**

```
不使用 MQ：
  10万请求/秒 ──→ 应用服务器 ──→ 数据库（直接被压垮 ❌）

使用 MQ：
  10万请求/秒 ──→ 应用服务器 ──→ MQ（缓冲 10 万消息）
                                    │
                                    └──→ 数据库（每秒消费 2000 条，50秒处理完 ✅）
```

```java
// 秒杀请求入队
@PostMapping("/seckill")
public Result seckill(@RequestBody SeckillRequest request) {
    // 1. 基本校验（库存预检、重复下单检查）
    if (!preCheck(request)) {
        return Result.fail("秒杀失败");
    }

    // 2. 请求入队，快速返回
    rocketMQTemplate.syncSend("seckill-topic", request);
    return Result.success("排队中，请稍候查询结果");
}

// 消费端匀速处理
@RocketMQMessageListener(topic = "seckill-topic",
                          consumerGroup = "seckill-consumer-group")
public class SeckillConsumer implements RocketMQListener<SeckillRequest> {
    @Override
    public void onMessage(SeckillRequest request) {
        // 真正的扣减库存 + 创建订单（匀速消费）
        seckillService.doSeckill(request);
    }
}
```

## 三、消息模型

### 3.1 点对点模型（Queue）

> 每条消息**只能被一个消费者**消费，消费后从队列中移除。

```
Producer A ──→ ┌─────────────────┐ ──→ Consumer A
               │  Queue（队列）    │
Producer B ──→ │  [msg1][msg2]... │ ──→ Consumer B
               └─────────────────┘
                                       （msg1 只会被 A 或 B 其中之一消费）
```

**特点：**
- 一条消息只被消费一次
- 消费者之间是**竞争关系**
- 适合任务分发场景（如订单处理）

### 3.2 发布/订阅模型（Topic）

> 消息发布到 Topic，所有**订阅该 Topic 的消费者**都能收到消息。

```
                                  ┌──→ Subscriber A（收到 msg1）
Producer ──→ ┌──────────────┐ ──→├──→ Subscriber B（收到 msg1）
             │ Topic（主题）  │    └──→ Subscriber C（收到 msg1）
             │  [msg1]       │
             └──────────────┘
```

**特点：**
- 一条消息被**多个订阅者**消费
- 消费者之间是**独立关系**
- 适合事件广播场景（如配置变更通知）

### 3.3 两种模型的对比

| 维度 | 点对点（Queue） | 发布/订阅（Topic） |
|------|----------------|-------------------|
| 消费关系 | 竞争消费 | 独立消费 |
| 消息是否保留 | 消费后删除 | 多个订阅者都能消费 |
| 扩展性 | 增加消费者提高吞吐 | 增加订阅者扩展功能 |
| 典型场景 | 任务分发、负载均衡 | 事件通知、数据同步 |

> **注意：** RocketMQ 和 Kafka 实际上都是基于发布/订阅模型实现的，通过消费者组（Consumer Group）机制可以同时支持两种语义。

## 四、主流消息队列对比

### 4.1 综合对比表

| 维度 | RabbitMQ | RocketMQ | Kafka | Pulsar |
|------|----------|----------|-------|--------|
| **开发语言** | Erlang | Java | Scala/Java | Java |
| **单机吞吐** | 万级 | 十万级 | 百万级 | 十万级 |
| **消息延迟** | 微秒级 | 毫秒级 | 毫秒级 | 毫秒级 |
| **可用性** | 高（镜像队列） | 非常高（主从+Dledger） | 非常高（副本机制） | 非常高（BookKeeper） |
| **消息可靠性** | 高 | 高 | 高（配置得当） | 高 |
| **功能特性** | 丰富（延迟队列、优先级） | 丰富（事务、延迟、顺序） | 较少（专注流处理） | 丰富（多租户、分层存储） |
| **协议支持** | AMQP、STOMP、MQTT | 自定义协议 | 自定义协议 | 自定义 + 兼容Kafka |
| **社区生态** | 活跃（国际） | 活跃（国内，Apache顶级项目） | 非常活跃（大数据生态） | 快速增长（云原生） |
| **运维难度** | 低 | 中 | 中 | 高 |
| **适用场景** | 中小项目、复杂路由 | 电商、金融、互联网业务 | 大数据、日志、流计算 | 云原生、多租户 |

### 4.2 选型速查

```
需要复杂路由规则？ ──→ RabbitMQ
需要事务消息 + 延迟消息？ ──→ RocketMQ
需要超高吞吐 + 流处理？ ──→ Kafka
需要云原生 + 多租户？ ──→ Pulsar
```

## 五、核心概念

### 5.1 基本组件

```
┌─────────────────────────────────────────────────────────┐
│                    Message Queue 系统                     │
│                                                          │
│  ┌──────────┐    ┌─────────────────────┐   ┌──────────┐ │
│  │ Producer │───→│       Broker        │──→│ Consumer │ │
│  │  生产者   │    │  ┌───────────────┐  │   │  消费者   │ │
│  └──────────┘    │  │ Topic / Queue │  │   └──────────┘ │
│                  │  │  ┌──────────┐  │  │                │
│                  │  │  │Partition │  │  │                │
│                  │  │  │  分区     │  │  │                │
│                  │  │  └──────────┘  │  │                │
│                  │  └───────────────┘  │                │
│                  └─────────────────────┘                │
└─────────────────────────────────────────────────────────┘
```

| 概念 | 说明 |
|------|------|
| **Producer（生产者）** | 消息的发送方，负责创建消息并发送到 Broker |
| **Consumer（消费者）** | 消息的接收方，从 Broker 拉取或接收消息并处理 |
| **Broker（代理服务器）** | MQ 服务端，负责消息的存储、转发和管理 |
| **Topic（主题）** | 消息的逻辑分类，一个 Topic 可包含多个 Partition |
| **Queue（队列）** | 消息的物理存储单元，消费者从 Queue 中获取消息 |
| **Partition（分区）** | Topic 的物理分片，用于提高并行度和吞吐量 |
| **Consumer Group（消费者组）** | 一组消费者的集合，组内竞争消费，组间独立消费 |
| **Offset（偏移量）** | 消费进度标识，记录消费者消费到的位置 |

### 5.2 消费模式：Push vs Pull

| 维度 | Push（推模式） | Pull（拉模式） |
|------|--------------|---------------|
| **消息获取** | Broker 主动推送 | Consumer 主动拉取 |
| **实时性** | 高，消息到达立即推送 | 较低，取决于拉取间隔 |
| **消费速率** | Broker 控制，可能压垮消费者 | Consumer 控制，按需拉取 |
| **实现复杂度** | Broker 维护消费状态，较复杂 | 消费者自行管理 Offset，较简单 |
| **代表** | RabbitMQ（默认） | Kafka |
| **RocketMQ** | 表面是 Push，实际是**长轮询（Long Polling）** | 也支持 Pull 模式 |

### 5.3 消息确认机制（ACK）

> ACK（Acknowledgement）确保消息被成功处理，是消息可靠性的核心机制。

```
Producer ──发送消息──→ Broker ──投递消息──→ Consumer
                          ←── 存储ACK ──┘          │
                                                    │处理消息
                                                    ↓
                          ←────── 消费ACK ──────────┘
```

**ACK 模式：**

| 模式 | 说明 | 可靠性 | 性能 |
|------|------|--------|------|
| **自动 ACK** | 消息投递后立即确认 | 低（可能丢失） | 高 |
| **手动 ACK** | 消费者处理完成后确认 | 高 | 较低 |
| **批量 ACK** | 累计一批消息后统一确认 | 中 | 中 |

```java
// Kafka 手动提交 Offset（手动 ACK）
@KafkaListener(topics = "order-topic", groupId = "order-group")
public void consume(ConsumerRecord<String, String> record,
                    Acknowledgment ack) {
    try {
        // 处理消息
        orderService.processOrder(record.value());
        // 处理成功后手动 ACK
        ack.acknowledge();
    } catch (Exception e) {
        // 处理失败，不 ACK，消息会被重新投递
        log.error("消费失败", e);
    }
}
```

## 七、大厂常见面试题

### Q1：为什么要使用消息队列？有什么优缺点？

**答：**

**使用消息队列的三大核心原因：**
1. **异步处理**：将耗时操作异步化，提升系统响应速度
2. **应用解耦**：上下游系统通过消息通信，降低耦合度
3. **流量削峰**：高并发请求入队缓冲，保护下游系统

**缺点：**
1. **系统复杂度增加**：需要处理消息丢失、重复消费、消息积压等问题
2. **一致性问题**：异步消费可能导致数据最终一致性延迟
3. **运维成本增加**：MQ 集群本身的高可用和监控

### Q2：如何进行 MQ 技术选型？

**答：**

选型需要考虑以下因素：
1. **业务场景**：电商交易选 RocketMQ，大数据日志选 Kafka，中小项目选 RabbitMQ
2. **性能要求**：高吞吐选 Kafka（百万级TPS），低延迟选 RabbitMQ（微秒级）
3. **功能需求**：需要事务消息选 RocketMQ，需要复杂路由选 RabbitMQ
4. **团队技术栈**：Java 团队更易上手 RocketMQ/Kafka，Erlang 背景可选 RabbitMQ
5. **运维能力**：运维资源有限选 RabbitMQ（最易运维），资源充足可选 Kafka/RocketMQ

### Q3：消息队列如何保证消息不丢失？

**答：**

需要从三个环节保证：
1. **生产端**：使用同步发送 + 失败重试 + 消息落库
2. **Broker 端**：消息持久化 + 主从复制 + 同步刷盘
3. **消费端**：手动 ACK + 消费幂等 + 失败重试

### Q4：消息队列有哪些常见问题？如何解决？

**答：**

| 问题 | 解决方案 |
|------|---------|
| **消息丢失** | 同步发送、持久化、手动ACK |
| **重复消费** | 幂等设计（唯一键、Token、状态机） |
| **消息积压** | 扩容消费者、批量消费、跳过非核心消息 |
| **消息顺序** | 单分区有序、业务ID哈希路由 |
| **消息延迟** | 优化消费逻辑、多线程消费、分区扩展 |

### Q5：Push 模式和 Pull 模式有什么区别？

**答：**

- **Push**：Broker 主动推送，实时性高，但可能压垮消费者。代表：RabbitMQ
- **Pull**：Consumer 主动拉取，消费速率自控，但有拉取间隔延迟。代表：Kafka
- **长轮询**：结合两者优点，Consumer 发起拉取请求，Broker 有消息时返回，没消息时挂起等待。代表：RocketMQ

### Q6：点对点模型和发布/订阅模型的区别？

**答：**

- **点对点（Queue）**：一条消息只被一个消费者消费，消费者之间竞争消费，适合任务分发
- **发布/订阅（Topic）**：一条消息被所有订阅者消费，消费者之间独立消费，适合事件广播
- **消费者组**：RocketMQ/Kafka 通过 Consumer Group 实现两种模型的统一——同组内竞争（Queue 语义），不同组间广播（Topic 语义）
