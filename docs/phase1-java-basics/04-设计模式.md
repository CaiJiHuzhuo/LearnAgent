## 一、单例模式

### 1.1 定义

确保一个类只有一个实例，并提供全局访问点。

```
单例模式结构：

  ┌──────────────────┐
  │    Singleton      │
  ├──────────────────┤
  │ - instance: S     │  ← 唯一实例（static）
  ├──────────────────┤
  │ - Singleton()     │  ← 私有构造
  │ + getInstance(): S│  ← 全局访问点
  └──────────────────┘
```

### 1.2 饿汉式（线程安全）

```java
public class Singleton {
    // 类加载时就创建实例（JVM 保证线程安全）
    private static final Singleton INSTANCE = new Singleton();
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

**优点：** 简单，线程安全。**缺点：** 不支持延迟加载，类加载就创建。

### 1.3 懒汉式（线程不安全）

```java
public class Singleton {
    private static Singleton instance;
    
    private Singleton() {}
    
    // 多线程下可能创建多个实例
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton(); // 非原子操作，线程不安全
        }
        return instance;
    }
}
```

### 1.4 DCL 双重检查锁（推荐）

```java
public class Singleton {
    // volatile 防止指令重排序
    private static volatile Singleton instance;
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {                // 第一次检查（避免不必要的同步）
            synchronized (Singleton.class) {
                if (instance == null) {        // 第二次检查（防止重复创建）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么需要 volatile？**

`new Singleton()` 实际上分三步：
1. 分配内存空间
2. 初始化对象
3. 将引用指向内存地址

JVM 可能重排为 1 → 3 → 2。线程 A 执行 1 → 3 后，线程 B 看到 `instance != null` 直接返回，但对象尚未初始化完成。`volatile` 禁止重排序，保证安全。

### 1.5 静态内部类（推荐）

```java
public class Singleton {
    private Singleton() {}
    
    // 静态内部类在被使用时才加载，JVM 保证线程安全
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**优点：** 延迟加载 + 线程安全 + 无锁，利用了 JVM 类加载机制。

### 1.6 枚举单例（最佳实践）

```java
public enum Singleton {
    INSTANCE;
    
    private final SomeResource resource;
    
    Singleton() {
        resource = new SomeResource();
    }
    
    public void doSomething() {
        resource.execute();
    }
}
// 使用
Singleton.INSTANCE.doSomething();
```

**优点：** 天然线程安全、防止反射攻击、防止反序列化破坏单例。**《Effective Java》推荐的最佳方式。**

### 1.7 五种方式对比

| 方式 | 线程安全 | 延迟加载 | 防反射 | 防序列化 |
|------|---------|---------|--------|---------|
| 饿汉式 | ✅ | ❌ | ❌ | ❌ |
| 懒汉式 | ❌ | ✅ | ❌ | ❌ |
| DCL | ✅ | ✅ | ❌ | ❌ |
| 静态内部类 | ✅ | ✅ | ❌ | ❌ |
| **枚举** | ✅ | ❌ | **✅** | **✅** |

**在 Spring/JDK 中的应用：** `Runtime.getRuntime()` 使用饿汉式；Spring Bean 默认单例（通过容器管理，非传统单例模式）。

## 二、工厂模式

### 2.1 简单工厂

由一个工厂类根据参数创建不同的产品对象。

```
  ┌──────────┐      ┌──────────────────┐
  │  Client   │─────►│  SimpleFactory   │
  └──────────┘      ├──────────────────┤
                    │ +create(type): P  │
                    └────────┬─────────┘
                             │ 创建
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ ProductA │  │ ProductB │  │ ProductC │
        └──────────┘  └──────────┘  └──────────┘
```

```java
public interface Shape {
    void draw();
}

public class Circle implements Shape {
    @Override
    public void draw() { System.out.println("画圆形"); }
}

public class Rectangle implements Shape {
    @Override
    public void draw() { System.out.println("画矩形"); }
}

// 简单工厂
public class ShapeFactory {
    public static Shape create(String type) {
        switch (type) {
            case "circle":    return new Circle();
            case "rectangle": return new Rectangle();
            default: throw new IllegalArgumentException("未知类型: " + type);
        }
    }
}

// 使用
Shape shape = ShapeFactory.create("circle");
shape.draw();
```

**缺点：** 新增产品时需要修改工厂类，违反开闭原则。

### 2.2 工厂方法

每个产品对应一个工厂，新增产品只需新增工厂，不修改已有代码。

```
  ┌──────────────┐          ┌──────────┐
  │ ShapeFactory  │◄─────── │  Shape   │
  │ +create(): S  │          └──────────┘
  └──────┬───────┘               ▲
         │                       │
    ┌────┴─────┐           ┌─────┴─────┐
    ▼          ▼           ▼           ▼
 CircleF   RectF       Circle     Rectangle
```

```java
public interface ShapeFactory {
    Shape create();
}

public class CircleFactory implements ShapeFactory {
    @Override
    public Shape create() { return new Circle(); }
}

public class RectangleFactory implements ShapeFactory {
    @Override
    public Shape create() { return new Rectangle(); }
}

// 使用
ShapeFactory factory = new CircleFactory();
Shape shape = factory.create();
```

### 2.3 抽象工厂

创建**一组相关对象**（产品族），如不同主题的按钮和文本框。

```java
// 抽象产品
public interface Button { void render(); }
public interface TextField { void render(); }

// 具体产品：暗色主题
public class DarkButton implements Button {
    @Override
    public void render() { System.out.println("渲染暗色按钮"); }
}
public class DarkTextField implements TextField {
    @Override
    public void render() { System.out.println("渲染暗色文本框"); }
}

// 具体产品：亮色主题
public class LightButton implements Button {
    @Override
    public void render() { System.out.println("渲染亮色按钮"); }
}
public class LightTextField implements TextField {
    @Override
    public void render() { System.out.println("渲染亮色文本框"); }
}

// 抽象工厂
public interface UIFactory {
    Button createButton();
    TextField createTextField();
}

// 具体工厂
public class DarkThemeFactory implements UIFactory {
    @Override
    public Button createButton() { return new DarkButton(); }
    @Override
    public TextField createTextField() { return new DarkTextField(); }
}

public class LightThemeFactory implements UIFactory {
    @Override
    public Button createButton() { return new LightButton(); }
    @Override
    public TextField createTextField() { return new LightTextField(); }
}
```

**在 Spring/JDK 中的应用：** `BeanFactory` / `ApplicationContext` 是工厂模式的典型应用；`Calendar.getInstance()` 是简单工厂。

## 三、建造者模式

### 3.1 定义

将复杂对象的构建过程与表示分离，使得同样的构建过程可以创建不同的表示。

```java
// 链式建造者（最常用形式）
public class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String, String> headers;
    private final String body;
    private final int timeout;

    private HttpRequest(Builder builder) {
        this.url = builder.url;
        this.method = builder.method;
        this.headers = builder.headers;
        this.body = builder.body;
        this.timeout = builder.timeout;
    }

    public static class Builder {
        private final String url;              // 必填
        private String method = "GET";         // 可选，有默认值
        private Map<String, String> headers = new HashMap<>();
        private String body;
        private int timeout = 5000;

        public Builder(String url) {
            this.url = url;
        }

        public Builder method(String method) {
            this.method = method;
            return this;
        }

        public Builder header(String key, String value) {
            this.headers.put(key, value);
            return this;
        }

        public Builder body(String body) {
            this.body = body;
            return this;
        }

        public Builder timeout(int timeout) {
            this.timeout = timeout;
            return this;
        }

        public HttpRequest build() {
            // 可以在这里做参数校验
            if (url == null || url.isEmpty()) {
                throw new IllegalStateException("URL 不能为空");
            }
            return new HttpRequest(this);
        }
    }
}

// 使用
HttpRequest request = new HttpRequest.Builder("https://api.example.com/users")
    .method("POST")
    .header("Content-Type", "application/json")
    .body("{\"name\": \"张三\"}")
    .timeout(3000)
    .build();
```

**在 Spring/JDK 中的应用：** `StringBuilder`、`Stream.Builder`、Spring 的 `UriComponentsBuilder`、Lombok 的 `@Builder`。

## 四、代理模式

### 4.1 定义

为目标对象提供一个替代品（代理），以控制对目标对象的访问。

```
代理模式结构：

  ┌──────────┐     ┌────────────┐     ┌──────────────┐
  │  Client   │────►│   Proxy    │────►│  RealSubject │
  └──────────┘     ├────────────┤     ├──────────────┤
                   │ -target: S  │     │ +request()   │
                   │ +request()  │     └──────────────┘
                   │  前置处理    │            ▲
                   │  target.req │            │
                   │  后置处理    │     ┌──────┴───────┐
                   └────────────┘     │   Subject     │
                                      │ +request()    │
                                      └───────────────┘
```

### 4.2 静态代理

```java
public interface UserService {
    void save(User user);
}

public class UserServiceImpl implements UserService {
    @Override
    public void save(User user) {
        System.out.println("保存用户: " + user.getName());
    }
}

// 静态代理
public class UserServiceProxy implements UserService {
    private final UserService target;
    
    public UserServiceProxy(UserService target) {
        this.target = target;
    }
    
    @Override
    public void save(User user) {
        System.out.println("【日志】开始保存用户");
        long start = System.currentTimeMillis();
        target.save(user);
        System.out.println("【日志】保存完成，耗时: " +
            (System.currentTimeMillis() - start) + "ms");
    }
}
```

**缺点：** 每个接口都需要一个代理类，代码量大。

### 4.3 JDK 动态代理

基于 **反射 + 接口**，运行时动态生成代理类（代理对象必须实现接口）。

```java
import java.lang.reflect.*;

public class JdkProxyFactory {
    
    @SuppressWarnings("unchecked")
    public static <T> T createProxy(T target) {
        return (T) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            new InvocationHandler() {
                @Override
                public Object invoke(Object proxy, Method method, Object[] args) 
                        throws Throwable {
                    System.out.println("【前置】调用: " + method.getName());
                    long start = System.currentTimeMillis();
                    
                    Object result = method.invoke(target, args); // 调用真实方法
                    
                    System.out.println("【后置】耗时: " +
                        (System.currentTimeMillis() - start) + "ms");
                    return result;
                }
            }
        );
    }
}

// 使用
UserService proxy = JdkProxyFactory.createProxy(new UserServiceImpl());
proxy.save(new User("张三"));
```

### 4.4 CGLIB 动态代理

基于 **ASM 字节码生成**，通过继承目标类来创建代理（不需要接口）。

```java
import net.sf.cglib.proxy.*;

public class CglibProxyFactory {
    
    @SuppressWarnings("unchecked")
    public static <T> T createProxy(Class<T> targetClass) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(targetClass);
        enhancer.setCallback(new MethodInterceptor() {
            @Override
            public Object intercept(Object obj, Method method, Object[] args,
                    MethodProxy proxy) throws Throwable {
                System.out.println("【前置】调用: " + method.getName());
                Object result = proxy.invokeSuper(obj, args); // 调用父类方法
                System.out.println("【后置】完成");
                return result;
            }
        });
        return (T) enhancer.create();
    }
}

// 使用（不需要接口）
UserServiceImpl proxy = CglibProxyFactory.createProxy(UserServiceImpl.class);
proxy.save(new User("李四"));
```

### 4.5 JDK 代理 vs CGLIB 代理

| 维度 | JDK 动态代理 | CGLIB 代理 |
|------|-------------|-----------|
| 实现方式 | 反射 + 接口 | ASM 字节码 + 继承 |
| 要求 | 目标类必须实现接口 | 目标类不能是 final |
| 性能（生成） | 较快 | 较慢（生成字节码） |
| 性能（调用） | JDK 8+ 接近 | 快（MethodProxy 缓存） |
| Spring 默认 | 有接口时使用 | 无接口时使用（SpringBoot 2.x 默认 CGLIB） |

**在 Spring 中的应用：** Spring AOP 底层就是动态代理。`@Transactional`、`@Cacheable`、`@Async` 等注解都依赖代理实现。

## 五、适配器模式

### 5.1 定义

将一个类的接口转换为客户端期望的另一个接口，使原本不兼容的类可以协作。

```
适配器模式结构：

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │  Client   │────►│   Adapter    │────►│   Adaptee    │
  │           │     │ +request()   │     │ +specificReq │
  └──────────┘     └──────────────┘     └──────────────┘
                   将 specificReq 适配为 request
```

```java
// 目标接口：统一的日志接口
public interface Logger {
    void info(String message);
    void error(String message);
}

// 被适配者：第三方日志库
public class ThirdPartyLogger {
    public void writeLog(int level, String msg) {
        System.out.println("[Level " + level + "] " + msg);
    }
}

// 适配器
public class LoggerAdapter implements Logger {
    private final ThirdPartyLogger adaptee;
    
    public LoggerAdapter(ThirdPartyLogger adaptee) {
        this.adaptee = adaptee;
    }
    
    @Override
    public void info(String message) {
        adaptee.writeLog(1, message);
    }
    
    @Override
    public void error(String message) {
        adaptee.writeLog(3, message);
    }
}
```

**在 Spring/JDK 中的应用：**
- `InputStreamReader`：将 `InputStream`（字节流）适配为 `Reader`（字符流）
- Spring MVC 的 `HandlerAdapter`：适配不同类型的 Controller

## 六、装饰器模式

### 6.1 定义

动态地给对象添加额外职责，比继承更灵活。

```
装饰器模式结构：

  ┌──────────────┐
  │  Component    │ ◄─── 抽象组件
  │ +operation()  │
  └──────┬───────┘
         │
    ┌────┴────────────────┐
    ▼                     ▼
 Concrete            ┌──────────────┐
 Component           │  Decorator   │ ◄─── 持有 Component 引用
                     │ -component   │
                     │ +operation() │
                     └──────┬───────┘
                            │
                   ┌────────┴────────┐
                   ▼                 ▼
            DecoratorA          DecoratorB
```

```java
// Java IO 就是经典的装饰器模式
InputStream raw = new FileInputStream("data.txt");         // 原始组件
InputStream buffered = new BufferedInputStream(raw);        // 装饰：缓冲
InputStream data = new DataInputStream(buffered);           // 装饰：基本类型读取

// 自定义装饰器示例
public interface DataSource {
    void writeData(String data);
    String readData();
}

public class FileDataSource implements DataSource {
    private String filename;
    
    @Override
    public void writeData(String data) { /* 写文件 */ }
    
    @Override
    public String readData() { /* 读文件 */ return ""; }
}

// 加密装饰器
public class EncryptionDecorator implements DataSource {
    private final DataSource wrappee;
    
    public EncryptionDecorator(DataSource source) {
        this.wrappee = source;
    }
    
    @Override
    public void writeData(String data) {
        wrappee.writeData(encrypt(data)); // 加密后写入
    }
    
    @Override
    public String readData() {
        return decrypt(wrappee.readData()); // 读取后解密
    }
    
    private String encrypt(String data) { return Base64.getEncoder().encodeToString(data.getBytes()); }
    private String decrypt(String data) { return new String(Base64.getDecoder().decode(data)); }
}

// 压缩装饰器
public class CompressionDecorator implements DataSource {
    private final DataSource wrappee;
    
    public CompressionDecorator(DataSource source) {
        this.wrappee = source;
    }
    
    @Override
    public void writeData(String data) {
        wrappee.writeData(compress(data));
    }
    
    @Override
    public String readData() {
        return decompress(wrappee.readData());
    }
    
    private String compress(String data) { /* 压缩 */ return data; }
    private String decompress(String data) { /* 解压 */ return data; }
}

// 组合使用：先加密再压缩
DataSource source = new CompressionDecorator(
    new EncryptionDecorator(
        new FileDataSource("data.txt")));
source.writeData("敏感数据");
```

**装饰器 vs 代理的区别：**
- **代理**：控制访问，客户端通常不知道代理的存在
- **装饰器**：增强功能，客户端主动组合装饰器

## 七、策略模式

### 7.1 定义

定义一系列算法，将每个算法封装起来，使它们可以互相替换。

```
策略模式结构：

  ┌─────────────────┐
  │    Context       │
  │ -strategy: S     │
  │ +execute()       │───► strategy.doAlgorithm()
  └─────────────────┘
           │
           ▼
  ┌─────────────────┐
  │    Strategy      │ ◄─── 策略接口
  │ +doAlgorithm()   │
  └────────┬────────┘
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  StratA  StratB  StratC
```

```java
// 支付策略
public interface PaymentStrategy {
    void pay(double amount);
}

public class AlipayStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("支付宝支付: " + amount + " 元");
    }
}

public class WechatPayStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("微信支付: " + amount + " 元");
    }
}

// 上下文
public class PaymentContext {
    private PaymentStrategy strategy;
    
    public void setStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }
    
    public void executePayment(double amount) {
        strategy.pay(amount);
    }
}

// Java 8 简化（Lambda 替代策略类）
Map<String, PaymentStrategy> strategies = Map.of(
    "alipay",  amount -> System.out.println("支付宝: " + amount),
    "wechat",  amount -> System.out.println("微信: " + amount),
    "unionpay", amount -> System.out.println("银联: " + amount)
);
strategies.get("alipay").pay(100.0);
```

**在 Spring/JDK 中的应用：**
- `Comparator` 就是排序策略
- Spring 的 `Resource` 接口（ClassPathResource / FileSystemResource / UrlResource）
- `ThreadPoolExecutor` 的拒绝策略

## 八、观察者模式

### 8.1 定义

定义对象间的一对多依赖关系，当被观察者状态改变时，所有观察者都会收到通知。

```
观察者模式结构：

  ┌──────────────────┐          ┌──────────────┐
  │    Subject        │ ◄─ 注册 │   Observer   │
  │ -observers: List  │          │ +update()    │
  │ +attach(observer) │          └──────────────┘
  │ +detach(observer) │               ▲
  │ +notify()         │          ┌────┴─────┐
  └──────────────────┘          ▼          ▼
    状态改变时                ObserverA  ObserverB
    遍历通知所有observer
```

```java
// 事件系统
public interface EventListener {
    void onEvent(String eventType, Object data);
}

public class EventBus {
    private final Map<String, List<EventListener>> listeners = new ConcurrentHashMap<>();

    public void subscribe(String eventType, EventListener listener) {
        listeners.computeIfAbsent(eventType, k -> new CopyOnWriteArrayList<>())
                 .add(listener);
    }

    public void unsubscribe(String eventType, EventListener listener) {
        List<EventListener> list = listeners.get(eventType);
        if (list != null) list.remove(listener);
    }

    public void publish(String eventType, Object data) {
        List<EventListener> list = listeners.get(eventType);
        if (list != null) {
            list.forEach(listener -> listener.onEvent(eventType, data));
        }
    }
}

// 使用
EventBus bus = new EventBus();
bus.subscribe("order.created", (type, data) ->
    System.out.println("发送确认邮件: " + data));
bus.subscribe("order.created", (type, data) ->
    System.out.println("扣减库存: " + data));

bus.publish("order.created", "订单#12345");
```

**在 Spring/JDK 中的应用：**
- Spring 的 `ApplicationEvent` + `ApplicationListener`
- JDK 的 `java.util.Observable`（已废弃）
- Guava 的 `EventBus`

## 九、责任链模式

### 9.1 定义

将请求沿着处理者链传递，每个处理者决定是否处理请求以及是否传递给下一个处理者。

```
责任链结构：

  请求 → [Handler1] → [Handler2] → [Handler3] → ...
            │             │            │
            ▼             ▼            ▼
          能处理？      能处理？      能处理？
          是→处理       是→处理       是→处理
          否→传递       否→传递       否→传递
```

```java
// 审批链
public abstract class ApprovalHandler {
    protected ApprovalHandler next;

    public ApprovalHandler setNext(ApprovalHandler next) {
        this.next = next;
        return next; // 返回 next 支持链式设置
    }

    public abstract void approve(PurchaseRequest request);
}

public class ManagerApproval extends ApprovalHandler {
    @Override
    public void approve(PurchaseRequest request) {
        if (request.getAmount() <= 5000) {
            System.out.println("经理审批通过: " + request.getAmount() + " 元");
        } else if (next != null) {
            next.approve(request);
        }
    }
}

public class DirectorApproval extends ApprovalHandler {
    @Override
    public void approve(PurchaseRequest request) {
        if (request.getAmount() <= 20000) {
            System.out.println("总监审批通过: " + request.getAmount() + " 元");
        } else if (next != null) {
            next.approve(request);
        }
    }
}

public class CEOApproval extends ApprovalHandler {
    @Override
    public void approve(PurchaseRequest request) {
        System.out.println("CEO 审批通过: " + request.getAmount() + " 元");
    }
}

// 构建链
ApprovalHandler chain = new ManagerApproval();
chain.setNext(new DirectorApproval()).setNext(new CEOApproval());

chain.approve(new PurchaseRequest(3000));   // 经理审批
chain.approve(new PurchaseRequest(15000));  // 总监审批
chain.approve(new PurchaseRequest(50000));  // CEO审批
```

**在 Spring/JDK 中的应用：**
- Servlet 的 `Filter` 链
- Spring Security 的 `FilterChain`
- Netty 的 `ChannelPipeline`
- MyBatis 的 `Plugin`（拦截器链）

## 十、模板方法模式

### 10.1 定义

定义算法的骨架，将某些步骤延迟到子类实现。

```
模板方法结构：

  ┌───────────────────────┐
  │  AbstractClass         │
  │ +templateMethod()      │  ← final，定义算法骨架
  │   ├─ step1()           │  ← 具体方法
  │   ├─ step2()           │  ← 抽象方法（子类实现）
  │   └─ step3()           │  ← 钩子方法（可选覆写）
  └───────────┬───────────┘
              │
       ┌──────┴──────┐
       ▼             ▼
   ConcreteA     ConcreteB
   step2()实现   step2()实现
```

```java
public abstract class DataParser {
    
    // 模板方法：定义解析流程
    public final void parse(String filePath) {
        String rawData = readFile(filePath);
        Object parsed = parseData(rawData);
        if (shouldValidate()) {          // 钩子方法
            validate(parsed);
        }
        processResult(parsed);
    }
    
    // 具体方法（通用步骤）
    private String readFile(String path) {
        System.out.println("读取文件: " + path);
        return "raw data";
    }
    
    // 抽象方法（子类必须实现）
    protected abstract Object parseData(String rawData);
    
    // 抽象方法
    protected abstract void processResult(Object data);
    
    // 钩子方法（子类可选覆写）
    protected boolean shouldValidate() { return true; }
    
    protected void validate(Object data) {
        System.out.println("数据校验通过");
    }
}

// JSON 解析器
public class JsonParser extends DataParser {
    @Override
    protected Object parseData(String rawData) {
        System.out.println("解析 JSON 数据");
        return new JSONObject(rawData);
    }
    
    @Override
    protected void processResult(Object data) {
        System.out.println("处理 JSON 结果");
    }
}

// CSV 解析器
public class CsvParser extends DataParser {
    @Override
    protected Object parseData(String rawData) {
        System.out.println("解析 CSV 数据");
        return rawData.split(",");
    }
    
    @Override
    protected void processResult(Object data) {
        System.out.println("处理 CSV 结果");
    }
    
    @Override
    protected boolean shouldValidate() { return false; } // 跳过校验
}
```

**在 Spring/JDK 中的应用：**
- `AbstractList` 中的 `get()` 是抽象方法，`indexOf()` 是模板方法
- Spring 的 `JdbcTemplate`、`RestTemplate`
- `HttpServlet` 中的 `service()` 方法调用 `doGet()`/`doPost()`

## 十一、大厂常见面试题

### Q1：单例模式中，DCL 为什么需要 volatile？

**答：**

`instance = new Singleton()` 不是原子操作，分为三步：
1. 分配内存
2. 初始化对象
3. 将引用指向内存

JVM 可能重排为 1 → 3 → 2。若线程 A 执行完 1、3，线程 B 在第一次检查时发现 `instance != null`，直接返回了一个**尚未初始化完成**的对象。`volatile` 通过内存屏障禁止这种重排序。

### Q2：Spring AOP 用的是什么代理？

**答：**

- 目标类**实现了接口** → 默认使用 **JDK 动态代理**
- 目标类**未实现接口** → 使用 **CGLIB 代理**
- SpringBoot 2.x 起默认全部使用 CGLIB（`spring.aop.proxy-target-class=true`）

可通过配置切换：
```yaml
spring:
  aop:
    proxy-target-class: false  # 强制使用 JDK 动态代理
```

---

### Q3：装饰器模式和代理模式的区别？

**答：**

| 维度 | 装饰器模式 | 代理模式 |
|------|-----------|---------|
| 目的 | **增强功能** | **控制访问** |
| 使用方 | 客户端主动组合装饰器 | 客户端通常不知道代理存在 |
| 关注点 | 给对象动态添加职责 | 控制对象的创建和访问 |
| 典型场景 | Java IO、BufferedXxx | Spring AOP、RPC 远程调用 |

---

### Q4：策略模式和工厂模式的区别？

**答：**

- **工厂模式**关注的是**创建对象**，解决的是 "创建什么" 的问题
- **策略模式**关注的是**行为封装**，解决的是 "怎么做" 的问题

它们经常结合使用：工厂负责创建策略对象，策略负责封装算法。

---

### Q5：责任链模式在 Spring 中有哪些应用？

**答：**

1. **Spring Security FilterChain**：请求依次经过多个 Filter（认证、授权、CSRF 等），每个 Filter 决定是否放行
2. **Spring MVC HandlerInterceptor**：`preHandle` → Controller → `postHandle` → `afterCompletion`
3. **Spring AOP**：多个切面形成拦截器链，按 `@Order` 排序执行
4. **Spring Web 的 `WebFilter`**（WebFlux）：响应式 Filter 链
