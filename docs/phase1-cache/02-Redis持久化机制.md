## 一、持久化概述

Redis 是内存数据库，一旦进程退出，数据就会丢失。**持久化** 就是将内存中的数据写入磁盘，保证重启后数据可恢复。

### 1.1 三种持久化方式

| 方式 | 原理 | 引入版本 |
|------|------|---------|
| **RDB** | 某个时间点的全量内存快照 | 1.0 |
| **AOF** | 记录每条写命令的日志 | 1.0 |
| **混合持久化** | RDB 快照 + 增量 AOF 日志 | 4.0 |

### 1.2 核心问题

```
内存数据 ──持久化──► 磁盘文件
    │                    │
    │   Redis 宕机重启    │
    │                    │
    ◄──── 数据恢复 ◄─────┘
```

## 二、RDB 持久化

### 2.1 什么是 RDB

RDB（Redis Database）将某个时间点的 **全量内存数据** 生成一个压缩的二进制快照文件（默认 `dump.rdb`）。

### 2.2 触发方式

#### 2.2.1 手动触发

| 命令 | 说明 |
|------|------|
| `SAVE` | **同步**执行，阻塞主线程直到 RDB 完成，**生产环境禁用** |
| `BGSAVE` | **异步**执行，fork 子进程在后台生成 RDB，主线程继续处理请求 |

#### 2.2.2 自动触发

```conf
# redis.conf 默认配置
save 900 1      # 900秒内至少1个key被修改
save 300 10     # 300秒内至少10个key被修改
save 60 10000   # 60秒内至少10000个key被修改
```

#### 2.2.3 其他触发场景

- 主从复制时，主节点自动执行 `BGSAVE` 生成 RDB 发送给从节点
- 执行 `SHUTDOWN` 命令时，如果没有开启 AOF，会自动执行 `BGSAVE`
- 执行 `DEBUG RELOAD` 命令时

### 2.3 BGSAVE 执行流程（核心考点 ⭐⭐⭐）

```
┌────────────┐     1. fork()     ┌────────────┐
│            │ ───────────────► │            │
│  主进程     │                  │  子进程     │
│ (继续处理   │                  │ (生成 RDB) │
│  客户端请求) │                  │            │
│            │                  │            │
│ 页表 ──────┼──► 共享内存页     │ 页表 ──────┤
│            │    ┌─────────┐  │            │
│            │    │ Page A  │  │            │
│            │    │ Page B  │  │            │
│            │    │ Page C  │  │            │
│            │    └─────────┘  │            │
└────────────┘                  └────────────┘

        2. 主进程修改 Page B（触发 Copy-On-Write）

                  ┌──────────┐
                  │ Page A   │◄── 子进程仍读旧数据
                  │ Page B   │◄── 子进程读到快照时刻的值
                  │ Page C   │
                  └──────────┘
                  ┌──────────┐
                  │ Page B'  │◄── 主进程写入新页面
                  └──────────┘
```

### 2.4 fork() + Copy-On-Write 详解

| 步骤 | 说明 |
|------|------|
| 1. `fork()` | 主进程创建子进程，**子进程复制父进程的页表**，而不是复制整个内存 |
| 2. 共享内存 | fork 后，父子进程的页表指向 **相同的物理内存页**，标记为只读 |
| 3. Copy-On-Write | 当主进程 **写入** 某个页时，操作系统复制该页，主进程修改副本 |
| 4. 子进程读取 | 子进程始终读到 fork 时刻的一致性快照数据 |

> ⚠️ **注意**：fork 的耗时与 Redis 内存使用量成正比。如果 Redis 占用 10GB 内存，fork 可能耗时几十毫秒到几百毫秒，期间主线程会 **短暂阻塞**。

### 2.5 RDB 文件格式

```
┌──────────┬─────────┬──────────┬───────────┬──────────┬──────────┐
│ "REDIS"  │ RDB版本 │ 辅助字段  │ 数据库数据 │ Lua脚本  │ EOF校验  │
│ (5字节)  │ (4字节) │ (可选)   │ (Key-Value)│ (可选)   │ (8字节)  │
└──────────┴─────────┴──────────┴───────────┴──────────┴──────────┘
```

### 2.6 RDB 优缺点

| 优点 | 缺点 |
|------|------|
| RDB 是紧凑的二进制文件，**体积小** | 无法做到秒级持久化，可能 **丢失最后一次快照后的数据** |
| 恢复速度 **远快于** AOF | fork 子进程时，大内存实例可能 **短暂阻塞** |
| 适合做 **全量备份和灾难恢复** | fork + COW 可能导致 **内存翻倍**（极端情况） |
| 对 Redis 主进程性能影响较小 | 不适合对数据完整性要求很高的场景 |

### 2.7 RDB 相关配置

```conf
# 文件名
dbfilename dump.rdb

# 文件存储目录
dir /var/lib/redis

# 是否压缩（LZF 算法）
rdbcompression yes

# 是否校验（CRC64）
rdbchecksum yes

# bgsave 出错时是否停止写入
stop-writes-on-bgsave-error yes
```

## 三、AOF 持久化

### 3.1 什么是 AOF

AOF（Append Only File）以 **日志追加** 的方式记录每条 **写命令**，重启时通过重放命令恢复数据。

### 3.2 AOF 写入流程（核心考点 ⭐⭐⭐）

```
客户端写命令
     │
     ▼
┌──────────┐    1. 命令追加    ┌──────────────┐
│ Redis    │ ──────────────► │ AOF 缓冲区    │
│ 命令执行  │                 │ (aof_buf)    │
└──────────┘                 └──────┬───────┘
                                    │
                             2. 刷盘策略
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              ┌──────────┐   ┌──────────┐   ┌──────────┐
              │ always   │   │ everysec │   │   no     │
              │ 每条命令  │   │ 每秒刷盘  │   │ OS 决定  │
              │ 立即刷盘  │   │ (默认)   │   │          │
              └────┬─────┘   └────┬─────┘   └────┬─────┘
                   │              │              │
                   ▼              ▼              ▼
              ┌─────────────────────────────────────┐
              │          AOF 文件 (磁盘)             │
              │  *2\r\n$6\r\nSELECT\r\n$1\r\n0...  │
              └─────────────────────────────────────┘
```

**关键点**：Redis 先执行命令，再写 AOF 日志。这样即使写日志失败，也不会记录错误命令。

### 3.3 三种刷盘策略

| 策略 | 说明 | 数据安全 | 性能 |
|------|------|---------|------|
| `always` | 每条写命令执行后立即调用 `fsync()` 刷盘 | **最高**（最多丢 1 条命令） | **最差** |
| `everysec` | 每秒由后台线程调用一次 `fsync()`（**默认推荐**） | 最多丢 **1 秒** 数据 | **折中** |
| `no` | 不主动刷盘，由操作系统决定（通常 30 秒） | 最多丢 **30 秒** 数据 | **最好** |

```conf
# redis.conf
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
```

### 3.4 AOF 文件格式（RESP 协议）

AOF 文件以 Redis 的 RESP（Redis Serialization Protocol）格式存储：

```
*3          ← 数组长度为 3
$3          ← 第一个字符串长度为 3
SET         ← 命令
$4          ← 第二个字符串长度为 4
name        ← key
$8          ← 第三个字符串长度为 8
zhangsan    ← value
```

对应命令：`SET name zhangsan`

### 3.5 AOF 重写机制（BGREWRITEAOF）（核心考点 ⭐⭐⭐）

随着时间推移，AOF 文件会越来越大（包含大量冗余命令）。AOF 重写可以 **压缩** AOF 文件。

**重写原理**：不是分析旧 AOF 文件，而是读取当前内存中的数据，用 **最少的命令** 重新生成 AOF 文件。

```
重写前 AOF 文件：                   重写后 AOF 文件：
SET name "Tom"                    SET name "Amy"
SET name "Jim"                    RPUSH list "a" "b" "c"
SET name "Amy"
RPUSH list "a"
RPUSH list "b"
RPUSH list "c"

6 条命令 → 压缩为 2 条命令
```

### 3.6 AOF 重写流程

```
┌────────────┐                    ┌────────────┐
│  主进程     │   1. fork()        │  子进程     │
│            │ ──────────────►   │            │
│            │                    │ 2. 遍历内存 │
│            │                    │    生成新AOF│
│ 3. 继续处理 │                    │            │
│    客户端   │                    │            │
│            │                    │            │
│ 4. 新写命令 │                    │ 5. 新AOF   │
│    同时写入 │                    │    写入完成 │
│    ├► AOF缓冲区                 │            │
│    └► AOF重写缓冲区             │            │
│            │                    │            │
│            │   6. 子进程通知完成  │            │
│            │ ◄────────────────  │            │
│            │                    └────────────┘
│ 7. 将重写缓冲区                 
│    的增量数据追加                
│    到新 AOF 文件                
│                                
│ 8. 原子替换旧 AOF               
└────────────┘                    
```

### 3.7 AOF 重写时的增量数据处理

AOF 重写期间，主进程仍在处理新的写命令。这些增量命令需要 **同时写入两个地方**：

| 缓冲区 | 作用 |
|--------|------|
| **AOF 缓冲区**（aof_buf） | 保证旧 AOF 文件的完整性（万一重写失败，旧文件仍可用） |
| **AOF 重写缓冲区**（aof_rewrite_buf） | 子进程完成后，追加到新 AOF 文件尾部，保证数据一致性 |

### 3.8 AOF 重写触发条件

```conf
# 自动触发条件（同时满足）
auto-aof-rewrite-percentage 100   # AOF 文件大小超过上次重写后的 100%
auto-aof-rewrite-min-size 64mb    # AOF 文件至少达到 64MB
```

手动触发：`BGREWRITEAOF`

### 3.9 AOF 优缺点

| 优点 | 缺点 |
|------|------|
| 数据安全性高（最多丢 1 秒数据） | AOF 文件通常 **比 RDB 大** |
| AOF 文件可读，便于分析和修复 | 恢复速度 **比 RDB 慢**（需要重放命令） |
| `everysec` 策略性能影响小 | 每秒 `fsync` 仍有一定 IO 开销 |

### 3.10 AOF 文件修复

当 AOF 文件损坏（如写入时宕机导致末尾不完整）时：

```bash
# 检查并修复 AOF 文件（截断不完整的命令）
redis-check-aof --fix appendonly.aof
```

## 四、混合持久化（Redis 4.0+）

### 4.1 为什么需要混合持久化

| 方案 | 问题 |
|------|------|
| 纯 RDB | 可能丢失最后一次快照后的所有数据 |
| 纯 AOF | 文件体积大，恢复速度慢 |
| **混合持久化** | **兼顾 RDB 的快速恢复和 AOF 的数据安全** |

### 4.2 混合持久化原理

在 AOF 重写时，不再生成纯 AOF 文件，而是将 **RDB 快照写入 AOF 文件头部**，再将重写期间的增量写命令追加到 **AOF 文件尾部**。

```
混合持久化 AOF 文件结构：
┌──────────────────────────────────────────────┐
│              RDB 格式数据（头部）               │
│  ┌─────────────────────────────────────────┐ │
│  │ REDIS0009...（二进制 RDB 快照）          │ │
│  │ 包含 fork 时刻的全量内存数据              │ │
│  └─────────────────────────────────────────┘ │
├──────────────────────────────────────────────┤
│              AOF 格式数据（尾部）               │
│  ┌─────────────────────────────────────────┐ │
│  │ *3\r\n$3\r\nSET\r\n$3\r\nkey\r\n...    │ │
│  │ 包含重写期间的增量写命令                   │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

### 4.3 混合持久化配置

```conf
# redis.conf（Redis 4.0+）
aof-use-rdb-preamble yes   # 开启混合持久化（Redis 5.0+ 默认开启）
appendonly yes              # 必须同时开启 AOF
```

### 4.4 恢复流程

```
启动 Redis
     │
     ▼
检测 AOF 文件头部
     │
     ├── 是 RDB 格式 → 按 RDB 方式加载头部数据
     │                      │
     │                      ▼
     │                 按 AOF 方式重放尾部增量命令
     │                      │
     │                      ▼
     │                   恢复完成
     │
     └── 是纯 AOF 格式 → 按 AOF 方式重放所有命令
                             │
                             ▼
                          恢复完成
```

### 4.5 混合持久化优缺点

| 优点 | 缺点 |
|------|------|
| 快速加载（RDB 部分二进制直接加载） | AOF 文件可读性变差（头部是二进制） |
| 数据安全（AOF 部分保证增量不丢失） | 兼容性问题（旧版本 Redis 无法识别） |
| 文件体积适中 | 需要 Redis 4.0+ |

## 五、持久化方案选型

### 5.1 不同场景推荐

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **纯缓存**（数据可丢失） | 关闭持久化 | 最佳性能，数据丢失后可从 DB 重建 |
| **允许分钟级数据丢失** | RDB only | 备份简单，恢复快，适合定期快照 |
| **要求秒级数据安全** | AOF（everysec） | 最多丢 1 秒数据 |
| **兼顾安全与性能**（推荐） | **混合持久化** | RDB 快速恢复 + AOF 增量安全 |
| **金融级零丢失** | AOF（always）+ 从节点备份 | 每条命令刷盘 + 多副本 |

### 5.2 持久化对性能的影响

| 操作 | 性能影响 | 优化建议 |
|------|---------|---------|
| `fork()` | 内存越大，fork 越慢，短暂阻塞主线程 | 控制 Redis 实例内存 ≤ 10GB |
| `fsync()` | 磁盘 IO 开销 | 使用 SSD，避免 always 策略 |
| AOF 重写 | 额外的磁盘写入 | 错开高峰期执行 |
| Copy-On-Write | 可能导致内存翻倍 | 预留足够内存，避免与其他进程争抢 |

### 5.3 持久化最佳实践

```
1. 主节点：开启混合持久化（aof-use-rdb-preamble yes）
2. 从节点：开启 RDB，用于冷备份和快速恢复
3. 控制 Redis 实例内存不超过 10GB（减少 fork 开销）
4. 使用 SSD 磁盘存储持久化文件
5. 监控 fork 耗时：INFO 中的 latest_fork_usec
6. 定期将 RDB/AOF 文件备份到异地存储
```

## 六、Redis 启动数据恢复流程

### 6.1 恢复优先级（核心考点 ⭐⭐⭐）

```
Redis 启动
    │
    ▼
是否开启 AOF？ ──── 是 ──► AOF 文件是否存在？
    │                           │
    否                    ┌─────┴─────┐
    │                     是          否
    ▼                     ▼           ▼
RDB 文件             加载 AOF 文件  加载 RDB 文件
是否存在？           恢复数据       恢复数据
    │                     │           │
  ┌─┴─┐                  ▼           ▼
  是   否            启动成功     启动成功
  ▼    ▼
加载   空数据库
RDB    启动
恢复
```

**核心规则**：
1. **AOF 优先级高于 RDB**：如果开启了 AOF，优先使用 AOF 恢复
2. AOF 文件不存在时才回退到 RDB
3. 如果两者都不存在，以空数据库启动

### 6.2 恢复速度对比

| 方式 | 速度 | 原因 |
|------|------|------|
| RDB 恢复 | **快** | 直接加载二进制文件到内存 |
| AOF 恢复 | **慢** | 需要逐条重放命令 |
| 混合持久化恢复 | **较快** | RDB 头快速加载 + 少量 AOF 增量重放 |

### 6.3 验证持久化配置

```bash
# 查看持久化状态
127.0.0.1:6379> INFO persistence

# 关键指标
rdb_last_bgsave_status:ok          # 上次 RDB 是否成功
rdb_last_bgsave_time_sec:1         # 上次 RDB 耗时
aof_enabled:1                      # AOF 是否开启
aof_last_rewrite_time_sec:0        # 上次重写耗时
aof_current_size:1024              # 当前 AOF 文件大小
latest_fork_usec:512               # 上次 fork 耗时（微秒）
```

## 七、大厂常见面试题

### Q1：RDB 和 AOF 的区别是什么？各自的优缺点？

**答：**

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 持久化方式 | 全量快照（二进制文件） | 增量日志（记录写命令） |
| 文件大小 | 小（压缩二进制） | 大（文本格式命令日志） |
| 恢复速度 | 快 | 慢（需要重放命令） |
| 数据安全 | 可能丢失最后一次快照后的数据 | 最多丢 1 秒（everysec） |
| 性能影响 | fork 时短暂阻塞 | 每秒 fsync 有持续 IO 开销 |
| 适用场景 | 备份、灾难恢复、冷备 | 对数据完整性要求高的场景 |

---

### Q2：BGSAVE 的 fork + Copy-On-Write 是怎么工作的？

**答：**

1. 主进程调用 `fork()` 创建子进程，子进程 **复制主进程的页表**（而非复制内存数据）
2. fork 之后，父子进程共享相同的物理内存页，页面标记为 **只读**
3. 子进程遍历内存，将数据写入 RDB 文件（读取的是 fork 时刻的一致性快照）
4. 主进程继续处理请求。当主进程 **修改** 某个内存页时，操作系统会 **复制该页**（Copy-On-Write），主进程在副本上修改
5. 子进程完成写入后退出，主进程用新 RDB 文件替换旧文件

**优点**：fork 速度快（只复制页表），子进程和主进程互不干扰。  
**风险**：如果重写期间大量数据被修改，COW 可能导致内存使用量接近翻倍。

---

### Q3：AOF 重写期间如何保证数据一致性？

**答：**

AOF 重写期间，主进程产生的新写命令会 **同时写入两个缓冲区**：

1. **AOF 缓冲区**（aof_buf）：正常写入旧 AOF 文件，保证旧文件完整性
2. **AOF 重写缓冲区**（aof_rewrite_buf）：暂存增量命令

当子进程完成重写后：
1. 通知主进程
2. 主进程将重写缓冲区中的增量命令追加到新 AOF 文件末尾
3. 用新 AOF 文件 **原子替换** 旧 AOF 文件

这样即使重写过程中发生故障，旧 AOF 文件仍然完整可用。

### Q4：Redis 启动时如何恢复数据？AOF 和 RDB 同时存在时加载哪个？

**答：**

Redis 的数据恢复遵循以下优先级：
1. 如果开启了 AOF（`appendonly yes`），优先加载 AOF 文件
2. 如果 AOF 未开启或 AOF 文件不存在，加载 RDB 文件
3. 如果两者都不存在，以空数据库启动

**AOF 优先于 RDB** 的原因：AOF 的数据通常比 RDB 更完整（更新频率更高）。

如果使用混合持久化，AOF 文件头部是 RDB 格式，Redis 会先按 RDB 方式加载头部，再按 AOF 方式重放尾部增量命令。

---

### Q5：AOF 的三种刷盘策略怎么选？生产环境推荐哪种？

**答：**

| 策略 | 数据丢失风险 | 性能 | 适用场景 |
|------|------------|------|---------|
| `always` | 最多丢 1 条命令 | 差（每条命令都 fsync） | 金融级别，对数据零丢失要求极高 |
| `everysec` | 最多丢 1 秒数据 | 好 | **生产环境推荐**，兼顾安全与性能 |
| `no` | 最多丢 30 秒数据 | 最好 | 对性能要求极高，允许数据丢失 |

**生产环境推荐 `everysec`**：
- 最多丢失 1 秒数据，可以接受
- 性能接近不开 AOF 的水平
- 配合混合持久化使用效果最佳

### Q6：fork 操作为什么会阻塞主线程？如何优化？

**答：**

`fork()` 需要复制主进程的 **页表**，虽然不复制实际内存数据，但页表本身的大小与内存使用量成正比：

- Redis 使用 10GB 内存 → 页表约 20MB → fork 耗时约 20ms
- Redis 使用 20GB 内存 → fork 耗时可能达 40-100ms

**优化建议**：
1. **控制实例大小**：单个 Redis 实例内存控制在 10GB 以内
2. **使用物理机**：避免虚拟机的内存虚拟化开销
3. **关闭 THP**：Transparent Huge Pages 会增加 COW 的拷贝粒度（2MB vs 4KB）
4. **监控 fork 耗时**：关注 `INFO` 中的 `latest_fork_usec`
5. **避免频繁持久化**：合理配置 `save` 和 `auto-aof-rewrite` 参数

```bash
# 关闭 Transparent Huge Pages
echo never > /sys/kernel/mm/transparent_hugepage/enabled
```

### Q7：混合持久化的原理是什么？为什么是 Redis 4.0+ 的推荐方案？

**答：**

混合持久化在 AOF 重写时，将 **RDB 快照写入 AOF 文件头部**，再将重写期间的增量写命令以 AOF 格式追加到文件尾部。

恢复时：先按 RDB 方式快速加载头部（全量数据），再按 AOF 方式重放尾部（增量命令）。

**推荐原因**：
1. **恢复速度快**：RDB 部分直接加载，不需要逐条重放
2. **数据安全**：AOF 增量部分保证不丢失重写后的数据
3. **文件体积适中**：比纯 AOF 小，比纯 RDB 更新频率高

是目前 **兼顾性能和数据安全的最佳方案**，Redis 5.0+ 默认开启。
