## 一、Redis 基础

### Q1：Redis 为什么这么快？

**答：**

Redis 的高性能来自以下五个方面：

1. **纯内存操作**：所有数据存储在内存中，读写速度为纳秒级别，远快于磁盘数据库
2. **单线程命令执行**：避免了多线程上下文切换和锁竞争的开销
3. **IO 多路复用**：基于 epoll/kqueue 实现，单线程可同时处理数万连接
4. **高效数据结构**：SDS、跳表、ziplist、intset 等都是针对性能优化的结构
5. **渐进式 Rehash**：Hash 扩容时分多次搬迁，避免一次性阻塞

单线程模型并不慢，因为 Redis 的瓶颈在网络 IO 而非 CPU，IO 多路复用已经解决了并发连接问题。

---

### Q2：Redis 6.0 为什么引入多线程？和 Memcached 的多线程有什么区别？

**答：**

**引入原因**：Redis 6.0 之前，网络 IO 的读写是单线程的，在高并发场景下成为性能瓶颈。6.0 引入多线程仅用于**网络 IO 的读写**（read/write 系统调用），**命令执行仍然是单线程**。

| 对比项 | Redis 6.0 多线程 | Memcached 多线程 |
|--------|-----------------|-----------------|
| 多线程范围 | 仅网络 IO 读写 | 所有操作（IO + 命令执行） |
| 命令执行 | 单线程串行 | 多线程并行 |
| 线程安全 | 无需加锁（命令串行） | 需要加锁保护共享数据 |
| 复杂度 | 低 | 高 |

这种设计既提升了网络吞吐量（官方称提升约 2 倍），又保持了代码的简洁性和线程安全性。

---

### Q3：Redis 的 String、Hash、List、Set、ZSet 分别适合什么场景？

**答：**

| 类型 | 适用场景 | 示例 |
|------|---------|------|
| **String** | 计数器、分布式锁、缓存对象（序列化后） | `INCR page:view:count`、`SET lock:order NX` |
| **Hash** | 存储对象属性 | `HSET user:10001 name "张三" age 25` |
| **List** | 消息队列、最新列表 | `LPUSH msg:queue "新消息"`、最新文章列表 |
| **Set** | 去重、集合运算（交集/并集） | 共同关注、共同好友 |
| **ZSet** | 排行榜、延迟队列 | `ZADD rank 100 "player1"` |

**补充**：
- **Bitmap**：签到、在线状态
- **HyperLogLog**：UV 统计（基数统计）
- **GEO**：附近的人
- **Stream**：消息队列（Redis 5.0+）

## 二、持久化

### Q4：RDB 和 AOF 有什么区别？各自的优缺点？

**答：**

| 对比项 | RDB | AOF |
|--------|-----|-----|
| **持久化方式** | 定时生成内存快照（二进制文件） | 记录每条写命令（追加写日志） |
| **文件大小** | 小（压缩的二进制） | 大（文本命令，可重写压缩） |
| **恢复速度** | 快（直接加载二进制） | 慢（逐条重放命令） |
| **数据安全性** | 低（两次快照间的数据可能丢失） | 高（取决于 fsync 策略，最多丢 1 秒） |
| **写性能影响** | BGSAVE 时 fork 子进程，有内存翻倍风险 | 追加写入，重写时有额外 IO |

**生产建议**：同时开启 RDB + AOF（Redis 4.0+ 推荐混合持久化），用 AOF 保证数据安全，用 RDB 加速恢复。

---

### Q5：混合持久化是怎么工作的？

**答：**

混合持久化（Redis 4.0+）结合了 RDB 和 AOF 的优点：

1. AOF 重写时，不再只写 AOF 命令，而是先将**当前数据以 RDB 格式写入 AOF 文件头部**
2. 重写期间的增量写命令以 **AOF 格式追加**到文件尾部
3. 最终 AOF 文件 = RDB 快照 + 增量 AOF 日志

```
  AOF 文件结构（混合持久化）：
  ┌─────────────────────────┐
  │  RDB 快照（二进制格式）   │ ← 全量数据，恢复快
  ├─────────────────────────┤
  │  增量 AOF 日志           │ ← 重写后的新命令
  │  SET key1 value1        │
  │  HSET key2 f1 v1        │
  └─────────────────────────┘
```

**优点**：兼顾恢复速度（RDB 部分）和数据安全性（AOF 部分）。

```conf
# 开启混合持久化
aof-use-rdb-preamble yes
```

### Q6：Redis 重启后如何恢复数据？

**答：**

Redis 启动时的数据恢复流程：

1. 如果开启了 AOF（`appendonly yes`），优先加载 AOF 文件
2. 如果 AOF 关闭或文件不存在，加载 RDB 文件（`dump.rdb`）
3. 如果两者都不存在，以空数据启动

```
  启动 → AOF 开启？ ── 是 ──→ AOF 文件存在？ ── 是 ──→ 加载 AOF
                │                    │
                否                   否
                │                    │
                ▼                    ▼
         RDB 文件存在？ ── 是 ──→ 加载 RDB
                │
                否
                ▼
           空数据启动
```

**为什么 AOF 优先？** 因为 AOF 的数据完整性更高（最多丢失 1 秒数据），而 RDB 可能丢失最后一次快照后的所有数据。

## 三、高可用

### Q7：Redis 主从复制的原理？全量复制和增量复制的区别？

**答：**

**主从复制原理**：从节点连接主节点后，主节点执行 BGSAVE 生成 RDB 发送给从节点，从节点加载 RDB 完成全量同步。之后主节点将新的写命令通过**复制积压缓冲区**同步给从节点。

| 对比项 | 全量复制 | 增量复制 |
|--------|---------|---------|
| **触发场景** | 首次连接、repl_backlog 溢出 | 短暂断连恢复 |
| **过程** | BGSAVE → 传输 RDB → 加载 RDB | 通过 offset 从积压缓冲区获取增量命令 |
| **开销** | 大（生成+传输 RDB） | 小（只传差异命令） |

**复制积压缓冲区（repl_backlog）**：环形缓冲区，默认 1MB。从节点断连后重连，如果断连期间的命令还在缓冲区内（通过 offset 比较），则增量复制；否则全量复制。

---

### Q8：哨兵模式的工作原理？怎么判断主节点下线？

**答：**

**Sentinel 三大功能**：监控、通知、自动故障转移。

**主观下线**（SDOWN）：单个 Sentinel 发现主节点在 `down-after-milliseconds` 时间内没有有效回复 PING，判定为主观下线。

**客观下线**（ODOWN）：Sentinel 向其他 Sentinel 询问，如果**超过 quorum 个** Sentinel 都认为主节点主观下线，则判定为客观下线，触发故障转移。

**故障转移流程**：
1. Sentinel 之间通过 Raft 算法选举出 Leader Sentinel
2. Leader 从存活的从节点中选择一个作为新主节点（优先级 → offset → runid）
3. 将其他从节点指向新主节点
4. 通知客户端新主节点地址

---

### Q9：Redis Cluster 是怎么分片的？怎么做故障转移？

**答：**

**数据分片**：Cluster 将整个数据空间划分为 **16384 个 slot（槽）**，每个节点负责一部分槽。key 通过 `CRC16(key) % 16384` 计算所属槽。

```
  Node-A: slot 0 ~ 5460
  Node-B: slot 5461 ~ 10922
  Node-C: slot 10923 ~ 16383
```

**故障转移**：每个主节点有对应的从节点。当主节点被集群中过半数主节点标记为 PFAIL 后，从节点发起选举（需要过半主节点投票），当选后提升为新的主节点接管原主节点的槽。

**扩缩容**：通过 `CLUSTER ADDSLOTS` / `CLUSTER RESHARD` 在线迁移槽，无需停机。

## 四、缓存策略

### Q10：缓存穿透、缓存击穿、缓存雪崩有什么区别？怎么解决？

**答：**

| 问题 | 核心特征 | 解决方案 |
|------|---------|---------|
| **穿透** | 数据在缓存和 DB 中**都不存在** | 布隆过滤器、缓存空值、参数校验 |
| **击穿** | **热点 key** 过期瞬间被大量并发请求打穿 | 互斥锁、逻辑过期、永不过期 |
| **雪崩** | **大量 key 同时过期**或 Redis 宕机 | 随机 TTL、多级缓存、集群高可用、限流降级 |

记忆口诀：**穿透查不到，击穿热点爆，雪崩一片倒**。

---

### Q11：Cache Aside 策略为什么是先更新 DB 再删缓存？为什么不更新缓存而是删除？

**答：**

**为什么先更新 DB 再删缓存（而非先删缓存再更新 DB）**：

先删缓存的问题：线程 A 删缓存 → 线程 B 读到旧 DB 并回填缓存 → 线程 A 更新 DB，**缓存长期为旧值**。先更新 DB 再删缓存发生不一致的概率极低（需要"缓存刚过期 + 读比写慢"），加上 TTL 兜底。

**为什么删除而非更新缓存**：
1. **避免并发写覆盖**：线程 A 和 B 同时更新，更新缓存的顺序可能与更新 DB 的顺序不一致
2. **懒加载**：删除后下次读时再构建缓存，避免频繁写但很少读的场景浪费资源
3. **简单可靠**：删除是幂等操作，更新不是

---

### Q12：布隆过滤器的原理？为什么会有误判？能不能删除元素？

**答：**

**原理**：使用 k 个哈希函数将元素映射到 m 位的位数组上。添加时将对应位置 1，查询时检查所有位是否都为 1。

**误判原因**：不同元素的哈希值可能碰撞，导致某些位被多个元素共同置 1。当查询一个不存在的元素时，如果它对应的所有位恰好都被其他元素置为 1，就会误判为"存在"。

**不能删除**：因为多个元素可能共享某一位，将某位置 0 会影响其他元素的判断。如果需要删除，可以使用**计数布隆过滤器（Counting Bloom Filter）**，用计数器代替位。

---

### Q13：如何保证缓存和数据库的一致性？

**答：**

没有银弹，根据一致性要求选择方案：

1. **Cache Aside + TTL**（最常用）：先更新 DB 再删缓存，TTL 兜底。简单有效，适合 90% 的场景。
2. **延迟双删**：先删缓存→更新 DB→延迟再删。解决并发读旧值问题，延迟一般 500ms~1s。
3. **订阅 Binlog（Canal）**：Canal 监听 MySQL binlog 变更后异步更新缓存，无代码侵入。
4. **消息队列异步重试**：删缓存失败时发送到 MQ 重试，保证最终一致。

**核心原则**：缓存一致性一般追求**最终一致性**，不追求强一致。强一致需要分布式事务，代价太高。

## 五、分布式锁

### Q14：Redis 分布式锁怎么实现？有什么问题？

**答：**

**实现**：`SET lock_key unique_value NX EX 30`

- `NX`：key 不存在才设置（保证互斥）
- `EX 30`：过期时间 30 秒（防死锁）
- `unique_value`：客户端唯一标识（UUID + 线程ID，防止误删）

**释放**：通过 Lua 脚本原子地 "先比较 value 再删除"：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**问题**：
1. 锁过期但业务未完成 → Redisson Watchdog 自动续期
2. 不可重入 → Redisson Hash 结构 + 重入计数
3. 单点故障 → RedLock / Redis 集群
4. 非公平 → 需自行实现排队逻辑

---

### Q15：Redisson 的看门狗（Watchdog）机制是什么？

**答：**

当使用 `lock()` 或 `tryLock(waitTime)` 不指定 `leaseTime` 时，Redisson 启动 Watchdog：

1. 加锁默认 TTL 为 **30 秒**（`lockWatchdogTimeout`）
2. 后台每 **10 秒**（30/3）检查一次锁是否仍被持有
3. 仍持有则将 TTL **续期到 30 秒**
4. `unlock()` 后取消 Watchdog 定时任务

**注意**：手动指定 `leaseTime` 时 Watchdog **不会启动**，锁会在指定时间后自动过期。

---

### Q16：RedLock 算法是什么？有什么争议？

**答：**

**算法**：基于 N 个独立 Redis 实例（非集群），向每个实例发起加锁请求。**过半数（≥ N/2+1）成功**且**总耗时 < 锁 TTL** 则加锁成功。

**争议**（Martin Kleppmann vs Antirez）：

Martin 的质疑：
- **时钟漂移**：依赖各实例时钟同步，NTP 校时可能导致锁提前过期
- **GC 暂停/网络延迟**：客户端获取锁后暂停，锁过期被其他客户端获取，恢复后两者同时持锁
- **缺少 Fencing Token**：无法通过递增 token 保护资源端

**实践建议**：
- 大部分场景用 Redisson 单实例 + Watchdog 即可
- 强一致性场景考虑 Zookeeper
- RedLock 适合对一致性要求高于单实例但不需要强一致的场景

## 六、内存管理

### Q17：Redis 有哪些内存淘汰策略？LRU 和 LFU 有什么区别？

**答：**

**8 种策略**：noeviction、allkeys-lru、volatile-lru、allkeys-random、volatile-random、volatile-ttl、allkeys-lfu（4.0+）、volatile-lfu（4.0+）

| 维度 | LRU | LFU |
|------|-----|-----|
| 依据 | 最近访问时间 | 访问频率 |
| 缺陷 | 偶发访问的冷数据也能续命 | 历史热点不再访问也难淘汰 |
| Redis 实现 | 随机采样近似 LRU | 概率计数器 + 衰减 |
| 适用 | 访问有时间局部性 | 热点稳定、冷热分明 |

**推荐**：通用场景用 `allkeys-lru`，热点明显用 `allkeys-lfu`。

---

### Q18：大 Key 问题怎么排查和处理？

**答：**

**判定标准**：String > 10KB，集合类型 > 5000 元素。

**排查**：
- `redis-cli --bigkeys`：快速扫描，找每种类型最大的 key
- `MEMORY USAGE key`：查单个 key 内存占用
- RDB 分析工具（rdb-tools）：离线分析所有大 key

**处理**：
- **删除**：使用 `UNLINK`（异步）代替 `DEL`（同步阻塞），或分批删除
- **预防**：数据拆分（大 Hash 拆为多个小 Hash）、压缩（GZIP/Snappy）、Protobuf 序列化

---

### Q19：热 Key 怎么处理？

**答：**

**发现**：`redis-cli --hotkeys`（需 LFU）、客户端统计、代理层统计

**解决**：
1. **本地缓存**（最有效）：Caffeine/Guava Cache 作为 L1，短 TTL（10~60s）
2. **Key 分散**：将热 Key 复制到 N 个副本（`key:0` ~ `key:9`），随机读取
3. **读写分离**：增加从节点分散读压力

**核心思路**：将集中在单节点的流量分散到多个节点或本地。

## 七、实战问题

### Q20：如何用 Redis 实现排行榜？

**答：**

使用 **ZSet（有序集合）**，score 作为排序依据。

```bash
# 添加/更新分数
ZADD rank:game 1000 "playerA"
ZADD rank:game 2000 "playerB"
ZADD rank:game 1500 "playerC"

# 增加分数（如打怪加分）
ZINCRBY rank:game 500 "playerA"

# 获取 Top N（降序）
ZREVRANGE rank:game 0 9 WITHSCORES
# 返回前 10 名及分数

# 获取某玩家排名（从0开始）
ZREVRANK rank:game "playerA"

# 获取某玩家分数
ZSCORE rank:game "playerA"
```

**注意**：
- 实时排行榜：直接用 ZSet，时间复杂度 O(log N)
- 海量数据排行榜：可按时间维度分桶（日/周/月榜），避免单 key 过大
- 并列排名：ZSet 默认相同 score 按字典序排序，如需处理并列需额外逻辑

---

### Q21：如何用 Redis 实现延迟队列？

**答：**

使用 **ZSet**，score 存储执行时间戳。

```java
// 添加延迟任务：5分钟后执行
long executeTime = System.currentTimeMillis() + 5 * 60 * 1000;
redisTemplate.opsForZSet().add("delay:queue", taskId, executeTime);

// 轮询消费（定时任务）
@Scheduled(fixedDelay = 1000)
public void consumeDelayQueue() {
    long now = System.currentTimeMillis();
    // 取出所有到期的任务（score ≤ 当前时间）
    Set<String> tasks = redisTemplate.opsForZSet()
        .rangeByScore("delay:queue", 0, now);

    for (String taskId : tasks) {
        // 原子地移除并处理（防止重复消费）
        Long removed = redisTemplate.opsForZSet()
            .remove("delay:queue", taskId);
        if (removed != null && removed > 0) {
            // 处理任务
            processTask(taskId);
        }
    }
}
```

**注意**：
- 多消费者场景需要使用 `ZRANGEBYSCORE` + `ZREM` 的原子操作（Lua 脚本）
- 可靠性不如专业消息队列（如 RabbitMQ/RocketMQ），任务可能丢失

---

### Q22：Redis 做消息队列有哪些方式？有什么局限性？

**答：**

| 方式 | 实现 | 特点 |
|------|------|------|
| **List** | `LPUSH` + `BRPOP` | 简单，点对点消费 |
| **Pub/Sub** | `PUBLISH` + `SUBSCRIBE` | 广播模式，消息不持久化 |
| **Stream** | `XADD` + `XREADGROUP` | 持久化、消费者组、ACK 确认（5.0+） |

**Redis 做消息队列的局限性**：

1. **消息可靠性**：非专业 MQ，不保证消息不丢失（尤其 Pub/Sub 断连即丢）
2. **消息堆积**：大量消息堆积在 Redis 中会占用大量内存
3. **消费确认**：List 模式无 ACK 机制，Stream 有但不如 RabbitMQ/Kafka 完善
4. **功能有限**：不支持死信队列、延迟消息（需自行实现）、事务消息
5. **不适合海量数据**：Redis 数据在内存中，成本高

**建议**：简单场景用 Stream，复杂场景用专业 MQ（RabbitMQ/RocketMQ/Kafka）。

---

### Q23：Redis 如何实现分布式 Session？

**答：**

将用户 Session 存储到 Redis 中，所有应用实例共享。

```java
// Spring Session + Redis 配置
@EnableRedisHttpSession(maxInactiveIntervalInSeconds = 1800)
public class SessionConfig {
    @Bean
    public LettuceConnectionFactory connectionFactory() {
        return new LettuceConnectionFactory("redis-host", 6379);
    }
}
```

**原理**：
1. 用户登录后，Session 数据以 Hash 结构存入 Redis：`spring:session:sessions:<sessionId>`
2. 每次请求，通过 Cookie 中的 SessionId 从 Redis 获取 Session
3. 所有应用实例共享同一个 Redis，实现 Session 共享

**优点**：无状态部署、Session 共享、支持水平扩展  
**注意**：Session 中的对象需要实现 `Serializable` 接口

### Q24：如何设计一个 Redis 缓存系统来支撑百万 QPS？

**答：**

**架构设计**：

```
                    ┌──────────────┐
                    │  网关层       │  限流 + 路由
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ App-1    │ │ App-2    │ │ App-N    │
        │┌────────┐│ │┌────────┐│ │┌────────┐│
        ││Caffeine││ ││Caffeine││ ││Caffeine││  L1 本地缓存
        │└────────┘│ │└────────┘│ │└────────┘│
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │             │             │
             └─────────────┼─────────────┘
                           ▼
              ┌─────────────────────────┐
              │     Redis Cluster       │  L2 分布式缓存
              │  (主从 + 多分片)         │
              └────────────┬────────────┘
                           │
                           ▼
              ┌─────────────────────────┐
              │     MySQL 集群          │  持久层
              │  (主从 + 分库分表)       │
              └─────────────────────────┘
```

**核心要点**：
1. **多级缓存**：Caffeine（L1）+ Redis Cluster（L2）+ DB
2. **Redis Cluster**：数据分片 + 读写分离，单集群可达百万 QPS
3. **热 Key 处理**：本地缓存兜底，Key 分散
4. **防雪崩**：随机 TTL + 限流降级
5. **缓存预热**：上线前加载热点数据
6. **监控报警**：监控命中率、延迟、内存使用

### Q25：Redis 的 Pipeline 和事务有什么区别？

**答：**

| 对比项 | Pipeline | 事务（MULTI/EXEC） |
|--------|----------|-------------------|
| **作用** | 批量发送命令减少网络 RTT | 保证一组命令的原子执行 |
| **原子性** | 无（命令间可能被其他命令插入） | 有（EXEC 时一次性执行所有命令） |
| **回滚** | 不支持 | 不支持（Redis 事务不保证回滚） |
| **性能** | 大幅减少网络开销 | 无显著性能提升 |

**Pipeline** 的核心是减少网络往返次数，适合批量读写：

```java
List<Object> results = redisTemplate.executePipelined((RedisCallback<Object>) conn -> {
    for (int i = 0; i < 1000; i++) {
        conn.set(("key:" + i).getBytes(), ("value:" + i).getBytes());
    }
    return null;
});
```

**注意**：Redis 事务不支持回滚。如果事务中某条命令执行失败（如对 String 执行 INCR），其他命令仍会执行。这与传统数据库事务不同。
